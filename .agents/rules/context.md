# Context Rules

`CONTEXT.md` は、このリポジトリで使うドメイン語彙の定義元である。

## 基本原則

設計議論、docs、コード説明では、`CONTEXT.md` に定義された用語を優先して使う。
同じ概念に別名を作らず、既存の canonical name と日本語名を保つ。

## MUST

- 既存の `CONTEXT.md` に該当用語がある場合、その用語を使う。
- 初出では `English Term (日本語名)` の形式を優先する。
- `CONTEXT.md` の `_Avoid_` にある語は、その概念を指す用語として使わない。
- 新しい重要概念を確定した場合は、次の phase、検証、PR 化へ進む前に `CONTEXT.md` へ反映する。
- 追加判断に必要な情報が不足する場合は、glossary gap として扱い、人間へ一問だけ確認する。

## MUST NOT

- `CONTEXT.md` にある用語を、別の日本語訳や略称に言い換えない。
- 曖昧な一般語で `CONTEXT.md` の用語を置き換えない。
- `CONTEXT.md` を実装仕様、設計メモ、タスクリストとして使わない。
- 未確定の用語を確定語彙として追加しない。

## 用語追加の基準

`CONTEXT.md` に追加するのは、このプロジェクト固有のドメイン概念だけにする。
一般的な技術語、ファイル名、関数名、単発の作業ラベルは追加しない。

## 同期規約

`CONTEXT.md` は語彙の唯一の定義元（正準）であり、`amadeus/spaces/<space>/knowledge/glossary.md` は workspace 運用で頻用する語の抜粋である（詳細は同ファイル冒頭の抜粋宣言を参照）。同期は次の 4 点に従う。

1. 新しい語彙が確定した場合は、対象 Intent の gate 承認までに `CONTEXT.md` へ反映する（前節「MUST」の維持・明文化）。
2. 実装・merge 済みの差分が `CONTEXT.md` とずれている場合は、現在の実装を優先して読み、確定している語彙だけを `CONTEXT.md` へ逆同期する。未決定事項、一時的な実装都合、ローカル変数名、単発のファイル名、未採用の過去案は確定語彙として追加しない。
3. `glossary.md` への反映は、workspace 運用語彙に変化があった場合だけ、`CONTEXT.md` から抜粋する形で行う。`glossary.md` 自身が独自の定義を持つことはない。
4. 未決定事項は、確定するまでどちらの成果物にも確定語彙として追加しない。
