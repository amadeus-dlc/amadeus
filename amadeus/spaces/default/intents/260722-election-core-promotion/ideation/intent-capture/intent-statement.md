# Intent Statement — チーム機能のコア昇格

> 上流入力(consumes 全数): なし(本ステージは consumes 宣言なし。入力はユーザーの intent 記述と 2026-07-23 グリル裁定 — intent-capture-questions.md)

## Problem Statement(解決するビジネス課題)

amadeus 独自の強みであるエージェントチーム機能(チーム起動・メンバー間メッセージング・選挙による合意形成・チームモード運用契約)が、公式配布物に含まれていない。現状の実装は開発リポジトリ内の非配布層に散在する:

- 選挙エンジン: CLI 本体は `scripts/`(開発専用層)、スキルは `contrib/skills/`(ドッグフード専用層)の層またぎ状態
- チーム起動(team-up.sh)・メッセージングは herdr / agmsg という開発者個人環境の外部ツールに依存し、配布経路が存在しない

このため amadeus を自プロジェクトへインストールした利用者は、チーム機能を一切利用できない。選挙 CLI 等の一部だけを配布しても投票者(チーム)が存在せず利用者体験はゼロであることがグリルで確認され(Q3)、**チーム機能一式を利用可能な形で公式配布に昇格する**ことが本 intent の目的である(Q1)。

なお発端の懸念だった「開発用/配布用スクリプトの混同による自己開発の失敗リスク」は、既存の3層構造(`scripts/`=開発専用 / `contrib/`=ドッグフード専用 / `packages/framework/`=配布正本)の実在確認により払拭済み(Q1)。この構造を機械的に保存する境界ガードは本 intent のスコープに含める(Q2)。

## Target Customer(誰が恩恵を受けるか)

- **主顧客: 外部利用者** — amadeus を自プロジェクトへインストールし、複数エージェントのチーム開発(チームモード)を試したい開発者・チーム。現状はチーム機能に到達する手段がない(Q5)
- **副顧客: ドッグフードチーム(私たち自身)** — 公式配布物への移行により `scripts/` 直依存の運用から卒業し、利用者と同じ経路でチーム機能を使う(Q5)

## Success Metrics(成功の定義)

- **クリーン環境 E2E(最重要)**: amadeus を新規インストールした環境で、公式 docs の手順だけを頼りにエージェントチームを結成し、選挙を1回完走できる(Q4)。ドッグフード環境固有の前提(herdr / agmsg のホームディレクトリ設置等)の混入がないことをこの基準で実証する
- **機械検証**: `dist:check` / `promote:self:check` / 既存テストスイートが green、新設の境界ガード(配布ツリー→ `scripts/` 参照禁止)が「落ちる実証」込みで稼働する(Q2)
- **docs 完備**: チーム機能のセットアップから選挙完走までを導く公式ガイドが存在する(Q4 の E2E が docs 駆動であることに含意)

## Initiative Trigger(なぜ今か)

- 選挙 CLI は E-ETF-CANON(2026-07-19)で選挙プロトコルの正本に既に裁定済みであり、実装の配置だけが非公式のまま残っている
