## Q1. ストーリー分割方針

ストーリーの束ね方の選択肢:

- A. **ワークフロー(ジャーニー)別** — 「install ジャーニー」「upgrade ジャーニー」「メンテナ publish ジャーニー」の3エピックに分け、各分岐(対話/非対話、衝突、バージョン境界)をストーリー化する(推奨: assessment の価値領域と一致し、ペルソナと1:1に近い対応になる。E2E テスト設計(NFR-001)にもそのまま流用できる)
- B. 機能(FR)別 — FR-001〜018 をそのままストーリー化する(要件と1:1だが、ジャーニー横断の価値が見えにくい)
- C. ペルソナ別 — ペルソナごとに全機能を列挙する(重複が多い)
- X. Other(自由記述)

[Answer]: A. ジャーニー別 — install / upgrade / publish の3エピックに分けて各分岐をストーリー化(2026-07-08、Mode: grilling)

## Q2. ペルソナの確定

stakeholder-map 由来の3ペルソナ案:

1. **新規ユーザー**(例: OSS 利用者の開発者)— Amadeus 未導入。目標: クローンせず1コマンド・1分で使い始める。痛点: 手動コピーの煩雑さ・ハーネス取り違え
2. **既存ユーザー**(例: 導入済みプロジェクトのオーナー)— 目標: 自分のカスタマイズを失わずに最新版へ更新。痛点: 更新手段がなく古い版に取り残される
3. **メンテナ**(このリポジトリの保守者)— 目標: publish・タグ発行・パッケージ検証を低負荷で回す。痛点: 手順の暗黙知化・メタデータ不備の公開事故

この3ペルソナで確定するか?

- A. 確定 — 3ペルソナで進める(推奨: FR 18件のすべてがこの3者のいずれかに帰属し、過不足がない)
- B. 修正 — ペルソナを追加/削除/変更する(内容を指定)
- X. Other(自由記述)

[Answer]: A. 確定 — 3ペルソナ(新規ユーザー / 既存ユーザー / メンテナ)で進める(2026-07-08、Mode: grilling)

## Q3. ストーリー粒度と件数の目安

ジャーニー別3エピック配下のストーリー粒度:

- A. **分岐単位で 12〜16 本** — 各 FR の主要分岐(対話/非対話、導入済み検出、md5 一致/相違、バージョン境界、pack 検証など)を独立ストーリーにする(推奨: Standard 深度の要件カバレッジと INVEST の Small/Testable を両立し、intent-backlog P1〜P5 との対応も追いやすい)
- B. 粗く 6〜8 本 — ジャーニーの主要シナリオのみ(細部は受け入れ基準に押し込む)
- C. 細かく 20 本超 — 受け入れ基準1つ相当まで分解する(管理コスト増)
- X. Other(自由記述)

[Answer]: A. 分岐単位で 12〜16 本(2026-07-08、Mode: grilling)
