## タイムアウト閾値(数値の強制メカニズム)

| 通信 | 1試行タイムアウト | 実装 |
|------|-------------------|------|
| GitHub REST(releases / tags) | 10秒 | `AbortSignal.timeout(10_000)` を fetch に渡す |
| アーカイブダウンロード | 20秒 | 同上(20_000)。リトライは BR-F06 のちょうど1回 |

## 時間バジェット(NFR-001「1分以内」の U1 配分 — タイムアウトから導出)

| 処理 | 通常経路 | リトライ経路(最悪) | 根拠 |
|------|----------|----------------------|------|
| バージョン解決(REST 最大2リクエスト) | ≤ 5秒 | ≤ 20秒(2req × 10秒) | BR-F09+上記タイムアウト |
| アーカイブ取得 | ≤ 20秒 | ≤ 40秒(2試行 × 20秒) | 上記タイムアウト×BR-F06 |
| 展開+ExtractedPayload.locate | ≤ 5秒 | ≤ 5秒 | ローカル I/O のみ |
| マニフェスト読み書き | ≤ 1秒 | ≤ 1秒 | 単一 JSON |
| **U1 合計** | **≤ 31秒** | **≤ 66秒** | — |

- **NFR-001 との整合**: 「1分以内」は NFR-001 の文言どおり**通常のネットワーク条件下**(リトライ不要経路)の目標であり、通常経路合計 31秒+U2 の適用/検証で1分に収まる。リトライ発生経路は例外系(劣化許容)で、NFR-001 の E2E アサート対象から除外する — この区別を E2E テスト仕様に明記する
- 計測点: fetcher が取得開始/完了を計測し、E2E テストが全体経過時間をアサートする(モック/フィクスチャ時はネットワーク項を除外)
- **md5 計算の帰属訂正**: 期待 md5 は U2/U3 の**プラン時に planner が計算**し(U2 functional-design の確定契約)、U1 の `buildManifest` は計算済みの `ManifestFiles` を受け取るだけ(U1 workflow 3 も同契約に整合済み)。よって md5 計算コストは U2/U3 のプラン予算に属する

## 実装上の性能規律

- ストリーミング展開(全体をメモリへ載せない)。ピークメモリ ≤ 256MB — **検証**: 大型フィクスチャを使う integration テストで `process.memoryUsage().rss` のピークをサンプリングしてアサート
- 逐次処理で十分(並列化は複雑性に見合わない — 数千ファイル規模で I/O バウンド)
