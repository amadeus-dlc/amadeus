# Intent Statement — Amadeus Grilling 統合

**Intent**: mattpocock の grilling スキルを Amadeus Grilling として統合する(stage-protocol の対話モード追加+スタンドアロンスキル)
**Date**: 2026-07-06 / **Stage**: intent-capture (1.1) / **Scope**: grilling-integration

## Problem Statement

Amadeus の対話モードは現在3つ(Guide me / I'll edit the file / Chat)だが、いずれも「前提を深掘りして設計判断の質を上げる」用途には最適化されていない。中心課題は **Guide me(選択式)の浅さ**である: 選択肢ベースの質問応答では、ユーザーは提示された想定内の回答から選んで先へ進んでしまい、隠れた前提・見落とした制約・未検討の代替案が表面化しないまま成果物が生成される。結果として設計判断が浅くなり、後段ステージでの手戻り(承認ゲートでの Request Changes)や、承認後に発覚する認識ズレのコストを生む。

mattpocock の grilling スキル(MIT ライセンス、約10行のプロンプトスキル)は、この対極のアプローチを実証している: 1問ずつ・推奨回答つきで相互理解に達するまで問い続け、事実はコードベース調査で自己解決し、**判断だけ**を人間に問う。この規律を Amadeus の構造化された対話契約(質問ファイル+監査ログ)の上に載せることで、深さとトレーサビリティを両立させる。

## Target Customer

二層の受益者を想定する:

1. **この repo の開発チーム自身(一次)** — ドッグフーディング。Amadeus 自体の開発ワークフローで grilling モードを使い、フレームワーク開発の設計判断の質を上げる。本 Intent がその最初の実走となる。
2. **Amadeus を導入する外部チーム(二次)** — 配布物(dist)の標準機能として全ハーネスに同梱し、導入チームがどのハーネスでも同じ grilling 体験を得られるようにする。

## Success Metrics

| 指標 | 種別 | 測定方法 |
|---|---|---|
| ゲート付き全ステージで「Grill me」モードが選択・完走できる | 機能完成 | 受け入れテスト(モード選択→対話→回答書き戻し→ゲートの完走) |
| 承認ゲートでの Request Changes(手戻り)が減る | 品質シグナル | 監査ログの REJECTED/RECOMPOSED イベント頻度の前後比較 |
| 「前提の見落としを拾えた」実感が得られる | 定性評価 | ドッグフーディングでの振り返り(§13 learnings への記録) |
| grilling 経由の決定も既存の監査契約に完全準拠する | 監査完全性 | 質問ファイルの [Answer]: 書き戻しと QUESTION_ANSWERED イベントが他モードと同一形式で残る |

## Initiative Trigger

**自身の利用体験**。grilling スキルの「1問ずつ・推奨回答つき・共通理解まで」という規律に触れ、Amadeus の対話品質のギャップ(選択式の浅さ)を埋める即効性のある手法だと判断した。市場圧力や規制ではなく、良いプラクティスの発見駆動(コミュニティのスキルエコシステムからの取り込み)である。

## Initial Scope Signal

カスタムスコープ `grilling-integration`(11/32 ステージ、Standard 深度)で確定済み。スコープ境界の初期シグナル:

- **In**: (1) stage-protocol への第4対話モード「Grill me」追加 — 既存の質問ファイル書き戻し・監査ログ契約を完全に保持する。(2) スタンドアロンスキル `/amadeus-grilling` — **read-only で汎用**(ワークフロー状態を進めない。ステージ成果物にもワークフロー外の任意の計画・設計にも使える)。(3) **全4ハーネス**(claude / codex / kiro / kiro-ide)同時展開。(4) 帰属表示: スキルファイルに MIT 帰属コメント+docs にクレジット。
- **Out**: 既存3モードの変更・削除。新しい監査イベント種別の追加。ハーネス別の挙動差し込み。

**前提(仮説として明示)**: grilling の「1問ずつ」規律は、既存ハーネスの構造化質問レンダリング(question-rendering annex)の枠内で表現できる — バッチサイズを1に固定する運用で成立する見込み。これは inception の設計ステージで検証する。
