# Personas — Amadeus Grilling 統合

**Intent**: Amadeus Grilling 統合 / **Stage**: user-stories (2.4)
**Upstream**: `../requirements-analysis/requirements.md` の Intent Analysis(二層受益者)と codekb `business-overview.md`(ドッグフーディング原則)に基づく3ペルソナ構成。

**優先順位ランキング**: P1 > P2 > P3。P1/P2 は一次受益者(ドッグフーディング — requirements の Intent Analysis「二層受益者」の一層目)、P3 は二次受益者(dist 配布経由の外部導入チーム)。

## P1: ワークフロー実行者(Wataru)

- **像**: Amadeus ワークフローを日常的に回す開発者。このリポジトリのオーナー自身が原型(ドッグフーディング)。
- **ゴール**: ステージの意思決定で隠れた前提を潰し、承認ゲート後の手戻りをなくしたい。
- **ペイン**: Guide me の選択式では提示された選択肢から選ぶだけになり、深掘りされないまま成果物が生成される(intent-statement の中心課題)。
- **利用文脈**: ゲート付きステージのモード選択で「Grill me」を選ぶ。監査ログと質問ファイルの完全性を重視する。

## P2: スタンドアロン利用者(Sole)

- **像**: ワークフローを走らせていない場面で、計画・設計・成果物を叩きたい開発者。P1 と同一人物のことも多いが利用文脈が異なる。
- **ゴール**: ワークフローのセレモニーなしに、任意の対象(ファイル/テーマ)へ grilling の深掘りだけを適用したい。
- **ペイン**: 従来はゲート前レビューや計画の壁打ちに使える構造化された深掘り手段がなかった。
- **利用文脈**: `/amadeus-grilling <対象>` を任意のタイミングで起動。ワークフロー状態が汚れないこと(read-only)を期待する。

## P3: 外部導入チームメンバー(Gaia)

- **像**: dist/<harness>/ を自プロジェクトへコピーして Amadeus を導入したチームの開発者。ハーネスは claude / codex / kiro / kiro-ide のいずれか。
- **ゴール**: どのハーネスでも文書どおりの Grill me 体験を得たい。出自(mattpocock の grilling)とライセンスが明確であってほしい。
- **ペイン**: ハーネスによって機能の有無や挙動が違うと導入判断や社内展開が難しい。
- **利用文脈**: docs を読んで機能を発見し、自ハーネスで利用する。NFR-2(パリティ)と FR-4(帰属・docs)の直接の受益者。
