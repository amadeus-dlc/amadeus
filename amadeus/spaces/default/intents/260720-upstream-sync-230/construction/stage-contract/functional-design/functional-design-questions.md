## Q1: UnitKind の単一定義をどのモジュールが所有するか

`requirements.md` FR-2 item 7 / FR-6 item 18、`components.md` C1、`component-methods.md` C1 は schema・graph・directive・sensor が同じ closed vocabulary を import し、第二定義を作らないことを固定している。実装配置は Functional Design へ委譲されている。

- A: `amadeus-stage-schema.ts` が `UNIT_KINDS` と `UnitKind` を所有し、graph/directive/sensor/runtime がそこから import する。契約の凝集度は高いが、既存 schema → lib の依存方向を変える可能性がある。
- B: upstream と同じく `amadeus-lib.ts` が `UNIT_KINDS` と `UnitKind` を所有し、stage-schema と全 consumer が import する。移植差分は最小だが、汎用 lib の公開面が増える。
- C: 新しい `amadeus-unit-kind.ts` を単一定義モジュールとして置き、stage-schema と全 consumer が import する。所有は明瞭だが単一語彙のための新規モジュールが増える。

[Answer]: B。E-USSUFD1で2–1、GoA favor 3 / against 0、留保なし。`amadeus-lib.ts` が `UNIT_KINDS` と `UnitKind` を単一定義し、stage-schema、graph、orchestrate、runtime、state、sensorが共有する。既存 `stage-schema → lib` の依存方向とupstream一次差分を保ち、第二定義や単一語彙だけの新moduleを作らない。少数票AはStage Contract Kernelへの凝集を優先したが、lib側frontmatter parserとの循環回避と移植差分最小性を多数票が優先した。根拠: `amadeus/spaces/default/elections/E-USSUFD1/record.md`。

## Q2: 既存12 Unitへどの kind を割り当てるか

kind は「Unit が何を生成するか」ではなく「Unit が何であるか」を表す。`unit-of-work.md` は全 Unit を既存 repository/CLI への embedded とし、常駐 service・UI を0件としているため `service` / `ui` は該当しない。一方、現在の edge block に kind がなく、未指定のままでは全 Unit が full matrix を受けて pruning の実証にならない。

- A: contract/schema を所有する U01=`spec`、distribution/reference artifact を所有する U09/U11=`packaging`、残り U02–U08/U10/U12=`library`。
- B: U01=`spec`、plugin 4分割の外部成果面を強調して U09/U10/U11=`packaging`、残り=`library`。
- C: 現行12 Unitはすべて embedded code として `library`。`spec` / `packaging` は将来 Unit のみで使う。
- D: 今回は全 Unit を untagged のままにし、kind vocabulary と pruning 機構だけを導入する。

[Answer]: A。E-USSUFD2で3–0、GoA favor 3 / against 0、留保なし。U01=`spec`、U09/U11=`packaging`、U02–U08/U10/U12=`library`。U10はpluginの配布物ではなく、inspect/plan/merge/compile/doctor/atomic apply/dropを担うembedded composition engineなので`library`とする。`service`/`ui`は0件、untaggedはpruning実証を失うため採用しない。根拠: `amadeus/spaces/default/elections/E-USSUFD2/record.md`。

## Q3: 追加frontmatter 5 fieldのaccepted language・正規化・invalid境界をどう固定するか

E-OC1分類（`2026-07-20T14:55:47Z`）により、本問は未決design judgmentとして選挙へ付議する。一次正本はupstream `e89162f`（stage-schema extensions）と、既決FR-6 item 18/C1である。全案とも5 fieldはoptional、欠落時はproperty自体を生成せず既存core stageのcompile/emit bytesを変えない。`when`は保存・deferred表示だけで、本intentでは評価しない。未知field・不正型は`ContractResult.invalid`で全mutation前に拒否する。

### 候補

- A（推奨）: upstream exact-shapeを採用する。`number`はstringかつ`^\d+\.\d+$`、`name`/`bundle`は非空string、`required_sections`は非空stringの配列、`when`はexactly-one-key object `{ "producer-in-plan": <non-empty artifact slug> }`だけを受理する。block/inlineの`when`は同じobjectへparseするが、値・配列順・表記を追加trim/sort/dedupeせず保持する。validatorは全field errorを収集し、parse→emit→parseのtyped value同値と、未使用stageのbyte-identicalを保証する。
- B: compatibility優先の緩和shapeを採用する。`number`/`name`/`bundle`は任意の非空string、`required_sections`は非空string配列、`when`は未知predicateを含むsingle-key string mapまで保存し、既知predicateだけdeferred表示する。将来predicateや既存plugin proseを早期受理できるが、誤記・unsupported predicateをcompile時まで運び、fail-closed境界が後退する。
- C: strong canonicalizationを採用する。Aの型制約に加えてstringをtrimし、`required_sections`をtrim・dedupe・canonical sort、`when`をblock formへ統一してemitする。比較とmergeは単純になるが、入力bytes・authored orderを変更し、plugin 0件/既存stage byte-identicalと衝突する可能性がある。

### failure / compatibility tradeoff

- Aはupstream移植差分が最小で、unknown predicate・空値・複数keyを最も早いC1境界でloud rejectする。将来predicate追加にはclosed list更新が必要。
- Bはforward compatibilityが高い一方、未実装の意味をvalid graphへ混入させ、consumerごとの解釈差を招く。
- Cはcanonical bytesを強くできる一方、既存authoring bytesを変えるためNFR-3互換性リスクが最大。

### 必要test

- 各fieldのabsent、valid、wrong type、空string、`number` malformed、`when` 0/2 key・unknown predicate・空artifact slug、`required_sections`空要素をtable-drivenで対照する。
- block/inline `when`のtyped-value同値、parse→emit→parse同値、invalid時のgraph/state/audit/generated bytes不変を検証する。
- 追加fieldを持たない全core stageのcompile/emit byte-identicalと、plugin stageの`required_sections`順序保持をgolden testで固定する。

[Answer]: A。E-USSU01FD3で3–0、GoA favor 3、留保なし。upstream `e89162f` exact-shapeを採用し、追加canonicalizationを行わない。全5 fieldはoptionalで、`number`は`^\d+\.\d+$`のstring、`name`/`bundle`は非空string、`required_sections`は非空string配列、`when`はexactly-one-keyの`producer-in-plan` + 非空artifact slugだけを受理する。block/inline `when`は同じtyped objectへparseするが、値・配列順・表記を追加trim/sort/dedupeしない。欠落keyはproperty自体を生成せず、既存core stageのbytesを不変にする。根拠: `leader/amadeus/spaces/default/elections/E-USSU01FD3/record.md`。
