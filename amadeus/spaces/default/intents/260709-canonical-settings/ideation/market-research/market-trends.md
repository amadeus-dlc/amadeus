# Market Trends — 設定ファイル設計の潮流

> 上流入力: `../intent-capture/intent-statement.md`。規制シフトは本 intent に該当なし。以下は開発ツール生態系の観測トレンド(出典: 各プロジェクト公式ドキュメント/JSON Schema Store)。

## 関連トレンド

1. **スキーマ検証付き設定の標準化** — 主要ツールの設定は JSON Schema(schemastore.org 収載)を伴うのが事実上の標準になり、エディタ補完・未知キー警告・CI 検証が同じスキーマから駆動される。「型と validation を TypeScript 側で明示する」という Issue #623 の期待はこの潮流に沿う。
2. **多形式探索からの回帰** — cosmiconfig 型の「どこでも・何形式でも」から、固定パス・単一形式(biome.json、deno.json)への回帰が進む。理由は本 intent の問題定義と同型: 置き場所と形式の自由度はドリフトと診断困難を生む。
3. **parse-don't-validate 型ローダー** — 設定を「検証してから使う」のではなく「パースして型付きの値として運ぶ」設計(zod 等の普及が牽引)が一般化。プロジェクトの functional-domain-modeling-ts 決定(ブランド型+スマートコンストラクタ)と同じ方向。
4. **AI ハーネスのツール別設定ディレクトリの定着** — `.claude/`、`.codex/`、`.kiro/` のようなハーネス占有ディレクトリが常態化し、複数ハーネス併用リポジトリでは「同じ意味の設定が N 箇所に散る」問題が新たに顕在化している。ハーネス中立層(Amadeus の `amadeus/` ワークスペース)に機械的設定の canonical を置くのは、この新しい問題への応答であり、既存ツールにそのまま模倣できる先行例は少ない(= Amadeus 文脈での差別化点)。

## テーブルステークスと差別化

- **テーブルステークス**(比較群で当然視される要素): 固定された1パス、1形式、全キー省略可能な内蔵既定値、不備時の位置情報付きエラー。
- **差別化**(Amadeus 文脈): N ハーネスで同一 semantics(ハーネス別設定は起動・hooks・権限・env bridge に限定)、`--doctor` 統合による不備検出、space 単位のスコープ(`amadeus/spaces/<space>/`)。
