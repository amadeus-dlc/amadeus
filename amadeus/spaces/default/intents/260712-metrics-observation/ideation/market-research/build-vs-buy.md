# Build vs Buy — メトリクス定点観測(260712-metrics-observation)

## 判断: **Build(既存資産の再利用による薄い自作)を推奨**

| 判断軸 | Buy(SonarCloud/CodeClimate 等) | Build(既存経路の snapshot 化) |
|---|---|---|
| 依存 | 外部 SaaS の新設(project.md Forbidden「ランタイム依存の無断追加禁止」の趣旨に反する方向) | 追加依存ゼロ(lizard は #837 で導入済み・pin 済み、他は Bun/git 標準) |
| 物差し | ゲート(lizard)と別定義の複雑度 → 二重物差し | ゲートと同一経路 → 値の一貫性が構造的に保証 |
| 要望形 | 履歴は SaaS 側 DB(リポジトリ内台帳にならない) | #921 の要望どおり「ファイルに出力」— git 監査可能 |
| コスト | サービス設定・権限・通知の運用が恒常 | 実装1回+CI 1ステップ。観測は読むだけ |
| 可視化 | ◎ ダッシュボード即時 | △ 当面は JSON/表(可視化は #921 でも「論点」扱い — 必要になれば後付け可能) |
| 既保有 | **Codecov は導入済みでカバレッジ・トレンドを既に提供** | — |

## ハイブリッドの明確化(重複構築の回避)

- **カバレッジの時系列可視化は Codecov(既保有)に委ねる** — 新規に作らない。
- **リポジトリ内 snapshot には lcov 集計値(数値)を含めうる**(オフライン参照・横断集計のため)— これは可視化の重複ではなく台帳の完全性。採否は requirements で確定。
- CCN 分布・LOC・ファイル/関数数・テスト数・dist サイズは既保有ツールに時系列機構がないため Build 対象。

## リスクと緩和

- **Build の陳腐化リスク**(メトリクス追加のたびに手を入れる): snapshot スキーマを「計測器名→値の平坦な辞書」にして計測器の追加を疎結合にする(設計論点として requirements へ)。
- **lizard 断片計測の癖**(cid 記録済み): ゲートと同じ癖を共有する=値の解釈が既知、むしろ物差し統一の利点。癖の存在は snapshot スキーマの注記に残す。
- **CI 時間の増分**: lizard・lcov とも既存 CI で実行済みのため、増分は「保存」のみ(実質ゼロ)。cron 型を採る場合のみ新規実行コストが生じる — トリガー選定の論点として requirements へ。

## 結論

導入済みの Codecov を「買い」として温存しつつ、それ以外のメトリクスは**既存計測経路の出力を保存する薄い Build** が、依存ゼロ・物差し統一・要望形一致・コスト最小の全軸で優位。フルスイートの外部品質 SaaS 導入は現時点で不採(必要性が顕在化すれば将来再検討 — 本判断は「恒久の正」と断定しない)。
