# 実践トレンド — コードベース健全性の定点観測(260712-metrics-observation)

> 本書は実践動向の整理。一般動向の記述は業界で広く共有された実践(確信度: 高)だが、統計的な市場データの引用は行わない(出典を伴わない数値主張を避ける — ideation Evidence Standards)。

## 1. 「ゲート+観測」の二層構成が標準形

回帰防止(ゲート、PR 単位で赤/緑)と傾向観測(時系列、リリース/期間単位)を別レイヤーとして持つ構成が定着している。ゲートだけでは「境界内でのじわじわした劣化」(閾値未満の複雑度増・カバレッジの緩やかな低下)が見えない — 本 intent の課題認識はこの標準形の観測レイヤー欠落に相当する。

## 2. Metrics-as-Code: リポジトリ内に履歴をコミットする流儀

SaaS ダッシュボード一辺倒から、計測値スナップショットをリポジトリ自体へコミットする流儀(metrics-as-code)が CI 文化の中で広がっている。利点: (a) 履歴がコードと同じ精度で監査可能(git blame/diff で「いつ・どの変更で動いたか」を特定) (b) オフライン・ツール非依存で集計可能 (c) ベンダーロックインなし。本リポジトリの監査重視文化(audit-first、P2 実測主義)と親和的。

## 3. 計測は「既存ゲートの物差し」を再利用するのが定石

観測用に別の計測器を導入すると、ゲートと観測で値が食い違い「どちらが正か」の調停コストが生じる(本 repo では lizard CCN の断片計測仕様(cid 記録済み)のような計測器固有の癖が既知 — 物差しを増やすとこの癖の管理も倍になる)。ゲートの計測経路から観測値を派生させるのが実務上の定石。

## 4. スナップショット粒度は「マージ単位」が主流、cron は補完

計測トリガーは (a) main マージ時(変更と値の因果が最も明瞭)(b) 定期 cron(変更がなくても環境・依存の変動を拾う)(c) 手動、の3択で、主流は (a)。本 repo はトランクベース+スカッシュマージで「main 1コミット=1 Bolt」が成立しており、マージ単位 snapshot の因果対応が特に綺麗に出る条件が揃っている。

## 5. 追記型 vs 日付別ファイル

追記型単一台帳は集計が容易だが並行 PR の競合点になる(本チームは cid:shared-ledger-insert-collision として実測済み)。日付別/コミット別の個別ファイル+読み時集計が、並行開発チームでの趨勢。

## 本 intent への示唆

- 観測レイヤーの新設は標準形への追随であり、過剰投資ではない(課題実在は intent-statement の実測どおり)
- 計測は complexity-gate の lizard 経路・coverage:ci の lcov 経路の**再利用**を第一候補にする(物差し統一)
- トリガーは main マージ時を第一候補、保存形式は競合安全側(個別ファイル)を第一候補として requirements の論点に渡す — いずれも決定は requirements/design(Issue #921 の委譲どおり)
