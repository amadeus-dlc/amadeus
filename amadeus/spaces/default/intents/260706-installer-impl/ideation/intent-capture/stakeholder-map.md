# Stakeholder Map — インストーラの実装

> ステージ: intent-capture (Ideation) / 作成: 2026-07-07
> 出典: `intent-capture-questions.md` の合意済み回答(Q2 ほか)および intent-statement.md

## Key Stakeholders

| ステークホルダー | 関心事 | 区分 |
|------------------|--------|------|
| 新規ユーザー(OSS利用者) | クローン不要・1コマンドでの導入。導入手順の分かりやすさ | 受益者 |
| 既存ユーザー(導入済みプロジェクト) | 自身のカスタマイズを失わない安全なアップグレード | 受益者 |
| フレームワークメンテナ(このリポジトリのチーム) | 配布・更新経路の保守コスト。npm 公開の運用負荷。README/ドキュメントの簡素化 | 意思決定者 |
| npm レジストリ運用(パッケージ公開権限の保有者) | パッケージ名の確保、公開ポリシー、リリースフロー統合 | 影響者(前提条件の鍵) |
| 組織内プラットフォームエンジニア | 複数プロジェクトへの一括展開 — **初回スコープ外**だが将来の拡張顧客 | 影響者(将来) |

## Decision-Makers vs. Influencers

- **意思決定者**: フレームワークメンテナ — 提供形態(npm 公開)の最終判断、リリースフローへの統合、衝突ポリシーの承認
- **影響者**: npm 公開権限の保有者(公開可否がQ3の決定の成立条件)、既存ユーザー(非破壊マージ要件の妥当性を左右)

## Communication Requirements

- **新規/既存ユーザー向け**: README の導入セクションをインストーラ手順に置き換える(成功指標2)。アップグレード手順とバージョン互換性の告知は CHANGELOG と README で行う
- **メンテナ向け**: インストーラのリリースは既存のバージョンタグ運用(パッチ積み上げ → マイナーカット)に統合し、別系統のリリースフローを作らない
- **npm 公開の前提確認**: feasibility ステージまでにパッケージ名と公開権限の確認結果を意思決定者へ報告する
