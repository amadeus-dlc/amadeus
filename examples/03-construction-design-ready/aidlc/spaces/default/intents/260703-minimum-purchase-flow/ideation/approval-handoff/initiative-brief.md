# Initiative Brief：260703-minimum-purchase-flow

## 目的と成功条件

利用者が商品を選択して注文を作成できる最小の購入フローを実現する。

成功は次で観測する。

- 購入者が Web UI で商品を選択できる。
- 購入者が選択した商品から注文を作成できる。
- 作成した注文がリレーショナルデータベースに記録される。
- 注文作成の過程で、在庫管理システムの在庫情報を REST API 経由で参照できる。

## スコープ境界

対象: 商品選択、注文作成、在庫参照（在庫管理システムとの REST API 連携）。

対象外: 決済手段の拡張、会員ランク別価格、注文キャンセル、販売管理者向けの機能（役割と関心が未確認）。

|---|---|

## 制約

- TypeScript と Node.js の Web アプリケーションとして構築する。
- 購入者向け Web UI を提供する。
- 在庫管理システムとは REST API で連携する。
- 注文はリレーショナルデータベースに記録する。

詳細は [constraint-register.md](../feasibility/constraint-register.md) に記録した。

## 体制

単独開発者で開発する。
チーム構成の判断が意味を持たないため、Team Formation は実行していない。

## モック

- 低忠実度ワイヤーフレーム: [wireframes.md](../rough-mockups/wireframes.md)（商品一覧、注文内容確認、注文完了の 3 画面）
- ユーザーフロー: [user-flow.md](../rough-mockups/user-flow.md)（商品選択から注文作成まで）

## Inception への引き継ぎ

次の未確認事項を Inception で確定する。

- 在庫管理システムの API 仕様と接続条件。
- 在庫参照に失敗した場合の注文作成の振る舞い。
- 購入者の特定方法（認証の要否）。
- 販売管理者の役割と関心。
- 数量の指定方法と基本の決済の扱い。

順序の方針はリスク先行であり、統合制約である在庫参照を早い段階で通す。
