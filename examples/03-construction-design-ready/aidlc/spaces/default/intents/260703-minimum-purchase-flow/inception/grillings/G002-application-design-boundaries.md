# G002: アプリケーション設計の境界確定

## 概要

- 状態: completed
- 反映先: [application-design/components.md](application-design/components.md)

Application Design で、コンポーネントの責務境界とサービスの分割単位を確定した。
実行指示により人間への逐次質問を行わず、推奨回答を実行指示による事前承認として確定した。

## 確定判断

| ID | 判断 | 状態 | 反映先 | 置き換え先 |
|---|---|---|---|---|
| GD005 | UI、サービス層、ドメイン（商品カタログ、在庫参照、注文）の 3 層で構成する | active | [application-design/components.md](application-design/components.md) | |
| GD006 | 在庫参照を独立コンポーネントにし、商品一覧サービスと注文作成サービスで共有する | active | [application-design/services.md](application-design/services.md) | |
| GD007 | サービスは商品一覧サービスと注文作成サービスの 2 つに分割する | active | [application-design/services.md](application-design/services.md) | |

## 質問記録

### Q001

- 確認したいこと: 新しいコンポーネントの責務境界はどこか。
- 確認が必要な理由: 責務境界は Units Generation の Unit 境界の材料になり、後続の分割判断を左右する。
- 推奨回答: UI、サービス層、ドメインの 3 層で構成する。
- 推奨理由: 画面の関心、業務の流れ、外部連携と永続化を分離でき、在庫参照失敗の判定（GD001）を UI から切り離せる。
- ユーザー回答: 実行指示による事前承認（推奨回答を採用）。
- 確定判断: GD005

### Q002

- 確認したいこと: 在庫参照の置き場所をどうするか。
- 確認が必要な理由: 在庫参照は商品一覧（R006）と注文作成（R007、R008）の両方で使われ、API 仕様が未確認の外部連携であるため、重複させると変更の影響が広がる。
- 推奨回答: 独立コンポーネントにして両サービスで共有する。
- 推奨理由: 統合制約である在庫管理システムとの連携と失敗検出を一箇所へ集約できる。
- ユーザー回答: 実行指示による事前承認（推奨回答を採用）。
- 確定判断: GD006

### Q003

- 確認したいこと: サービスの分割単位はどうするか。
- 確認が必要な理由: サービスの分割は Unit の境界と実装順序に影響する。
- 推奨回答: 購入フローの 2 つの局面（選ぶ、注文する）に合わせて 2 分割する。
- 推奨理由: 要求の境界（R001/R006 と R004/R005/R007/R008）と一致し、単一サービスより責務が明確になる。
- ユーザー回答: 実行指示による事前承認（推奨回答を採用）。
- 確定判断: GD007
