## B001：注文作成の貫通（walking skeleton）

商品選択から注文作成までを、UI、サービス層、リレーショナルデータベースの全層で貫通させる最小スライスである。
在庫状況の表示と在庫にもとづく選択可否は B002 が扱うため、B001 の商品一覧は商品名と価格を表示し、注文作成の入力として商品を選択できるところまでを扱う。

Definition of Done:

- 購入者が商品一覧画面で商品名と価格を閲覧し、商品を選択できる（R001 の商品名と価格の表示、R002 の選択操作）。
- 購入者が数量を 1 以上の整数で指定し、注文内容（商品名、数量、金額、合計金額）を確認できる（R003）。
- 注文作成操作で注文がリレーショナルデータベースに記録され、注文完了画面に注文の識別子が表示される（R004、R007）。
- 上記が実装とテストコードを伴い、ビルドとテストが通る。

confidence hypothesis: TypeScript と Node.js の Web アプリケーションで、UI からサービス層、リレーショナルデータベースまでの全層が 1 本の経路で成立することを証明する。
以降の Bolt は、この骨格への追加として実装できる。

## B002：在庫参照の統合

在庫管理システムとの REST API 連携を成立させ、商品一覧の在庫状況表示と参照失敗時の縮退を足すスライスである。

Definition of Done:

- 商品一覧画面が、在庫管理システムの REST API から参照した在庫状況を商品ごとに表示する（R001、R005）。
- 在庫がない商品は選択不可として表示され、選択操作ができない（R002）。
- 在庫参照に失敗した場合、在庫情報を取得できない旨が表示され、在庫を確認できない商品では注文を作成できない（R006）。
- 上記が実装とテストコードを伴い、ビルドとテストが通る。

confidence hypothesis: 統合制約である在庫管理システムとの REST API 連携が実際に成立し、API 仕様の不確実性が解消されることを証明する。
