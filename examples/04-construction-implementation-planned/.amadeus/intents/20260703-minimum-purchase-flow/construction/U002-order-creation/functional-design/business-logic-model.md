# Business Logic Model：注文作成

## 目的

U002 注文作成の業務ロジックを、Code Generation が実装計画に使える粒度で確定する。

注文という新しいデータモデルと、在庫参照の結果に基づく注文可否という業務ルールがあるため、この Unit では Functional Design を実行する。

## 対象 Unit

U002 注文作成（C002 注文内容確認画面、C003 注文完了画面、C005 注文作成サービス、C008 注文リポジトリを含む）。

Bolt B001（注文作成の walking skeleton）に束ねられ、対応する要求は R003、R004 である。

## 業務ロジック

注文作成サービスが、次の流れで注文の作成と記録を進める。

1. 注文内容確認画面（C002）が、商品閲覧（U001）から受け渡された選択した商品と数量を受け取り、商品名、数量、価格を表示する（R003）。
2. 注文作成サービス（C005）が、選択した商品の価格に数量を乗じて合計金額を算出する。画面表示と注文の記録で同じ算出結果を使う（R003、Application Design の設計判断 5）。
3. 注文作成の操作を受けたら、注文作成サービスが注文を生成し、一意の注文番号を発行して、注文リポジトリ（C008）へ記録を指示する（R004）。
4. 注文リポジトリが注文をリレーショナルデータベースへ記録し、記録した注文は注文番号で参照できる（R004、tech.md）。
5. 作成の成功時は、注文完了画面（C003）が注文番号を表示する（R004）。
6. 注文の作成に失敗した場合は、注文完了画面へ遷移せず、失敗したことを注文内容確認画面に表示して留まる（interaction-spec.md）。
7. 注文内容確認画面から商品一覧へ戻る操作では、注文を作成しない（R003）。

注文作成は購入の申し込みの記録までとし、決済を含めない（GD002）。

在庫参照の結果に基づく注文可否の判定点は、商品選択の境界（U001 商品閲覧と U003 在庫参照連携）に置かれているため、注文作成サービスは商品一覧の選択操作を通過した商品だけを注文の作成対象にする（R002、R005、GD006）。
B001 では在庫参照連携が未接続であり、この判定は B002 で完成する。

## 入力

- 選択した商品と数量（商品の識別子、商品名、価格、数量。U001 商品閲覧から画面遷移で受け渡される）

## 出力

- 注文番号（作成の成功時に発行され、注文完了画面に表示される）
- 記録された注文（注文番号で参照できる）
- 作成に失敗したことの表示（失敗時）

## 未確認事項

- 注文番号の形式は未確認である。mockups.md の表示例（ORD-XXXXXXXX）は仮置きであり、一意であることだけを要求として扱う。形式は Code Generation で確定する。
- リレーショナルデータベースの具体的な製品は未確認である（tech.md）。
