# Bolt Plan：最小購入フロー

体制は単独開発者であり、Team Formation を実行していないため、担当割り当ては該当なしである。
Bolt の承認は PR の人間 merge で行う。

## 一覧

| ID | Bolt | 束ねる Unit | 順序 |
|---|---|---|---|
| B001 | 注文作成の walking skeleton | U001 商品閲覧、U002 注文作成 | 1 |
| B002 | 在庫参照連携 | U003 在庫参照連携 | 2 |

## B001：注文作成の walking skeleton

商品選択から注文作成までの一連の流れを、画面、サービス、永続化の全層を貫通する最小スライスとして成立させる。
この Bolt では在庫状況の表示を含めず、商品一覧は商品名と価格だけを表示する。

Definition of Done:

- 購入者が商品一覧から商品を選択し、数量を指定して注文内容を確認できる（R002、R003。R001 は在庫状況の表示を除く）。
- 購入者が注文を作成すると、一意の注文番号が発行され、注文完了画面に表示される（R004）。
- 作成した注文がリレーショナルデータベースに記録され、注文番号で参照できる（R004）。
- 画面、サービス、永続化の各層が実際に接続されて動く。
- Bolt PR を人間が承認して merge する（walking skeleton ゲート）。

confidence hypothesis: TypeScript と Node.js の構成で、画面から注文の記録までがひと続きに動き、最小の購入体験（成功条件「購入者が商品を選択して注文を作成できる」）が成立することを証明する。

## B002：在庫参照連携

在庫管理システムとの REST 連携を組み込み、在庫状況を注文作成の判断材料にする。

Definition of Done:

- 商品一覧の在庫状況が、在庫管理システムから参照した在庫情報を反映する（R001、R005）。
- 在庫のない商品を選択できない（R002、R005）。
- 在庫情報を参照できない場合、参照できないことが表示され、注文作成を受け付けない（R006）。
- Bolt PR を人間が承認して merge する。

confidence hypothesis: 在庫管理システムとの REST 連携が成立し、成功条件「注文作成の判断に在庫管理システムの在庫情報を参照できる」が観測できること、参照不能時の縮退が設計どおりに動くことを証明する。
