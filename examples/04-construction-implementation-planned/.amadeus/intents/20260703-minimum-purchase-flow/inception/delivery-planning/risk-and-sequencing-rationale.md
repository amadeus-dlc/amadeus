# Risk and Sequencing Rationale：最小購入フロー

## 順序付けの根拠

順序付けの優先は、Ideation の判断 D003 に従い価値先行にする。

B001 は、人間の指示により注文作成を貫通する walking skeleton とする。
U001 商品閲覧と U002 注文作成を束ねると、商品選択から注文作成までの一連の流れが最初の Bolt で成立し、最小の購入体験という価値を最も早く観測できる。

U003 在庫参照連携を B002 に分離した理由は 2 つある。
第一に、在庫管理システムの連携仕様が未確認であり（RAID Log の依存）、未確定の外部依存を walking skeleton の進行から切り離すためである。
第二に、U003 は U001 だけに依存するため、B001 の完了後に独立して積み増せるためである。

この順序は Unit の依存 DAG（U002 が U001 に依存、U003 が U001 に依存）と矛盾しない。

## リスク

| リスク | 影響する Bolt | 対応 |
|---|---|---|
| 在庫管理システムの連携仕様が未確定のまま残る | B002 | B001 の実行中に連携仕様と確認先を確定する。確定できない場合は B002 の着手前に停止し、人間に確認する |
| Web UI フレームワークとリレーショナルデータベースの製品が未確定である | B001 | Construction の NFR Requirements（tech-stack-decisions）で確定してから Code Generation に入る |
| 商品情報の登録手段が対象外であり、動作確認用の商品データがない | B001 | 動作確認用の初期データ投入を B001 の Build and Test の手順に含める |
| 注文作成の失敗時の振る舞いの詳細が未確定である | B001 | Construction の Functional Design（業務ルール）で確定する |
