# Memory: requirements-analysis

## Interpretations

- 成功条件「注文作成の過程で在庫情報を参照できる」を、商品一覧での在庫参照（R006）と注文作成時の在庫確認（R007）の 2 つに分けた。参照の契機が異なり、受け入れ条件の観測方法も異なるためである。
- RAID Log のリスク「在庫管理システムが応答しない場合の注文作成の可否が未定義」を、R008 として要求に昇格した。

## Deviations

- 実行指示により逐次質問を行わず、指示内容と既存成果物から回答を確定した。質問と回答は requirements-analysis-questions.md に、要求の意味に影響する確定判断は inception/grillings.md（G001）に記録した。

## Tradeoffs

- 在庫参照失敗時は注文作成を止める安全側の振る舞いを選んだ。在庫不明のまま注文を受けると過剰受注のリスクがあり、最小フローでは可用性より整合性を優先した。
- 認証を導入しない判断は、スコープ最小化を優先した。会員制度の前提が未確認のため、購入者の特定を注文番号に委ねた。

## Open questions

- 在庫管理システムの API 仕様（エンドポイント、認証方式、応答形式）と接続条件。
- 販売管理者の役割と関心（対象外の再判断に影響する）。
