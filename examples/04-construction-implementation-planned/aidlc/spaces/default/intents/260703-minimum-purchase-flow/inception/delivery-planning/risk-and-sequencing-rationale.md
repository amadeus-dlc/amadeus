# Risk and Sequencing Rationale：260703-minimum-purchase-flow

## 順序付けの根拠

B001 を walking skeleton として最初に置き、B002 をその直後に置く。

B001 は、実行指示で指定された注文作成の貫通を扱う。
注文作成の経路（UI、サービス層、リレーショナルデータベース）は外部システムに依存せず内部で閉じるため、外部確認を待たずにアーキテクチャの骨格を証明できる。

B002 は、Ideation の順序方針が早期に通すとした在庫参照を扱う。
在庫管理システムの API 仕様が未確認のまま walking skeleton に含めると、B001 が外部確認待ちでブロックされる。
そのため在庫参照を B001 から分離し、骨格の成立直後に統合リスクを解消する順序にした。
この分離は、リスク先行の方針を「未確認の外部依存を最初の Bolt に入れない」形で適用したものである。

## リスク

| リスク | 影響する Bolt | 対応 |
|---|---|---|
| 在庫管理システムの API 仕様が設計の想定と異なる。 | B002 | 在庫参照コンポーネントに連携の影響を閉じ込める設計で吸収し、B002 の着手前に API 仕様と接続条件を確認する。 |
| B001 の時点では在庫状況を表示しないため、在庫の裏付けのない注文を作成できる。 | B001 | B001 は骨格の証明を目的とし、購入者への公開は B002 の完了後に判断する。 |
| リレーショナルデータベースの製品選定が未確認である。 | B001 | Construction の NFR Requirements の技術スタック判断で確定してから Code Generation に入る。 |
| 単独開発者のため作業が直列になり、Bolt の切り替えコストが進捗に直結する。 | B001、B002 | Bolt を 2 個に抑え、walking skeleton の承認後に残りを進める。 |
