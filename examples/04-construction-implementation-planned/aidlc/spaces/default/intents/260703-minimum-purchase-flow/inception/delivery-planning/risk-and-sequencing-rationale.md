# Risk and Sequencing Rationale：260703-minimum-purchase-flow

## 順序付けの根拠

順序付けの優先は、Ideation の D004（リスク先行）を踏襲した。
統合制約であり API 仕様が未確認の在庫参照（U001）を、B001 内の最初の Unit として最も早く通す。

Bolt を単一の B001 にしたのは次の理由による。

- 依存 DAG（U003 → U001, U002）により、注文作成を貫通する walking skeleton には全 Unit の最小実装が必要である。
- U001 や U002 だけを先行 Bolt にしても、購入者が観測できる価値（注文を作成できる）を出荷できず、Bolt を分ける経済的な利点がない。
- 単独開発者のため、Bolt 間の並行実施による短縮効果もない。

Bolt 内の Unit の実施順序は依存 DAG に従う: U001（在庫参照）→ U002（商品選択）→ U003（注文作成）。

## リスク

| リスク | 影響する Bolt | 対応 |
|---|---|---|
| 在庫管理システムの API 仕様と接続条件が未確認である | B001 | U001 の着手時に仕様を確認する。確定しない場合は在庫参照の契約（在庫を参照する: 商品識別子の一覧 → 在庫状況）を仮置きし、スタブに対して実装を進め、仕様確定後に接続部分だけを差し替える |
| 在庫参照失敗時の振る舞いの作り込みが漏れる | B001 | GD001 と R007、R008 の受け入れ条件を Definition of Done に含めた。Build and Test で例外系を検証する |
| 単独開発者のためレビューが開発者自身に集中する | B001 | walking skeleton ゲートで設計成果物と生成コードをまとめて人間が承認する |
| Web フレームワーク、ORM、UI ライブラリが未選定である | B001 | Construction の NFR Requirements（Stage 3.2）で技術スタックを確定してから Code Generation に入る |
