# Code Generation Plan：注文作成

B001 注文作成の貫通（walking skeleton）における U003 注文作成の実装計画である。
入力は functional-design の 4 成果物、requirements.md（R003、R004）、application-design の成果物である。
U001 商品選択の実装計画が確定したプロジェクトの土台（package.json、tsconfig.json、Express、node:test）を前提にする。
パスは対象リポジトリのルートからの相対パスである。

## 実装判断

| # | 判断 | 理由 |
|---|---|---|
| 1 | リレーショナルデータベースは SQLite にする | external-dependency-map.md の「注文の記録に必要な最小構成から選ぶ」への判断であり、business-logic-model.md と domain-entities.md の未確認事項（製品はこの実装計画で確定して記録する）をここで確定する。SQLite は単一ファイルで動きサーバーの構成が不要であり、walking skeleton の全層貫通（UI、サービス層、リレーショナルデータベース）の証明に足りる。代替のサーバー型（PostgreSQL など）は、walking skeleton の時点では接続先の構成が過剰である |
| 2 | データベースアクセスは Node.js 標準の node:sqlite モジュールを使う | 追加依存なしで SQLite を扱え、注文の記録に必要な最小構成という判断 1 の基準と一致する |
| 3 | 注文の識別子は、保存と同じトランザクションの中で当日の注文数から日内連番を採番して組み立てる | business-rules.md の 5（保存時に発行し、一意にする）と 6（ORD-作成日-日内連番 4 桁）に従う。保存と同じトランザクションで採番するため、同時の注文作成でも連番が重複しない |
| 4 | 数量の検証と金額の算出は、画面から独立した注文作成の関数に置く | component-methods.md の注文作成の責務に合わせる。画面を起動せずに業務ルール（business-rules.md の 1、2）をテストできる |
| 5 | 数量の変更に応じた金額と合計金額の更新は、注文内容確認画面の中の最小のスクリプトで行い、検証と算出の定義元はサーバー側の注文作成にする | interaction-spec.md の数量の指定（変更で金額と合計金額を更新する）を、フロントエンドのビルドパイプラインを増やさずに満たす。画面側の更新は表示の補助であり、注文の作成時はサーバー側で数量を再検証する |
| 6 | 注文完了画面は、注文の識別子を画面遷移（リダイレクト）で受け取る | component-dependency.md の「注文完了画面はサービスを呼ばない」に従う。注文の作成後にリダイレクトするため、完了画面の再読み込みが注文の重複作成にならない |

## 変更順序

1. SQLite への接続と注文テーブル（order-database）を作る。
2. 注文作成（数量の検証、金額と合計金額の算出、注文の組み立て）を実装し、テストを書く。
3. 注文記録（注文の保存と識別子の発行）を実装し、テストを書く。
4. 注文サービスで注文作成と注文記録をまとめる。
5. 注文内容確認画面と注文完了画面を実装し、テストを書く。
6. ルーティングへ注文内容確認画面、注文の作成、注文完了画面の 3 経路を追加し、商品一覧画面の選択操作から注文完了画面までの遷移をつなぐ。

## 検証方法

テストの実行と結果の記録は Stage 3.6 Build and Test の責務であり、ここでは確認する観点だけを定める。

- ビルド（tsc）が通る。
- 数量の検証が 1 以上の整数だけを受け付け、0 以下と整数以外を拒否する（R003、business-rules.md の 1）。
- 金額が価格 × 数量、合計金額が金額の合計として算出される（R003、business-rules.md の 2）。
- 注文が商品（商品識別子と、注文作成時点の商品名、価格の写し）、数量、金額、合計金額、作成日時とともに SQLite に記録され、ORD-作成日-日内連番 4 桁の一意の注文の識別子が発行される（R004、business-rules.md の 5〜8）。
- 同じ作成日の 2 件目の注文で日内連番が増える（business-rules.md の 6）。
- 記録に失敗した場合、注文の識別子が発行されず、注文内容確認画面に留まって注文が作成されなかった旨が表示される（business-rules.md の例外、interaction-spec.md の注文の作成）。
- 注文内容確認画面に商品名、数量、金額、合計金額が表示され、数量の既定値が 1 である（R003、bolt-plan.md の B001 の Definition of Done）。
- 注文完了画面に注文を受け付けた旨と注文の識別子が表示される（R004）。
- 認証と決済の操作なしで注文作成が完了する（R004、business-rules.md の 9）。
