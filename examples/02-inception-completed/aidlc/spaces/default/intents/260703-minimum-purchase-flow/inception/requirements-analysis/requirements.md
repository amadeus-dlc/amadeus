# Requirements：260703-minimum-purchase-flow

## 一覧

| ID | 要求 | 由来する成功条件 |
|---|---|---|
| R001 | 商品一覧の表示 | 購入者が Web UI で商品を選択できる |
| R002 | 商品選択と数量指定 | 購入者が Web UI で商品を選択できる |
| R003 | 注文内容の確認 | 購入者が選択した商品から注文を作成できる |
| R004 | 注文作成と注文番号の提示 | 購入者が選択した商品から注文を作成できる |
| R005 | 注文の永続化 | 作成した注文がリレーショナルデータベースに記録される |
| R006 | 商品一覧での在庫参照 | 在庫管理システムの在庫情報を REST API 経由で参照できる |
| R007 | 注文作成時の在庫確認 | 在庫管理システムの在庫情報を REST API 経由で参照できる |
| R008 | 在庫参照失敗時の振る舞い | 在庫管理システムの在庫情報を REST API 経由で参照できる |

## R001：商品一覧の表示

購入者は、Web UI で販売対象の商品一覧を閲覧できる。
一覧には、商品ごとに商品名、価格、在庫状況を表示する。

受け入れ条件:

- 商品一覧画面に、商品名、価格、在庫状況が商品ごとに表示される。
- 在庫がない商品は、選択不可であることが表示で判別できる。

## R002：商品選択と数量指定

購入者は、在庫がある商品を選択し、数量を指定できる。
数量の既定値は 1 とし、1 以上の整数だけを受け付ける。

受け入れ条件:

- 在庫がある商品を選択でき、在庫がない商品は選択できない。
- 数量を指定しない場合は 1 として扱われる。
- 0 以下または整数でない数量は受け付けず、その旨が表示される。

## R003：注文内容の確認

購入者は、注文を作成する前に、選択した商品、数量、金額、合計金額を確認できる。

受け入れ条件:

- 注文内容確認画面に、選択した商品の商品名、数量、金額、合計金額が表示される。
- 注文を作成せずに商品一覧へ戻れる。

## R004：注文作成と注文番号の提示

購入者は、確認した注文内容で注文を作成できる。
作成した注文には一意の注文番号が付与され、購入者は注文完了画面で注文番号を確認できる。

受け入れ条件:

- 注文内容確認画面から注文を作成できる。
- 作成した注文に一意の注文番号が付与される。
- 注文完了画面に注文番号が表示される。

## R005：注文の永続化

作成した注文は、リレーショナルデータベースに記録される。
記録した注文は、注文番号で照会できる。

受け入れ条件:

- 注文の作成後、注文（注文番号、商品、数量、金額、合計金額、作成日時）がリレーショナルデータベースに記録されている。
- 注文番号を指定して、記録した注文を照会できる。

## R006：商品一覧での在庫参照

商品一覧の表示時に、在庫管理システムの REST API から在庫情報を参照し、商品ごとの在庫状況に反映する。

受け入れ条件:

- 商品一覧の表示時に、在庫管理システムの REST API へ在庫参照の要求が送られる。
- 参照した在庫情報が、商品一覧の在庫状況の表示に反映される。

## R007：注文作成時の在庫確認

注文の作成時に、在庫管理システムの在庫情報を参照し、注文対象の商品の在庫が確認できた場合だけ注文を作成する。

受け入れ条件:

- 注文の作成時に、注文対象の商品の在庫情報が在庫管理システムから参照される。
- 在庫が不足している場合、注文は作成されず、その旨が購入者に表示される。

## R008：在庫参照失敗時の振る舞い

在庫管理システムから在庫情報を参照できない場合、注文の作成を進めず、購入者に在庫を確認できない旨を表示して再試行を促す。

受け入れ条件:

- 在庫参照に失敗した場合、注文は作成されない。
- 在庫参照に失敗した場合、購入者に在庫を確認できない旨と再試行の案内が表示される。

## 対象外との境界

次の判断で対象外との境界を確定した。判断の経緯は [grillings.md](../grillings.md) に記録した。

- 購入者の認証は導入しない。購入者は認証なしで注文を作成でき、注文の確認は注文番号で行う（GD002）。
- 決済処理は含めない。本 Intent は注文の作成と記録までを対象にする（GD003）。
- 販売管理者向けの機能は対象外のままにする。役割と関心は引き続き未確認である（scope-document.md の対象外に従う）。

## 未確認事項

- 在庫管理システムの API 仕様（エンドポイント、認証方式、応答形式）と接続条件は未確認である。要求は振る舞いのレベルで定義し、API 仕様の確定は Construction の設計ステージで扱う。
