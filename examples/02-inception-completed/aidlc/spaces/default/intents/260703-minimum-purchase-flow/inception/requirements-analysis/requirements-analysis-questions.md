# Questions：Requirements Analysis

実行指示により質問を対話で行わず、推奨回答を実行指示と既存成果物にもとづく判断で確定した。
判断の理由は各行の回答に記録した。

| # | 確認したいこと | 推奨回答 | 回答 |
|---|---|---|---|
| 1 | 在庫参照に失敗した場合、注文作成をどう扱うか | 対象商品を選択不可とし、注文を作成しない | 推奨の通り確定した。注文キャンセルが対象外であり、在庫不明のまま受けた注文を回復する手段がないため、安全側に倒す（R006） |
| 2 | 購入者の特定方法（認証の要否）はどうするか | 本 Intent では認証を実装しない | 推奨の通り確定した。成功条件に認証が含まれず、rough mockups にも認証画面がないため、最小購入フローの範囲外と判断した（R004） |
| 3 | 数量の指定方法はどうするか | 注文内容確認画面で 1 以上の整数を指定でき、既定値は 1 にする | 推奨の通り確定した。wireframes.md の注文内容確認画面の数量列を検証可能にした（R003） |
| 4 | 基本の決済の扱いはどうするか | 決済処理を行わず、注文作成は決済なしで完了する | 推奨の通り確定した。決済手段の拡張はスコープバックログの Could 項目であり、基本の決済の前提自体が未確認のため、本 Intent に含めない（R004） |
| 5 | 在庫管理システムの API 仕様と接続条件は確認できたか | 確認先を特定し、仕様を要求に反映する | 未確認。確認先をこのセッションで特定できないため、要求では接続手段（REST API）だけを規定し、Delivery Planning の外部依存整理へ引き継ぐ（R005） |
| 6 | 販売管理者の役割と関心は確認できたか | 役割と関心を確認し、要求への影響を判断する | 未確認。販売管理者向けの機能はスコープ対象外のため、本 Intent の要求に含めない。確認は継続する |
