# Design Decisions：260703-minimum-purchase-flow

## 一覧

| # | 判断 | 理由 | 代替案 |
|---|---|---|---|
| 1 | UI、サービス層、ドメイン（商品カタログ、在庫参照、注文）の 3 層で構成する（GD005） | 画面の関心、業務の流れ、外部連携と永続化を分離し、Units Generation の Unit 境界の材料にするため | UI から直接 REST API とデータベースを扱う 2 層構成。最小だが、在庫参照失敗の判定（GD001）が画面側へ散らばる |
| 2 | 在庫参照を独立コンポーネントにし、商品一覧サービスと注文作成サービスで共有する（GD006） | 在庫管理システムとの連携は統合制約であり、API 仕様が未確認のまま進むため、変更の影響と失敗検出を一箇所へ集約するため | 各サービスが個別に REST API を呼ぶ。重複が生じ、失敗時の振る舞いが揺れる |
| 3 | サービスは商品一覧サービスと注文作成サービスの 2 つに分割する（GD007） | 購入フローの 2 つの局面（選ぶ、注文する）に対応し、要求の境界（R001/R006 と R004/R005/R007/R008）と一致するため | 単一の購入フローサービス。最小だが、在庫参照だけを使う画面表示と、記録を伴う注文作成で責務が混ざる |
| 4 | 注文番号の採番を注文コンポーネントの責務にする | 注文の一意性（R004）は注文の業務ルールであり、永続化手段から独立させるため | データベースの自動採番に委ねる。記録前に注文番号を提示できず、注文の組み立てが永続化に依存する |
| 5 | 商品カタログは最小の商品情報（商品識別子、商品名、価格）だけを持つ | 商品管理は最小スコープに含まれないため | 商品の登録、更新まで含める。スコープ外の作り込みになる |
