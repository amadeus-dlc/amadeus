# Design Decisions：260703-minimum-purchase-flow

## 一覧

| # | 判断 | 理由 | 代替案 |
|---|---|---|---|
| 1 | サービス層を商品サービスと注文サービスの 2 つに分ける | ドメインの領域（商品選択、注文作成、在庫参照）に合わせて責務を分け、在庫参照は利用先が商品一覧の表示だけなので商品サービスに含める | 在庫参照を独立したサービスにする案。最小購入フローでは呼び出し元が 1 つであり、分割が過剰になる |
| 2 | 在庫参照の失敗を、例外ではなく在庫状況の値（確認不可）で表現する | R006 の縮退を在庫状況の一種として UI まで一貫して伝えられ、呼び出し側の分岐が要求の語彙と一致する | 在庫参照が例外を投げ、UI 側で捕捉する案。縮退時の表示が要求の語彙から離れ、捕捉漏れの余地が生まれる |
| 3 | 注文作成時に在庫の再確認を行わない | R004 の受け入れ条件に在庫の再確認が含まれず、最小購入フローでは選択時の在庫確認（R002、R006）で足りる | 注文作成時に在庫を再参照する案。在庫管理システムの API 仕様が未確認のため、必要になれば Construction で再評価する |
| 4 | 注文の識別子を注文記録の保存時に発行する | リレーショナルデータベースに記録された注文だけが識別子を持つことを保証し、R004 の「記録される」と「識別子が表示される」を対応付けられる | 画面側で先に識別子を発行する案。記録に失敗した注文の識別子が購入者に見える余地が生まれる |
