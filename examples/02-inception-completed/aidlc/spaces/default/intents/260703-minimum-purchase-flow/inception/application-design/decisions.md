# Design Decisions：260703-minimum-purchase-flow

## 一覧

| # | 判断 | 理由 | 代替案 |
|---|---|---|---|
| 1 | UI は商品一覧画面、注文内容確認画面、注文完了画面に分ける。 | rough mockups の 3 つの状態と要求の境界が一致するため。 | 単一画面で完結させる。 |
| 2 | 在庫参照の影響を在庫参照コンポーネントに閉じ込める。 | 在庫管理システムの API 仕様が未確認であり、外部依存の変更を UI と注文作成へ広げないため。 | UI から直接 REST API を呼び出す。 |
| 3 | 注文作成と注文記録を分ける。 | 数量検証とリレーショナルデータベースへの保存は変更理由が異なるため。 | 注文サービス内で保存まで一体化する。 |
| 4 | 在庫参照を最初の walking skeleton には含めない計画にできる設計にする。 | ユーザー指定により B001 は注文作成を貫通する walking skeleton であり、未確認の外部依存で B001 を止めないため。 | B001 に在庫参照を含める。 |
