# Questions：Requirements Analysis

人間の指示が「質問せずに続行し、判断が必要な点は指示と既存成果物から最も妥当な内容を選ぶ」と確認論点への回答方針を先に与えたため、grilling による対話は行っていない。
回答列には、指示と既存成果物から確定した内容を記録する。

| # | 確認したいこと | 推奨回答 | 回答 |
|---|---|---|---|
| 1 | 在庫管理システムの REST API の連携仕様はどこで確認できるか | 連携仕様の確認先を指定する | 確認先は未確認である。要求は商品ごとの在庫状況を参照できる前提で定義し、連携仕様の確定は Delivery Planning の外部依存として管理する |
| 2 | 在庫管理システムから在庫情報を参照できない場合、注文作成をどう扱うか | 在庫を確認できない間は注文作成を受け付けない | 在庫不明として扱い、注文作成を受け付けず、購入者に参照できないことを表示する。R006 と GD001 に記録した |
| 3 | 注文作成と決済の境界はどこか | 注文作成は決済を含まない | 注文作成は購入の申し込みの記録までとし、決済を含めない。R004 と GD002 に記録した |
| 4 | 受け入れ条件は何で観測するか | Web UI の操作結果と記録された注文で観測する | 購入者向け Web UI の操作結果と、記録された注文の注文番号による参照で観測する |
