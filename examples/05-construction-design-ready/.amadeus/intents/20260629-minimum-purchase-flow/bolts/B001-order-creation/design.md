# Construction Design

## 概要

- 対象 Bolt は B001 注文作成である。
- B001 は、注文内容、購入者情報、販売可能在庫の参照結果をもとに、購入申込の記録として注文を作成する。
- Design Gate の evidence は、このファイルを指す。

## Domain Design

**注文作成依頼**は、顧客が確認済みの注文内容で注文作成を依頼した事実を表す。
注文作成依頼は、注文内容、購入者情報、販売可能在庫の参照結果を入力に持つ。

**注文内容**は、顧客が注文作成前に確認した購入対象の内容を表す。
注文内容は、商品選択と購入数量を含む候補として扱う。

**購入者情報**は、注文へ記録する購入者の連絡先または配送に必要な情報を表す。
購入者情報の詳細項目は未確認であるため、Construction では必須情報が不足する場合に注文を作成しない境界を扱う。

**販売可能在庫の参照結果**は、選択商品が注文作成時点で販売可能かを判断するための結果を表す。
B001 は販売可能在庫を参照結果として受け取り、在庫引当は実行しない。

**注文**は、購入者情報と商品選択をもとに作成される購入申込の記録である。
注文は注文作成の事後条件として作成され、決済詳細、売上確定、在庫引当、出荷を含まない。

不変条件は次である。

- 注文作成は、注文内容、購入者情報、販売可能在庫の参照結果がそろう場合だけ成立する。
- 販売可能在庫の参照結果が販売不可を示す場合、注文は作成しない。
- 注文作成時に、決済詳細、売上確定、在庫引当、出荷を起動しない。

## Logical Design

B001 の入力は、B002 で確認済みの注文内容、購入者情報、販売可能在庫の参照結果である。
B001 は B002 の確認結果に依存するが、B002 の画面や確認処理そのものは実装しない。

注文作成の流れは次である。

1. 注文作成依頼を受け取る。
2. 注文内容、購入者情報、販売可能在庫の参照結果が注文作成に使えるかを判定する。
3. 販売可能在庫の参照結果が販売可能であることを確認する。
4. 注文へ記録する内容を確定する。
5. 購入申込の記録として注文を作成する。
6. 注文作成結果として、作成された注文または作成拒否の理由を返す。

失敗条件は次である。

- 注文内容が不足している。
- 購入者情報が不足している。
- 販売可能在庫の参照結果が不足している。
- 販売可能在庫の参照結果が販売不可を示している。

既存コード調査では、対象 workspace にアプリケーションコードは存在しない。
そのため、実装対象のファイル配置、永続化方式、注文番号、注文状態は未確認事項として扱う。

## 実装設計

B001 の実装 Task は、アプリケーションコードが追加される前提で、次の責務に分ける。

- 注文作成入力を表す型またはデータ構造を定義する。
- 注文内容、購入者情報、販売可能在庫の参照結果を検証する。
- 販売可能在庫の参照結果が販売可能な場合に注文を作成する。
- 入力不足または販売不可の場合に注文を作成しない結果を返す。
- 決済詳細、売上確定、在庫引当、出荷を呼び出さない境界を保つ。

注文へ記録する項目は、注文内容、購入者情報、販売可能在庫の参照結果に含まれる販売可否の根拠に限定する。
注文番号、注文状態、永続化方式は未確認であるため、実装時には永続化境界を追加 Task に広げず、注文作成の振る舞いを先に成立させる。

## 検証設計

B001 の検証は、R004 と UC003 の受け入れ条件を確認できる単位で行う。

- 注文内容が不足する場合、注文を作成しないことを確認する。
- 購入者情報が不足する場合、注文を作成しないことを確認する。
- 販売可能在庫の参照結果が不足する場合、注文を作成しないことを確認する。
- 販売可能在庫の参照結果が販売不可の場合、注文を作成しないことを確認する。
