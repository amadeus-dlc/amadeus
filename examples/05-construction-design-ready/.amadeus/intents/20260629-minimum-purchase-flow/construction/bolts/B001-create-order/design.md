# Construction Design

## 概要

- B001 は、注文内容、購入者情報、販売可能在庫の参照結果をもとに注文を作成する Bolt である。
- Design Gate の evidence は、このファイルを指す。
- B001 は R004 と UC003 を主対象にし、R002 と R003 の成果を注文作成の入力として扱う。
- B001 は販売可能在庫の外部参照、在庫引当、決済、売上確定、出荷を開始しない。

## Domain Design

**注文**は、顧客が確認した注文内容を販売管理で作成済みとして扱うための記録である。
注文には、選択商品、数量、購入者情報、注文内容、販売可能在庫の参照結果を記録する。

**注文内容**は、注文作成前に顧客が確認した購入予定の内容である。
注文内容は B002 の成果として B001 に渡される。
B001 は注文内容の確認処理をやり直さず、確認済みであることを事前条件として扱う。

**販売可能在庫の参照結果**は、注文作成へ進めるかどうかを示す入力である。
B001 は参照結果が注文作成可能である場合だけ注文を作成する。
B001 は在庫数の更新、在庫引当、棚卸し、入荷を扱わない。

**購入者情報**は、注文へ記録する購入者の情報である。
B001 は購入者情報を注文へ転記する。
B001 は会員登録、ログイン、顧客台帳、購入履歴管理を扱わない。

**注文作成結果**は、注文作成の成否と、作成済み注文または作成不可の理由を表す。
作成不可の理由は、注文内容が未確認であること、販売可能在庫の参照結果が作成不可であること、購入者情報が注文作成に不足していることを区別できる粒度にする。

不変条件は、注文作成が決済、売上確定、在庫引当、出荷を開始しないことである。

## Logical Design

B001 の入力は、確認済み注文内容、購入者情報、販売可能在庫の参照結果である。
入力は B001 の境界で一つの注文作成依頼として受け取る。

B001 は、最初に注文作成依頼の事前条件を判定する。
注文内容が確認済みでない場合、注文を作成せず、作成不可の注文作成結果を返す。
販売可能在庫の参照結果が注文作成可能でない場合、注文を作成せず、作成不可の注文作成結果を返す。
購入者情報が注文へ記録できない場合、注文を作成せず、作成不可の注文作成結果を返す。

事前条件を満たす場合、B001 は注文を作成する。
作成する注文には、選択商品、数量、購入者情報、注文内容、販売可能在庫の参照結果を記録する。
注文番号と注文状態の詳細は未確認であるため、実装では既存の識別子生成方針または最小の作成済み状態に合わせる。
既存方針がない場合は、注文作成結果の中で未確認事項として扱い、追加の外部連携を作らない。

B001 の出力は注文作成結果である。
成功時は作成済み注文を含む。
失敗時は注文を含めず、作成不可の理由を含む。

この workspace は Amadeus 成果物の例示であり、現時点で実装コードと既存テストは存在しない。
そのため、Construction Design は新規実装を前提に、ドメイン境界、入力、出力、検証観点を Task 化できる粒度で固定する。

## 実装設計

B001 の実装は、注文作成依頼を受け取り、注文作成結果を返す処理として構成する。
実装対象候補は、注文作成の入口、注文作成の制御、注文、注文作成結果、作成不可理由である。
具体的なファイル配置は、後続の Implementation Execution で作業ツリーの実装構造に合わせて決める。

注文作成依頼は、少なくとも次を保持する。

- 確認済み注文内容。
- 購入者情報。
- 販売可能在庫の参照結果。

注文は、少なくとも次を保持する。

- 選択商品。
- 数量。
- 購入者情報。
- 注文内容。
- 販売可能在庫の参照結果。

注文作成結果は、少なくとも次を保持する。

- 成功または作成不可の状態。
- 成功時の作成済み注文。
- 作成不可時の理由。

入力検証は、注文作成依頼の境界でまとめて行う。
注文内容が未確認の場合、販売可能在庫の参照結果が注文作成不可の場合、購入者情報が不足する場合は、注文を作成しない。
複数の作成不可理由が同時にある場合の優先順位は未確認である。
後続実装では、観測可能な理由を失わない表現を優先する。

外部連携は追加しない。
販売可能在庫は参照結果として受け取り、B001 から在庫管理へ問い合わせない。
決済、売上確定、在庫引当、出荷の呼び出しも追加しない。

Task 分解は、入力契約、事前条件判定、注文作成、注文作成結果、検証の順に進める。
この順序により、注文作成の前提と副作用境界を確認しながら実装できる。

## 検証設計

成功系は、確認済み注文内容、購入者情報、注文作成可能な販売可能在庫の参照結果から注文を作成できることを検証する。
作成済み注文には、選択商品、数量、購入者情報、注文内容、販売可能在庫の参照結果が記録されることを確認する。

作成不可系は、注文内容が未確認の場合、販売可能在庫の参照結果が注文作成不可の場合、購入者情報が不足する場合に注文を作成しないことを検証する。
それぞれの作成不可理由が注文作成結果で観測できることを確認する。

境界系は、注文作成が決済、売上確定、在庫引当、出荷を開始しないことを検証する。
実装構造に外部連携の呼び出し点がある場合は、呼び出されないことをテストで確認する。
外部連携の呼び出し点が存在しない場合は、対象外処理を実装しないことをコードレビューの確認事項にする。

検証コマンドは、後続の Implementation Execution で実装構造に合わせて確定する。
現時点で既存コードと既存テストは存在しないため、具体的なテストコマンドは未確認である。
