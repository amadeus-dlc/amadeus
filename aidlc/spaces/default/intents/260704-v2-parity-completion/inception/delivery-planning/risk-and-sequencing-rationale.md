# Risk and Sequencing Rationale：260704-v2-parity-completion

## 順序の根拠

risk-first（GD011）を採用し、最大リスクの順に Bolt を並べた。

1. B001 が最初なのは、最大リスク（結線層がエンジンの directive 契約と噛み合わない。raid-log 参照）を最小コストで検証できるからである。ここが崩れると戦略（無改変コピー）自体の見直しになるため、他のどの作業より先に確かめる。
2. B002 が 2 番目なのは、B001 で確立した型の繰り返しであり、量は多いがリスクは低いからである。削除（U004）を置換完了後に置くのは、入口の空白を作らないためである。
3. B003 が 3 番目なのは、検査対象（配置と skill 一覧）が B002 完了で確定するからである。
4. B004 が最後なのは、文書は実装結果の反映であり、examples と dogfooding は全体が動いてはじめて成立するからである。

## リスクと Bolt の対応

| リスク（raid-log 由来） | 対応する Bolt | 扱い |
|---|---|---|
| 結線層の設計不整合 | B001 | 最初に検証。失敗時は halt-and-ask で戦略を再判断する |
| hook と既存環境の衝突 | B001 | settings マージ後に既存検証一式で回帰確認する |
| 移行中の二重構造（旧散文駆動と新エンジン駆動の併存） | B002 | 入口切替を B002 内で完結させ、期間を最短にする |
| examples の全面無効 | B004 | 再生成まで旧契約 examples を保持する（B002、B003 では test:examples の provenance 許容を維持） |

## 実行様式

- ソロ開発のため、Bolt の並行実行はせず B001 → B004 を直列に実行する。
- 中間 PR は人間指示により省略する。Bolt 境界は `BOLT_STARTED` / `BOLT_COMPLETED` の audit イベントで記録し、最終 PR 1 本（Inception + Construction）へ統合する。C002 の main 常時 green は「最終 PR が green」で担保し、この粒度例外の理由と確認先を最終 PR 説明に記録する。
