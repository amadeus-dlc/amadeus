# Guided Question Policy

このメモは、Amadeus skill の `guided` で行う質問数の扱いを定める。

## 背景

`guided` は、成果物作成や作業開始の前に不足している判断だけを確認するモードである。

従来の「最大N問」という固定上限は、質問が長引くことを防ぐ効果がある。
しかし、必要な判断が未確定のまま上限に達した場合に、作業を始める根拠が壊れる。

そのため、質問数は固定上限ではなく目安として扱う。

## 方針

質問数には skill ごとの目安を設ける。

目安を超えても、作業開始に必要な判断が未確定であれば質問を続ける。

目安を超えて質問を続ける場合は、追加確認が必要な理由を短く示す。

既存成果物、既存資料、会話、作業ツリー、テスト入口から分かることは質問しない。

質問した場合は、その場で成果物作成、実装、テスト実行を進めず、ユーザーの回答を待つ。

## 目安

| Skill | 質問数の目安 | 理由 |
|---|---:|---|
| `amadeus-discovery` | 5問 | 入力テーマの課題、成功状態、Intent 分割判断を入口で整理するため。 |
| `amadeus-intent-init` | 3問 | Intent の入れ物作成に必要な判断だけを扱うため。 |
| `amadeus-steering` | 5問 | workspace の目的、関係者、制約、主要領域を初期化するため。 |
| `amadeus-ideation` | 5問 | Intent の範囲、実現可能性、初期モック方針を整理するため。 |
| `amadeus-inception` | 5問 | 要求、相互作用、Unit、Bolt の実施設計判断を整理するため。 |
| `amadeus-construction` | 4問 | Inception 成果物を前提に、Bolt 実行前の不足だけを扱うため。 |

## 運用

目安以内で判断が揃った場合は、そこで質問を止める。

目安を超える必要がある場合は、どの判断が未確定で、なぜ作業開始を妨げるのかを示す。

目安に達したことを理由に、未確定事項を残したまま作業を開始しない。

質問を続ける場合も、一度に複数の論点を並べない。
`amadeus-grilling` の作法に従い、一問ずつ確認する。
