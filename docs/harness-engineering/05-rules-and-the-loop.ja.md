# ルールと学習ループ

ルールは、チームがすべてのワークフローに持ち込む恒常的な決定です — さもなければ各実行の冒頭で説明し直すことになる「常にこのやり方でやる」というものです。ハーネスエンジニアとして、あなたは2つの方法でルールを記述します。ルールファイルを直接編集するか、あるいは**学習ループ**にワークフロー内での是正を恒久的なルールへ昇格させてもらうかです。この章は、ユーザーガイドの [Rules and the Learning Loop](../guide/09-rules-and-the-learning-loop.md) 章のオーサリング側の対となるものです。そちらの章はループを概念的に扱い、ANZ 銀行の例を最初から最後まで追っています。ここでの焦点は運用面です。何のためにどのファイルを編集するか、ルールを積み重ねたときに strict-additive モデルがどう振る舞うか、そして一度きりの修正が恒常的なルールになるようループをどう回すか。

ルールは制御ループのフィードフォワード側の半分です — エージェントが作業する*前*に読む散文です。[センサー](06-sensors.md)はフィードバック側の半分です — 書き込みの*後*に発火する決定的なチェックです。ルールはそのコンパニオンとしてセンサーを指名できます。この2つの半分は対になるよう設計されています。

## 5つのレイヤーと、どのファイルを編集するか

ルールは `core/memory/` 配下の Markdown ファイル(space メモリレイヤーの記述元ソース)として存在し、スコープごとに1ファイルです。`scope:` フロントマターフィールドはありません — この実装はファイル名からスコープを導出するため、あなたが選ぶファイル*こそが*、あなたが記述するスコープです。

| 適用したいルールの対象… | 編集するファイル | スコープ |
|---|---|---|
| 組織のすべてのプロジェクト | `org.md` | org(フレームワークデフォルト) |
| チームが動かすすべてのプロジェクト | `team.md` | team |
| この1つのプロジェクト | `project.md` | project |
| 1つのフェーズ内のすべてのステージ | `phases/<phase>.md` | phase |

4つのフェーズファイルは `phases/ideation.md`、`phases/inception.md`、`phases/construction.md`、`phases/operation.md` です(initialization はブートストラップ専用で、ルールファイルを同梱しません)。5番目のレイヤー — ステージごとのルール(`amadeus-stage-<slug>.md`) — は将来のリリース向けに予約されており、今はまだ記述できません。

どのファイルに手を伸ばすかを決める判断は2つです。

- **`org.md` はフレームワーク同梱。** すべてのプロジェクトが継承するデフォルトを担います — トランクベース開発、walking-skeleton ポリシー、スコープごとのテスト姿勢。これは上流として扱ってください。ほとんどのハーネスエンジニアはこれをそのままにし、team か project で記述します。
- **project スコープは恒久的な逸脱のためであり、なんでも用ではない。** *この*プロジェクトがチーム全体のプラクティスから安定的に離れる場合のみ `project.md` に手を伸ばしてください — チームがスカッシュするところをリベースするモノレポ、テスト下限をスキップするレガシープロジェクトなど。もしそのルールがチームの動かすすべてのプロジェクトに役立つなら、それは `team.md` に属します。

各ファイルはトピック別の見出し — `## Way of Working`、`## Testing Posture`、`## Deployment`、`## Code Style` など — の下のプレーンな散文です。ルールを追加するには、それが属する見出しの下に箇条書きを1つ加えます。完全なファイル名対スコープのテーブルと解決の仕組みは規範的な契約です。[Rule System § Filename-derived scope](../reference/08-rule-system.md#filename-derived-scope) と [§ Layout](../reference/08-rule-system.md#layout) を参照してください。

## strict-additive: レイヤーは積み重なり、どれも静かに上書きしない

チェーンはすべてのワークフローの冒頭で5つのレイヤーを通して解決されます。

```
org → team → project → phase → stage
```

このモデルは **strict-additive** です。適用されるすべてのルールがエージェントのコンテキストに一度に現れます — 実行時に何かが落とされたり抑制されたりすることはありません。org のデフォルト、team のプラクティス、project の特殊化は連結され、マッチするフェーズルールは、ステージがすでにフロントマターで自身の `phase:` を宣言しているために付随します(ステージが他のすべてをインポートするのと同じプル型オーサリングの方向です)。解決されたチェーンは各ステージノードに一度だけ、ワークフロー開始時に焼き込まれます — ランタイムはそれを再び歩き回りません。[Rule System § Strict-additive runtime model](../reference/08-rule-system.md#strict-additive-runtime-model) と [§ Five-layer inheritance](../reference/08-rule-system.md#five-layer-inheritance) を参照してください。

これが作者としてのあなたにとって意味するのは、`overrides:` ブロックも、設定する `enforcement:` キーワードもない、ということです。より狭いレイヤーがより広いレイヤーに静かに勝つことはありません。代わりに、あなたが書くすべてのレイヤーがエージェントのビューに同時に存在します。それはオーサリングの仕方を変えます — ルールが適用されるべきスコープでそれを肯定的に述べ、上のレイヤーを抑制するスイッチに手を伸ばすのではなく、上のレイヤーとともに積み重なることを信頼するのです。

### 矛盾は実行時に解決されるのではなく、書き込み時に拒否される

実行時に何も上書きしないため、より広いスコープのルールと*矛盾する*ルールは、解決器が解きほぐせない問題になります。フレームワークは、実行時ではなく**書き込み時**にチェックすることでこれを未然に防ぎます。ある `## Heading` の下に team スコープのルールが追加されようとするとき、受理ゲートが提案されたテキストを `org.md` の同じ見出しと比較します。矛盾を見つけると、ゲートは書き込みを止め、3つの選択肢を提示します — **revise**(修正)、**skip**(スキップ)、または org ルールのオーナーへの **escalate**(エスカレーション)。project 層の書き込みは org のみと照合します。team 対 project の違いは正当なプロジェクト特殊化であり、ポリシー違反ではないからです。

このチェックは、フレームワークが所有する2つの受理ゲート — practices-discovery の承認ゲートと(下記の)学習ゲート — で実行されるため、ルールが解決器に到達するときには、すでに矛盾チェックを通過しています。2つの読み取り専用の `/amadeus --doctor` の行が、事後に状態を可視化します。**rule-drift** 行は、team または project のコンテンツが設定済みの org 見出しと重複する見出しをフラグし(人間が検証すべき矛盾候補)、**paired-coverage** 行は、実際に解決されるセンサーを指名しているルールがいくつあるかを報告します。どちらも助言的で、終了コードを変えることはありません。[Rule System § Rule-drift detection](../reference/08-rule-system.md#rule-drift-detection) を参照してください。

## 学習ループを回す

ルールが記述されるもう1つの方法は、あなたが一度もファイルを開かないというものです — ワークフロー中にエージェントを是正し、ゲートでその是正を確認すると、フレームワークがあなたのために書き込みます。それが学習ループです。ほとんどのステージ実行は何も追加しませんが、それは健全です。ループは、ステージ中に何か保持する価値のあることがあったときにのみ発火します。

その仕組みは[ユーザーガイドの章](../guide/09-rules-and-the-learning-loop.md)でカバーされています。ハーネスエンジニアの視点は、ループが何を生み出し、それがどこに着地するかです。

1. **ダイアリーはステージ中に記録する。** コンダクター(アクティブなステージを動かしているライブの `/amadeus` セッション)は、`<record>/<phase>/<stage>/memory.md`(intent のレコードディレクトリ `amadeus/spaces/<space>/intents/<YYMMDD>-<label>/` 配下)に観察ログを保持します。エントリは4つの見出し — Interpretations、Deviations、Tradeoffs、Open questions — の下に置かれます。これは自動作成され、あなたのために保守されます。決して手編集しないでください。ダイアリーを書くことが、このループで言語モデルが持つ*唯一の*仕事です — ステージ後のすべて(集計、可視化、ルーティング、書き込み)は決定的なツールか、ゲートでのあなたの明示的な選択です。
2. **ゲートは候補を可視化する。** 承認ゲートの前に、学習ゲートが `memory.md` を読み、空でない各ダイアリー行を候補として逐語的に表示します。加えて、次回に向けて観察を入力し、それが4つの見出しのどれに属するかを選ぶ自由記述の「次回のために何か追加することは?」チャネルもあります。
3. **何を保持するかを確認すると、書き込まれる。** 確認された学習は日付付きエントリとして着地します — あなたがファイルパスを選ぶことはなく、見出しが行き先を決めます。

### 保持された学習がどこに着地するか

確認された学習は*プラクティスそのもの*です。practices-discovery が承認するのと同じ space メモリファイル(`amadeus/spaces/<space>/memory/team.md` / `memory/project.md`)に着地します — 別個の `*-learnings.md` 面はありません。ゲートはそれをトピック別の見出しでルーティングします。

| ゲートでの見出し | 着地先 | 監査イベント |
|---|---|---|
| Interpretation / Deviation / Tradeoff | `amadeus/spaces/<space>/memory/project.md`(デフォルト) | `RULE_LEARNED` |
| 同上、ワンクリックで昇格 | `amadeus/spaces/<space>/memory/team.md` | `RULE_LEARNED` |
| Open question | なし — 調査項目は昇格しない | — |

デフォルトのスコープは最も狭い **project** です。ワンクリックの「team へ昇格」アフォーダンスが、教訓がこのプロジェクトを超えて適用されるときに学習を `memory/team.md` へと広げます。org スコープへ広げる道はありません — org のプラクティスはフレームワーク同梱か、別プロセスで組織が記述するものなので、ループが org スコープで書き込むことはありません。デフォルトを狭く保つことで、あるプロジェクトの想定外が偶然に組織全体のルールになるのを止めます。学習ループと practices-discovery は、2つの異なるライフサイクルでこれら同じファイルに書き込みます。ループは日付付きエントリを1つずつ追記し、practices-discovery はセクション全体を承認します。解決器はファイルを org → team → project → phase というクリーンな整数チェーンで並べます。

### 学習がルールではなくチェックであるとき(`SENSOR_PROPOSED`)

保持する価値のあるものが*繰り返される手動チェック*であることもあります — 同じ抜け漏れを求めてステージの出力を何度も見張ってしまう、というものです。その種の学習をゲートで確認すると、フレームワークはそれをルールではなくセンサーのバインディングとして扱います。2つの書き込みをアトミックにインストールします。`.claude/sensors/` の下にセンサーマニフェストをスキャフォールドし、発端となったステージの `sensors:` インポートリストに新しいセンサーの id を追記し、`SENSOR_PROPOSED` の監査行を残します。それがフレームワークのリリース外でステージファイルに許される唯一の編集です — インポートリストだけが増え、ステージ本体には決して触れません。そこから、[センサー](06-sensors.md)を使って手作業でマニフェストを肉付けします — ループはバインディングをスキャフォールドし、チェックはあなたが記述します。

### 学習は進行中のワークフローではなく、次のワークフローに適用される

ゲートで捉えられた学習は、現在の実行の残りのルールを**変えません**。あなたはこのワークフローについてはすでに会話でエージェントを是正済みです。ルールは次回のためのものです。新しい行はディスク上にありますが、進行中のワークフローは開始時にコンパイルしたビューを保ち続けます。*次*にあなたが実行する `/amadeus` が再コンパイルし、ディレクトリ走査が新しいファイルを拾い、ルールがステージ1以降から適用されます。

これは手作業で記述するときにも重要です。ワークフロー途中で `team.md` を編集しても、進行中の実行を遡って変えることはありません。ルールは次のコンパイル境界で効きます。変更を即座に効かせたいなら、ワークフローを終えるか再起動して、コンパイルがあなたの編集を再読するようにしてください。
