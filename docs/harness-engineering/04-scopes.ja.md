# スコープ

スコープは、ある種類の作業に対してフレームワークの32ステージのうち*どれ*を実行し、どれを実行しないかを決めるダイヤルです。bugfix には市場調査やデプロイパイプラインは不要ですし、規制対象のエンタープライズ機能にはそのすべてが必要になります。毎回ユーザーにステージを手作業で選ばせるのではなく、AI-DLC は9つの名前付きスコープを同梱しています。各スコープは全ステージ集合に対する EXECUTE/SKIP の判定を丁寧にキュレーションしたもので、デフォルトの深さ(depth)とテスト戦略(test strategy)が対になっています。スコープを選べば、あとは連鎖的に決まります。

ハーネスエンジニアにとって、スコープは他のすべてのプリミティブと同じ方法で — ファイルとして — 記述される純粋なデータです。スコープは2つの半分から成ります。1つは `core/scopes/amadeus-<name>.md` ファイル(そのアイデンティティ — 名前、深さ、キーワード、説明)、もう1つはステージごとのメンバーシップタグ(各ステージのフロントマターの `scopes:` リストが、そのステージが実行されるスコープを列挙する)です。スコープの追加やチューニングに TypeScript は不要です。この章ではワークフローを追っていきます。スコープが何で構成されるか、チームスコープをどう追加するか、既存のものをどうチューニングするか、そしてツールが何を検証してくれて何をあなたに委ねるか。

ユースケースとユーザーが読むルーティングテーブルを含む9スコープの完全なカタログについては、ユーザーガイドの [Scopes, Depth, and Test Strategy](../guide/05-scopes-and-depth.md) を参照してください。この章は同じデータを記述する(オーサリング)側です。

## スコープを構成するもの

スコープは2箇所で記述され、この分割こそが要点です。スコープの*アイデンティティ*はそれ自身のファイルに置かれ、その*メンバーシップ*(どのステージがそのスコープで実行されるか)はステージ側に転置されて置かれます。

**1. スコープファイル — `core/scopes/amadeus-<name>.md`。** スコープごとに1ファイルで、`core/sensors/` と同じ構成です。`feature` スコープのフロントマターは次のようになります。

```yaml
---
name: feature
depth: Standard
keywords: []
description: Default for new features, practical depth
---

# feature scope

Prose intent: why these stages, why skip those.
```

フロントマターのフィールドは、1つの必須フィールドと3つのオプションのつまみに分かれます。

| フィールド | 必須 | 役割 |
|-------|----------|--------------|
| `name` | Yes | スコープ名。ファイル名のステム(`amadeus-feature.md` → `feature`)と一致しなければならない。 |
| `depth` | Yes | デフォルトの詳細度 — `Minimal`、`Standard`、`Comprehensive` のいずれか。 |
| `testStrategy` | No | 深さとは独立してテスト量を上書きする。デフォルトは `depth` に一致。 |
| `keywords` | No | `/amadeus <freeform text>` の自動検出のための自然言語トリガー。空リストはオプトアウト。 |
| `description` | No | `/amadeus --help` に表示される1行説明。(SKILL.md のコンパイル済みスコープテーブルは Scope / Depth / TestStrategy / EXECUTE / Total のみを表示し、description は含めない。) |

**2. メンバーシップタグ — 各ステージの `scopes:` フロントマター。** ステージは、`core/amadeus-common/stages/<phase>/<slug>.md` にある自身のフロントマターで、自分が実行されるスコープを列挙します。

```yaml
scopes:
  - enterprise
  - feature
  - mvp
```

あるスコープを列挙しているステージはそのスコープで `EXECUTE`、列挙がなければ `SKIP` です。ビルドステップ `bun .claude/tools/amadeus-graph.ts compile` は、各ステージの `scopes:` リストを `.claude/tools/data/scope-grid.json` にあるコンパイル済みの EXECUTE/SKIP グリッドへ*転置*します。これは純粋な転置であり、`stage-graph.json` とまったく同様に `compile --check` によってドリフトガードされます。ランタイムが読むのはこのグリッドで、手作業で編集することはありません。3つの initialization ステージはすべてのスコープを列挙します(常に実行されるため)。

理解する価値のある唯一の判断は、`depth` と `testStrategy` の関係です。depth は各ステージの成果物がどれだけの詳細を持つかを制御し、test strategy は生成されるテストの数を制御します。これらは意図的に独立しています。ほとんどのスコープは `testStrategy` を省略するため `depth` から継承されます — Standard 深さのスコープは Standard の量でテストします。`workshop` スコープは、この結び付きを断ち切る同梱の例です。`"depth": "Standard"`(参加者が学習中なので完全な成果物)で動く一方、`"testStrategy": "Minimal"`(セッションを止めないよう高速な Nyquist テスト)です。スコープにこの分割が必要なら、両方を宣言してください。各レベルの意味については、ユーザーガイドの [The 3 Depth Levels](../guide/05-scopes-and-depth.md#the-3-depth-levels) と [The 3 Test Strategy Levels](../guide/05-scopes-and-depth.md#the-3-test-strategy-levels) を参照してください。

フィールドごとの網羅的な契約 — `keywords` が単語境界でどうマッチするか、あいまいなフリーフォーム呼び出しをアルファベット順のスコープタイブレークがどう解決するかを含む — は、Developer Reference の [Contributing § Adding a Scope](../reference/11-contributing.md#adding-a-scope) にあります。この章は決定事項を要約したもので、そのセクションが規範的な仕様です。

## スコープとステージの関係

スコープとステージは反対側の端から互いを指し示しており、両方向を同時に把握しておくと役立ちます。

**ステージ**は自身のアイデンティティ — そのフェーズ、リードエージェント、消費・生成する成果物、そして今や自分が実行されるスコープ(その `scopes:` リスト) — を宣言します。**スコープ**はそれ自身の `.md` ファイルでアイデンティティ — 名前、深さ、キーワード、説明 — を宣言し、内部にステージごとのメンバーシップは持ちません。メンバーシップはステージ側に置かれます。両者を結び付けるのはスコープ名です。新しいステージを追加するとき(参照: [Adding a Stage](02-adding-a-stage.md))、スコープのメンバーシップは*そのステージ上*に置きます — その `scopes:` リストが、それを実行すべきすべてのスコープを列挙します。どのスコープも列挙しないステージは、どこでも `SKIP` です。コンパイル時の転置がこれらステージごとのリストをグリッドに変えるため、メンバーシップは9つの別々のスコープブロックで再宣言するのではなく、ステージ上で一度だけ記述されます。

この分離は、このガイドの他の部分が依拠しているのと同じ「データ対コード」の線です(参照: [Harness Engineer Guide](00-overview.md))。スコープファイルは*アイデンティティ*についてのデータ、ステージの `scopes:` リストは*メンバーシップ*についてのデータ、そしてコンパイル済みグリッドはその2つの転置です。

## チームスコープを追加する

チームが `hotfix` スコープを欲しがっているとしましょう — `bugfix` より無駄をそぎ落とし、リグレッションテストとデプロイだけが欲しい緊急の本番パッチ向けです。変更は、新しいスコープファイル、実行すべき各ステージへの `scopes:` タグ、そして再コンパイルです。以下の作法をまねてください。検証手順と完全なコマンド行は [Contributing § Adding a Scope](../reference/11-contributing.md#adding-a-scope) にあります。

### 手順

1. **`core/scopes/amadeus-hotfix.md` を作成する。** `amadeus-bugfix.md`(最も近い既存スコープ)をコピーし、フロントマターを編集します。`name: hotfix` を設定し、`depth` を選び、フリーフォームの自動検出が欲しければ `keywords`(`[hotfix, urgent]`)を追加し、ヘルプテキスト用の `description` を、そして `depth` と分岐させたい場合のみ `testStrategy` を追加します。意図を説明する短い散文の本文を書きます。

2. **`hotfix` で実行すべきステージにタグを付ける。** `EXECUTE` にしたい各ステージ(`core/amadeus-common/stages/<phase>/` 配下)で、そのフロントマターの `scopes:` リストに `hotfix` を追加します。タグを付けないステージはそのスコープで `SKIP` です。3つの initialization ステージはこれを含まなければなりません(常に実行されるため)。

3. **再コンパイルする。** `bun .claude/tools/amadeus-graph.ts compile` を実行してタグを `scope-grid.json` へ転置し、次に SKILL.md のコンパイル済みスコープテーブルを再生成します(`bun .claude/tools/amadeus-utility.ts scope-table`、`<!-- BEGIN: compiled ... -->` / `<!-- END: compiled ... -->` マーカーの間に貼り付ける)。`bun .claude/tools/amadeus-graph.ts compile --check` と `bun .claude/tools/amadeus-utility.ts scope-table --check` を実行し、ドリフトがないことを確認します(終了コード 0)。

4. **スコープが解決され受理されることを確認する。** `/amadeus --doctor` を実行します。次に、新しいスコープでの init が正しい `Scope:` 行を持つ状態ファイルを生成すること、そして環境デフォルトとしても、ワークフロー途中の `--scope` 変更としても受理されることを確認します。

5. **キーワード推論を確認する**(`keywords` を設定した場合のみ)。あなたのトリガーの1つを含むフリーフォームの語句が、`feature` に落ちずに新しいスコープを検出することを確認します。

6. **スコープを意識するドキュメントを更新し、ルーティングテストを追加する。** いくつかのドキュメントはスコープを手作業で列挙しています — ユーザーガイドのスコープリファレンスとルーティングテーブル、カスタマイズ章の有効値リスト、オーケストレーターリファレンスのスコープ対ステージのマッピングです。これらを同じ変更で更新します。もしあなたのスコープが既存のどのスコープも使わないパターンでステージをスキップするなら、既存のスコープごとのテストをモデルにしたワークフローテストを追加します。

7. **(オプション)タイプ可能なランナーを生成する。** スコープはファイルが着地した瞬間から `/amadeus --scope <name>` で完全に利用可能です — ランナーは不要です。1語のコマンド(`/amadeus-hotfix`)が欲しければ、ランナースキルを生成します。`bun .claude/tools/amadeus-runner-gen.ts scopes --all` はすべてのスコープファイルについて `skills/amadeus-<scope>/SKILL.md` を出力します(あるいはジェネレーターの `FIRST_BATCH` を編集してキュレーションしたサブセットを同梱します)。各ランナーは `amadeus-orchestrate next --scope <name>` を `done` まで駆動する約6行のシェルで、スコープを組み込んでいます。ランナーはすでに実行可能なスコープをパッケージするものであり、スコープファイルがその定義です。ランナーは `hooks:` ブロックを持ちません。決定的なバックボーン(audit、sensors、runtime-compile、state validation)は `settings.json` にプロジェクト全体で登録されているため、すべてのランナーが無償でそれを継承します。スコープファイルを追加またはリネームするたびに、ジェネレーター(または `scopes --check`)を再実行してください。

### 自動で検証されるもの

この実装は、実行時に `.claude/scopes/*.md` の存在から `validScopes()` を通じて有効なスコープのリストを導出するため、ファイルが着地した瞬間に多くが所定の位置に収まります。

- スコープは一度にどこでも有効になります — `init`、`--scope` 変更、環境デフォルトの解決、`doctor` はすべて同じヘルパーを参照するため、どれもコード編集を必要としません。
- エラーメッセージは、変更なしであなたのスコープをアルファベット順に列挙します。
- `keywords` を与えていれば、フロントマターのリストが設定された瞬間にフリーフォームの `/amadeus <text>` がそれを自動検出します — SKILL.md の散文編集は不要で、テーブルの再生成だけです。
- 転置ドリフトガード(`compile --check`)は、ステージの `scopes:` タグが再コンパイルなしに編集された場合、ビルドを失敗させます。

### 自動では検証されないもの

- **スコープ名をタイプミスした `scopes:` タグでもパースは通る。** `hotfix` の代わりに `hotfx` を列挙したステージフロントマターは、きれいにコンパイルされます — ただ誰も求めないグリッド列を生成するだけです。ポイントは、`validScopes()` が `.md` ファイルから導出されるため、ファイルのないスコープは呼び出し時に拒否されますが、ステージ上のタイプミスしたタグはそのステージを実際のスコープから静かに落としてしまう、という点です。`/amadeus --doctor` とスコープごとのテストがガードレールです。
- **コンパイル済みスコープテーブルはドリフトしうる。** 手順3で `scopes:` を編集したのに再コンパイルとテーブル再生成をスキップすると、エンジンは古いグリッドを読み続けます。`--check` フラグ(テストスイートが実行する)がこれを捕えますが、実行した場合に限ります。
- **スコープごとのフェーズシーケンスのカバレッジ。** 同梱のフェーズシーケンステストは、既知のスコープ名のハードコードされたリストを反復します。新しいスコープは、そのリストを拡張するまでこのテストで実行されません。同じ変更でスコープをそこに追加してください。
- **手作業で保守されるドキュメント。** ドキュメントをあなたのために grep してくれるものはありません。スコープリファレンス、ルーティングテーブル、カスタマイズの有効値リストは散文です。スコープファイルと歩調を合わせるのは自分で行ってください。

## 既存のスコープをチューニングする

チューニングはより小さな編集ですが、スコープではなくステージに着地します。よく起きる変更は2つです。

- **ステージを組み入れる／外す。** ステージの `scopes:` リストからスコープ名を追加または削除します。たとえば、チームが常に最初のカットでも監視を配線するので、`observability-setup` の `scopes:` に `mvp` を追加する、というやり方です。1つのタグ、そして再コンパイル(`compile` + scope-table)と `--doctor` の実行です。
- **デフォルトの深さやテスト戦略を変える。** スコープの `core/scopes/amadeus-<name>.md` フロントマターで `depth` を調整するか、`testStrategy` を追加/削除して、スコープが生成するものを再調整します。各スコープが自身のデフォルトを持つため、この変更はそのスコープを選択するすべてのワークフローに連鎖します — 実行ごとのフラグは不要です。ユーザーは任意の実行で `--depth` や `--test-strategy` で上書きできますが、スコープの値がチーム全体のベースラインです。

いずれの場合も、上記の手順3の「再コンパイルと doctor」のペアが適用されます。編集は小さく、検証は同じです。

レイヤリングに関する注意: 同梱スコープをチューニングすることは、フレームワーク同梱ファイルを直接編集することを意味します — ステージの `scopes:` タグや、同梱の `core/scopes/amadeus-*.md` です。異なるデフォルトを望むフォークにとってこれは正当ですが、`amadeus-` の系譜を担うファイルを変更していること、そしてフレームワークのアップグレードがそれらを整合させたがるかもしれないことに注意してください。他の全員が依拠するデフォルトに触れずにチーム固有の振る舞いが欲しい場合は、同梱の9つと並べて完全に新規のスコープファイルを追加するのがよりクリーンな道です。
