# チーム知識(Team Knowledge)

知識(knowledge)とは、エージェントが作業に入る前に読むドメインコンテキストです。あなたのコーディング標準、アーキテクチャの好み、ドメイン用語集、チームが定着させたパターン — こうしたものを指します。これはハーネスの中でも、フレームワークが強制する制約ではなく、エージェントが読むファイルを追加することで形づくる唯一の部分です。この章では、エージェントにそのコンテキストを与えるためのワークフロー — ファイルをどこに置くか、どのエージェントがそれを見るか、そして知識とルールの間の判断 — を順に見ていきます。

[Rules and the Learning Loop](05-rules-and-the-loop.md) をすでに読んでいるなら、その区別を全体を通じて意識しておいてください。ルールはフレームワークが強制する既定の決定であり、知識はエージェントが作業しながら比較検討する参照資料です。どちらもエージェントの挙動を形づくりますが、両者は異なる次元に位置し、異なる方法でロードされます。

---

## 2つの階層: フレームワーク知識とあなたの知識

AI-DLC の知識は2つの階層に分かれており、そのうちあなたが編集できるのは片方だけです。

**Tier 1 — 方法論(methodology)知識** は、この実装に `.claude/knowledge/` 配下として同梱されます。ここには、各エージェントがステージを実行するために使う方法論の参照が含まれます — `amadeus-architect-agent/architecture-guide.md`、`amadeus-developer-agent/code-generation-guide.md`、そして `amadeus-shared/` 内のクロスエージェント資料です。**触れないでください。** これらのファイルはフレームワークのアップグレードのたびに上書きされます。ここに追加したものは、チームが次に新しいバージョンを pull した時点で消えます。

**Tier 2 — チーム知識** はあなたのものです。これは space レベルの `amadeus/knowledge/`(`amadeus/spaces/<space>/knowledge/` の略記)配下に置かれ、その space の `memory/`、`codekb/`、`intents/` の兄弟にあたります — つまり1つの intent のレコードに閉じ込められるのではなく、space 内のすべての intent にまたがって蓄積されます。ここにはあなたの会社固有の標準、ポリシー、規約が入ります。フレームワークがこれを上書きすることはありません。エンジンは最初の `/amadeus` で空の `amadeus/knowledge/` ディレクトリを作成するだけで、その中身はあなたに委ねられます。これがあなたが埋めていくディレクトリです。(参照資料としてエージェントが比較検討するのではなく、フレームワークが *強制* すべき既定のプラクティスは、代わりに space のメモリ層 `amadeus/spaces/<space>/memory/` に置かれます。)

この2階層の分割は、このガイドの残りの部分が拠って立つ、データとコードの同じ境界線を知識に適用したものです。フレームワークは自身の方法論を所有し、あなたは自分のコンテキストを所有し、アップグレードは一方をもう一方に触れることなく置き換えられます。両階層の完全なディレクトリ構造は [Knowledge System → Two-Tier Architecture](../reference/10-knowledge-system.md#two-tier-architecture) にあります。

## チーム全体向けとエージェント固有の配置

Tier 2 は慣習としてエージェントのレイアウトに従います。`amadeus-shared/` ディレクトリに加えて、エージェントごとに1つのディレクトリがあり、すべてが space レベルの `amadeus/knowledge/` 配下にあります。ファイルをどこに置くかで、どのエージェントがそれをロードするかが決まります。

| 配置 | ロードするエージェント | 用途 |
|-----------|-----------|------------|
| `amadeus/knowledge/amadeus-shared/` | **すべての** エージェント、すべてのステージで | 横断的な標準 — 命名規約、コミットフォーマット、プロジェクトのドメイン用語集 |
| `amadeus/knowledge/<agent>-agent/` | **その** エージェントのみ、それがアクティブなリードのときのみ | 1つのロール向けのドメインコンテキスト — アーキテクトのアーキテクチャパターン、devsecops のセキュリティポリシー |

ディレクトリ名はエージェントの slug と正確に一致しなければなりません — `architect/` ではなく `amadeus-architect-agent/` です。ディレクトリ名のタイプミスは、ファイルが黙って無視される最も一般的な原因です。フレームワークはエージェント自身のディレクトリを名前で辿り、何も見つからなければエラーを出さずに次へ進みます。エンジンはこれらのサブディレクトリを作成しません — `amadeus/knowledge/` はブートストラップ時点では空です(下記参照)。したがって各ディレクトリを正確な slug で自分で作成します。

`amadeus-shared/` に手を伸ばすのは、ある標準が本当に11エージェントすべてにまたがって適用される場合だけにしてください。アーキテクトにだけ重要で他の誰にも関係しないパターンは `amadeus-architect-agent/` に属し、そこではアーキテクチャステージにコンテキストを追加しつつ、他のすべてのエージェントのウィンドウを希釈しません。User Guide の [Adding Company Standards の実例](../guide/08-knowledge.md) には、完全なエンドツーエンドのウォークスルー — ディレクトリを作成し、書き、検証する — があり、最初のファイルを執筆する前に一度読んでおく価値があります。

各ディレクトリが何のためのものかというエージェントごとの表は、[Knowledge System → Adding Team Knowledge](../reference/10-knowledge-system.md#adding-team-knowledge) を参照してください。

## エージェントが知識をどうロードするか

知識ファイルを登録したり、どこかに配線したりする必要はありません。正しいディレクトリに存在することそのものが登録です。ステージが始まると、コンダクターは固定された6ステップの順序でコンテキストをロードし、あなたの Tier 2 ファイルはステップ4と5で入ってきます。

1. ルール — 解決済みの `amadeus/spaces/<space>/memory/` チェーン(最初にロード)
2. Tier 1 共有方法論 — `.claude/knowledge/amadeus-shared/`
3. Tier 1 エージェント方法論 — `.claude/knowledge/<agent>-agent/`
4. **Tier 2 チーム共有** — `amadeus/knowledge/amadeus-shared/`
5. **Tier 2 チームエージェント固有** — `amadeus/knowledge/<agent>-agent/`
6. 先行ステージの成果物 — 現在のステージが消費すると宣言している出力

ステップ4と5は、ディレクトリが存在しファイルを含んでいる場合のみ発火します。だからチーム知識のないプロジェクトはそれらを単にスキップします。このロードはすべてのステージ開始時に起こるため、ファイルを編集すると次の `/amadeus` 実行で有効になります — クリアすべきキャッシュも、再起動も不要です。ファイルの削除も同じく直接的です — 削除すれば、以降の実行はそれを見なくなります。同期を保つべきレジストリはありません。

内在化しておく価値のある帰結が1つあります。エージェントは知識ファイルを **文字通り、同じ重みで** 読みます。古かったり矛盾したりするファイルはエージェントを積極的に誤導します — それは現行のファイルと同じ権威を持つのです。Tier 2 ツリーは剪定が必要なコードのように扱ってください。レトロスペクティブでの短いレビューが、それを正直に保ちます。

完全なステップバイステップの契約、優先順位ルール、シーケンス図は [Knowledge System → 6-Step Knowledge Loading Order](../reference/10-knowledge-system.md#6-step-knowledge-loading-order) にあります。

## 知識か、ルールか?

ハーネスエンジニアが最もよく犯すミスは、意図がルールであるときに知識に手を伸ばす、あるいはその逆です。両者は交換可能ではなく、上記のロード順序がその理由を示しています。ルールは、フレームワークが実行に先立ってコンパイルする strict-additive(厳密加算的)なチェーンとして最初に解決されます。知識は、ステージ中にエージェントが比較検討する参照資料です。

有用なテスト: **その指示が破られたときに人間のレビュアーがステージの出力を却下するなら、それはルールに属します。** レビュー時に背景として使うだけなら、それは知識です。

| 知識に手を伸ばすのは… | ルールに手を伸ばすのは… |
|---------------------------|------------------------|
| エージェントが参照すべき参照資料を提供するとき | エージェントが従わなければならない挙動上の決定を述べるとき |
| 「これらが我々が使うパターンだ」 | 「Xを決してするな」/「Yを常にせよ」 |
| 内容が情報提供的でコンテキスト的 | 内容が規範的で交渉の余地がない |
| 長文の散文、表、図でありうる | 短く、命令的で、1行ずつであるべき |
| 例: API Gateway 標準、ドメイン用語集 | 例: 「PII を決してログに出すな」「すべてのデータアクセスはリポジトリ層を経由する」 |

したがって、チームが API をどう設計するかを記述したドキュメントは知識です。`amadeus/knowledge/amadeus-architect-agent/` に置きましょう。「すべてのアーキテクチャ決定は最低2つの代替案を記録しなければならない」のような交渉の余地のないものはルールです。これは space のメモリ層(`amadeus/spaces/<space>/memory/`)に属し、そこでフレームワークがエージェントにそれを守らせます。レイヤーチェーンにまたがってルールを執筆し、学習ループに是正をルールへ昇格させる方法については、[Rules and the Learning Loop](05-rules-and-the-loop.md) を参照してください。User Guide の [Knowledge vs Rules の表](../guide/08-knowledge.md) は、より多くの例で同じ判断を扱っています。

## Tier 2 ツリーはどこから来るか

チームがそれを構築します。最初の `/amadeus` でエンジンは単一の空ディレクトリ — `amadeus/knowledge/` — を作成し、そこで止まります。ツリーを足場として組んだり、エージェントごとのサブディレクトリを作成したり、READMEを種として置いたりはしません。Tier 2 のレイアウト(`amadeus-shared/` に加えてエージェントごとに1ディレクトリ)は、ペルソナが探す慣習であって、エンジンが書き出す構造ではありません。あなたはコンテンツを持つディレクトリを作成します。ローダーは各エージェント自身のディレクトリを名前で辿るため、ローダーが期待する正確な slug で作成してください(`architect/` ではなく `amadeus-architect-agent/`)— タイプミスした名前はエラーなしに黙ってスキップされます。

ディレクトリ内に命名規約はありません。任意の `.md` ファイルがロードされます。説明的で、1ファイル1トピックの名前(`architecture.md` ではなく `api-gateway-standards.md`)はローダーが要求するものではありませんが、四半期ごとの剪定をはるかに容易にします。ディレクトリに出発点となるREADMEが欲しい場合、Tier 1 は手作業でコピーできるオプションのテンプレートを同梱しています — [Knowledge System → Template System](../reference/10-knowledge-system.md#template-system)。

このガイドの残りとの境界についての注記: ここで埋めるエージェントディレクトリは、エージェントがそのペルソナファイルで宣言するものと同じです。[エージェントを追加する](03-adding-an-agent.md) とき、その Tier 2 知識ディレクトリは `amadeus/knowledge/<new-agent-slug>/` です — チームが作成し、同じステップ4と5でロードされるディレクトリです。[概要](00-overview.md) のメンタルモデルが成り立ちます: ステージがエージェントを指名し、エージェントが知識を読み、あなたはコードを書くのではなくデータを編集することでそのすべてを形づくります。

## Space: 複数チームのための知識

上記のすべては1チームを前提としています。**複数のチームが1つのプロジェクトを共有する** 場合、AI-DLC は各チームの方法論、知識、レコードをそれぞれの **space** — 同一の構造(`memory/`、`knowledge/`、`codekb/`、`intents/`)を持つ `amadeus/spaces/<name>/` — に保ちます。この章を通じてあなたが使ってきた `amadeus/knowledge/` の略記は、実際には `amadeus/spaces/<active-space>/knowledge/` です。1チームの場合、そのアクティブな space は常に `default` であり、この区別は表面化しません。([User Guide の Spaces and Intents 章](../guide/03-spaces-and-intents.md) はエンドユーザー向けの説明で、このセクションはハーネスエンジニアリングの観点です。)

これが、あなたが執筆する知識とルールにとって何を意味するか:

- **チーム知識は space ごと。** あなたが埋める `amadeus/knowledge/amadeus-<agent>-agent/` ファイルは1つの space の内側に存在します。2つ目のチームは埋めるべき自分の空の `knowledge/` ツリーを得ます — あなたのファイルは境界を越えて漏れず、彼らのファイルもあなたのエージェントのコンテキストを希釈しません。
- **方法論層も space ごと。** `amadeus/spaces/<space>/memory/`(`org.md` → `team.md` → `project.md`)のルールはアクティブな space 内で解決されます。新しい space はフレームワークのベースラインから種付けされます — `org.md` がコピーされ、新鮮な空の `team.md` / `project.md` が置かれます — したがって新しいチームはフレームワークのデフォルトから始まり、別のチームのプラクティスを継承するのではなく自分自身のプラクティスを獲得します。
- **space を `core/` で執筆することはありません。** space はランタイムのチームデータであり、インストール済みプロジェクトで `/amadeus space-create <name>` によって作成されます — チーム知識をフレームワークのソースから分ける、データ・非コードの同じ境界線です。複数チームをサポートするために `core/` に追加したり再生成したりするものは何もありません。この機能はエンジンに同梱されています。

この章のメンタルモデルは各 space の内側で変わらず成り立ちます: ステージがエージェントを指名し、エージェントが知識を読み、あなたはデータを編集することでそのすべてを形づくります。space は単に *誰の* データかをスコープするだけです — したがって2つのチームが、コンテキスト、プラクティス、レコードを衝突させることなく、1つのプロジェクトで AI-DLC を実行できます。
