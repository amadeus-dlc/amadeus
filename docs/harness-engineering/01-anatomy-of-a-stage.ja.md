# ステージの解剖学

> 言語: [English](01-anatomy-of-a-stage.md) | **日本語**

ステージは AI-DLC ワークフローの原子 — 1 ステップで*何が*起こるかです。ハーネスエンジニアが行う他のすべての変更はステージファイルの上に構築されるため、これが最初に読むべき章です。読み終わる頃には、任意のステージ `.md` ファイルを開き、各部分を誰が読むのかを知り、ファイルを編集して変えられるフィールドはどれで、フレームワークのコードによって固定されている挙動はどれかを理解できるようになっているはずです。

ステージファイルは、YAML フロントマターを持つ単一の Markdown ファイルです。フロントマターはグラフノード — ステージが消費・生成する成果物、そのリードエージェント、実行方法 — を宣言します。本文はエージェントが従う散文です。両方が 1 つのファイルに存在するのには意図があり、本章の残りではその理由を解き明かします。

---

## 2 人の読者、1 つのファイル

ステージファイルには 2 人の読者がいて、両者は決して重なりません:

- **パーサー**は YAML フロントマターだけを読みます。構造化されたグラフエントリ — エッジ、リードエージェント、実行モード — を生成し、本文は完全に無視します。
- ステージを実行する**エージェント**は本文だけを読みます。散文のステップに従い、成果物を生成し、フロントマターには決して触れません。

両方を 1 つのファイルに保つことで、ファイルを開いたときにグラフのエッジと実行ステップを並べて見ることができます。ステージを読むレビュアーは両方を一度に見ます: それが生成する成果物*と*、それを生成する手順です。両者を分割 — グラフ用の YAML ファイルとエージェント用の散文ファイル — すると、そのインラインな可視性が壊れます。それこそがステージをレビュー可能にしている性質です。

この分割は、あなたが何を編集しているのかを知る鍵です。フロントマターのフィールドを変えれば*グラフ*(依存エッジ、リードエージェント、実行モード)を変えたことになります。本文を変えれば*作業*(エージェントが実際に何をするか)を変えたことになります。両者は独立しています。

実際のステージは `core/amadeus-common/stages/<phase>/<slug>.md` で作成されます — 例えば `stages/inception/application-design.md` です。本章と並べて 1 つ開いてみてください。読み終わる頃には形が馴染んでいるはずです。

---

## フロントマターの概観

フロントマターは YAML キーのフラットなブロックです。一部は機械的なもの — `slug`、`phase` — で、隣接するステージからコピーします。いくつかは実際の判断を伴い、それらがステージを作成・編集する前に理解する価値のあるものです:

| フィールド | 何を決めるか | 判断のポイント |
|-------|-----------------|-------------------|
| `requires_stage` | 依存と順序付けのエッジ | このステージは本当に別のステージの出力を消費するのか、それとも単にその後に実行される必要があるだけか? |
| `consumes` | このステージが読む成果物 | 成果物ごとに: それは `required` か、そして brownfield/greenfield に `conditional_on` か? |
| `produces` | このステージが書く成果物 | これらは他のステージが発見する前方エッジ。 |
| `lead_agent` / `support_agents` | 誰がステージを実行するか | 1 つのペルソナが所有し、サポーターが視点を加える。 |
| `mode` | どう実行するか | `inline`(コンダクターのコンテキスト内)か `subagent`(新しいコンテキストに委譲)。 |
| `for_each` | 反復するかどうか | インスタンスごとに 1 回の実行を駆動する成果物を名指しする。 |

最も強く効いてくる判断についていくつか注記します:

- **`consumes[].required` はグローバルではなくアクティブなプランにスコープされます。** `required: true` は「生成側ステージが*この*ワークフローで実行されるなら、この消費は満たされねばならない」を意味します — 「生成側が常に実行される」ではありません。スコープは意図的に上流ステージをスキップするため、フラットなグローバル要件はそれらのスコープを構造的に無効にしてしまいます。ステージ本文は入力が欠けているケースを優雅に扱います(「生成されていれば」のような散文で)。
- **`consumes[].conditional_on` は brownfield/greenfield の分岐を捉えます。** `conditional_on: brownfield` とマークされた消費は、ワークフローが brownfield のときのみ必須です。無条件の消費については、フィールドを完全に省略します — `always` という値は存在しません。
- **`mode` はディスパッチの仕組みです。** `inline` は短いステージをコンダクター自身のコンテキストで実行します。`subagent` は長いステージ(Construction のコード生成など)を新しいコンテキストに委譲し、メインのコンテキストウィンドウを溢れさせないようにします。複数のエージェントが 1 つのステージに関わることは、`mode` ではなく `support_agents` で表現します — コンダクターはまずリードを、次に各サポーターを順に呼び出します。
- **`for_each` は反復の成果物を名指しします。** Unit ごとに 1 回実行される 5 つの Construction ステージは `for_each: unit-of-work` を宣言します。他のステージはこのフィールドを省略し、1 回だけ実行されます。集約はグラフから推論され、宣言されません。
- **`lead_agent` と `support_agents` は `core/agents/*.md` に対して検証されます。** ハードコードされたリストはありません — エージェントの追加は、そのファイルをそのディレクトリに置くことを意味します([エージェントの追加](03-adding-an-agent.ja.md)を参照)。

これはオリエンテーションであって契約ではありません。型、制約、そして AI-DLC が後に追加する予約名前空間フィールドを含む完全なフィールド表については、開発者リファレンスの[フィールドリファレンス — いつ使うか](../reference/15-stage-definition.ja.md#field-reference--when-to-use)を読んでください。

## 3 コンパートメントの本文

フロントマターの下、本文には 3 つのコンパートメントがあり、常にこの順序です: `## Steps`、`## Sensors`、`## Learn`。`application-design.md` を見ると、3 つすべてが埋められているのがわかります。

- **`## Steps`** はエージェントが従う命令的な散文です — ペルソナをロードし、先行コンテキストを読み、質問ファイルを作成し、成果物を生成し、承認ゲートを提示する。ここがステージのドメイン作業が存在する場所であり、グラフに触れずに*ステージが何をするか*を変えるときに最もよく編集するコンパートメントです。
- **`## Sensors`** はステージの出力に束縛された決定論的なチェックを文書化します。`application-design.md` では、`required-sections` と `upstream-coverage` がステージの Markdown 成果物で発火すること、そして各々が何を検証するかを説明しています。束縛そのものはフロントマター上部の `sensors:` リストです。このコンパートメントは、それらの束縛が何をするかの人間可読な説明です。センサーは[センサー](06-sensors.ja.md)で完全に扱います。
- **`## Learn`** は学習ループの儀礼を文書化します — ステージ実行中にエージェントが保持する `memory.md` の日誌と、保持された観察が承認ゲートでどのようにプラクティスやセンサーへ流れ込むか。重要なのは、この儀礼は*space メモリレイヤー*(`amadeus/spaces/<space>/memory/`)とハーネスのセンサー設定(`.claude/sensors/`)へ書き込み、ステージファイル自身へは決して書き戻さないことです。

これら 3 つのコンパートメントは、v0.5.0 の追加 — 埋められた Sensors と Learn の束縛 — が本文の再構成を強いることなくきれいに収まるよう、あらかじめ宣言されていました。完全な本文モデルと各コンパートメントが含みうる内容は[3 コンパートメント本文モデル](../reference/15-stage-definition.ja.md#three-compartment-body-model)にあります。

内面化する価値のある境界が 1 つあります: ステージファイルはフレームワークの成果物であり、その形は不変です。本文の `## Steps` / `## Sensors` / `## Learn` の構造がワークフローによって書き換えられることは決してありません。唯一認可されたワークフロー内での編集は、学習ループがフロントマターの `sensors:` インポートリストに新しいセンサー id を追記することです。ステージファイルで変えるその他すべては、ハーネスエンジニアとして意図的に変えるものです。

## グラフから見た consumes と produces

依存グラフはどこにも直接書かれていません — それはすべてのステージにわたる `consumes` と `produces` の宣言から立ち現れます。ステージの `produces` リストはその前方エッジです: 別のステージが「`scope-document` を生成するのは誰か?」と尋ねると、答えはそれを宣言しているステージです。ステージの `consumes` リストはその後方エッジです: それらの成果物が存在するまで、そのステージは実行できません。

`requires_stage` は依存を明示的にし、純粋な順序付けもエンコードします。2 つの役割を担います:

1. **意味的なデータ依存** — 「私は成果物 X を消費し、それをステージ Y が生成する」なので Y が `requires_stage` に入る。
2. **提示順序エッジ** — 同じフェーズ内でデータ依存はないが順序が固定された 2 つのステージ。アルファベット順のタイブレークに頼るのではなく、計算された表示順序が安定して定まるよう、弱いエッジを作成する。

ステージを作成または移動するとき、これら 3 つのリストがそれをワークフローへ配線するものです。正しく設定すれば、ステージは正しいスコープに、正しい順序で、入力が満たされた状態で現れます。まったく新しいステージを配線する仕組みは[ステージの追加](02-adding-a-stage.ja.md)でステップごとに扱います。

## stage-graph.json はコンパイルされる — 決して手編集しない

この実装は `dist/claude/.claude/tools/data/stage-graph.json` にある JSON ファイルからグラフを実行します。そのファイルは**ビルド成果物**であって、ソースファイルではありません。ステージファイル全体にわたる YAML フロントマターが正典であり、JSON はそれをコンパイルしたときに得られるものです。

流れはこうです:

```
ステージ .md の YAML を編集  →  コンパイル  →  stage-graph.json  →  実行時が読む
```

任意のステージのフロントマターを編集した後、コンパイルを実行して `stage-graph.json` を再生成し、両方を一緒にコミットします。CI はドリフトチェックを実行し、JSON が YAML のコンパイル結果と一致しなければマージを失敗させます — そのため、再コンパイル忘れは出荷されずに検出されます。

そこから導かれるルール: **`stage-graph.json` を手編集しない。** YAML を編集し、再コンパイルします。手編集は、次のコンパイルで上書きされるか、ドリフトチェックに引っかかるかのどちらかです。コンパイルコマンドとドリフトガードは[作成フロー](../reference/15-stage-definition.ja.md#authoring-flow)に文書化されています。

## ステージが*含むもの*と*行うこと*

本章はステージファイルが*含むもの* — そのフォーマット — についてです。ステージが実行時に*行うこと*については別の補完的な契約があります: 承認ゲート、質問フロー、状態のチェックボックス、完了メッセージ。その挙動的な契約はドメインに関わらずすべてのステージで同一であり、[ステージプロトコル](../reference/04-stage-protocol.ja.md)に存在します。ステージ本文が「承認ゲートについては stage-protocol.md に従う」と述べるとき、それが指している契約がこれです。

頭の中で 2 つを分けておいてください: ステージの*定義*(本章、[リファレンス 15](../reference/15-stage-definition.ja.md))はファイルフォーマットです。ステージの*プロトコル*([リファレンス 04](../reference/04-stage-protocol.ja.md))は、すべてのステージの作業を包む実行時の挙動です。あなたは定義を編集します。プロトコルはほとんどそのままにしておきます。

フロントマターと本文を結びつける完全な注釈付きの例については、[ワークト例](../reference/15-stage-definition.ja.md#worked-example)を参照してください。
