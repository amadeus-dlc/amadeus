# センサー

> 言語: [English](06-sensors.md) | **日本語**

センサーは、エージェントがステージの出力を書き込んだときに自動的に発火する、決定的で助言的なチェックです。ルールがエージェントの読む散文である([ルールと学習ループ](05-rules-and-the-loop.ja.md))のに対し、センサーは実行されるコードです — ルールのフィードフォワードに対する、制御ループのフィードバック側の半分です。ルールは「ユーザーストーリーは Given/When/Then に従う」と言い、センサーはエージェントが書いたばかりのファイルに必要な見出しが存在することを、バイト単位で検証します。

この章は、ハーネスエンジニアがセンサーで実際に行う作業を語ります。同梱される4つを理解し、新しいマニフェストを記述し、それを実行すべきステージにバインドすること。フィールドごとの完全な契約は Developer Reference の [Sensor System](../reference/07-sensor-system.ja.md) にあります — この章は各スキーマの決定点でそこを指し示すだけで、再掲はしません。

---

## センサーとは何か

センサーマニフェストは YAML フロントマター付きの Markdown ファイルで、`core/sensors/` の下に置かれます。フロントマターは純粋な**能力記述子**です — チェックが何であり、どう呼び出すかを述べます。どのステージがそれを使うかについては何も述べません。そのバインディングはステージ側に存在します。それがこの章の中心的な考えであり、マニフェストとステージが疎結合を保つ理由です。

2つの性質がランタイムの振る舞いを定義しており、何かを記述する前に両方を身につけておく価値があります。

- **ステージ中の Write と Edit で発火する。** エージェントが出力ファイルを書き込むか編集すると、`PostToolUse` フックがアクティブなステージに適用されるセンサーを確認し、マッチする各センサーを実行します。ワークフロー中に手作業でセンサーを呼び出すことはありません。すべてのファイル書き込みに便乗します。(マニフェストの `command:` はデバッグ用に人間が実行することもできます — リファレンスを参照 — が、ワークフローの経路はフックです。)
- **助言的で、決してブロックしない。** このリリースでのセンサー結果はテレメトリであり、ゲートではありません。失敗したセンサーは監査行と、何が欠けているかを正確に指し示す詳細ファイルを生成しますが、ステージの承認ゲートやあなたのワークフローを止めることはありません。あなたはシグナルを見て、それをどうするかを決めます。(`default_severity` は今日 `advisory` に固定されています。`blocking` 値は将来のリリース向けに予約されています — 下記の [`default_severity`](#judgment-calls-matches-and-default_severity) を参照。)

各発火は、intent の `audit/` シャードに1行を残します。イベント名 — ログを grep するときは正確な大文字小文字が重要です — は、センサーが開始したとき **`SENSOR_FIRED`**、クリアしたとき **`SENSOR_PASSED`**、抜け漏れを見つけたとき **`SENSOR_FAILED`** です。失敗した行は、`<record>/.amadeus-sensors/<stage-slug>/`(intent のレコードディレクトリ内)の下の詳細ファイルにリンクし、具体的な抜け漏れ — 欠けている見出し、参照されていない上流成果物、lint エラー — を名指しします。実行中にこれがどう見えるかのユーザー向けツアーは、ユーザーガイドの [Rules and the Learning Loop](../guide/09-rules-and-the-learning-loop.ja.md) にあります。

## 同梱される4つのセンサー

`.claude/sensors/` の下に4つのマニフェストが同梱され、それぞれ `amadeus-` が接頭辞に付いています。

| マニフェスト | 発火対象 | チェック内容 |
|----------|----------|--------|
| `amadeus-required-sections.md` | レコードディレクトリの markdown 出力 | 出力が必須の H2 見出しを持つこと — 汎用的なコンテンツ形状チェック |
| `amadeus-upstream-coverage.md` | レコードディレクトリの markdown 出力 | 散文が、ステージが消費すると宣言する各上流成果物を参照していること |
| `amadeus-linter.md` | `.ts` / `.js` コード出力 | 設定されたリンター(デフォルトは ESLint)をラップする |
| `amadeus-type-check.md` | `.ts` / `.tsx` コード出力 | 設定された型チェッカー(デフォルトは `tsc`)をラップする |

4つすべては `matches:` グロブ(詳細は下記)でゲートされます。最初の2つのドキュメント形状チェックは成果物ツリーにスコープし(同梱マニフェストは `**/{amadeus-docs,intents}/**` を持ちます — intent ごとのレコードツリーで、移行前のプロジェクト向けにレガシーの `amadeus-docs/` アームを残しています)、2つのコード品質チェックはそれぞれの言語グロブ(`**/*.{ts,js}`、`**/*.{ts,tsx}`)にスコープします。自分のものを記述する前に、`amadeus-required-sections.md` を最初から最後まで読んでください — 4つの中で最も小さく、フロントマターと散文本文という全体の形を示しています。

## バインディングの仕組み: プル型オーサリング

マニフェストは**ステージターゲティングのフィールドを持ちません**。`applies_to:` はありません — フレームワークが意図的に削除しました。ステージは、自身のフロントマターでセンサーを指名することで、自分の出力に何が発火するかを決めます。

```yaml
# core/amadeus-common/stages/construction/code-generation.md
---
slug: code-generation
phase: construction
sensors:
  - linter
  - type-check
---
```

これは**プル型オーサリング**であり、ハーネスモデルの他のすべてのバインディングと同じ方向です。消費者(ステージ)が能力(センサー)を指名するのであって、その逆ではありません。`sensors:` リストは裸の id — `amadeus-linter` ではなく `linter` — を保持します。id はマニフェストのフロントマターの `id:` フィールドと一致し、それは `amadeus-` 接頭辞を剥がしたファイル名のステムと等しいからです。

見返りは参照の局所性です。ステージファイルを開けば、そのステージが実行されるときにどのチェックが発火するかが正確に見えます — このステージをターゲットにすると主張するものを探して、すべてのマニフェストをスキャンする必要はありません。同じ `sensors:` コンパートメントは、ステージ側から [Anatomy of a Stage](01-anatomy-of-a-stage.ja.md) で説明されています。ここではセンサー側からそれを見ています。

ワークフロー開始時に、コンパイル解決器は `.claude/sensors/` を歩き、すべてのマニフェストを id でインデックスし、各ステージについて宣言された各インポートを引きます — id にマニフェストがない場合はコンパイル時に大声でスローし、ステージが実行されるときに静かに失敗することはありません。解決されたステージごとのビューはステージグラフノードに焼き込まれ、フックは発火時にそこからそれを読みます。心に留めておくべき帰結が1つあります。ワークフロー途中でマニフェストを編集しても、進行中の実行で何が発火するかは**変わりません**。コンパイルスナップショットは次のワークフローが始まるまで保たれます。解決器の完全な仕組みは [How stages import sensors](../reference/07-sensor-system.ja.md#how-stages-import-sensors) にあります。

## 新しいセンサーを記述する

センサーの追加は2つの書き込みです。マニフェスト、そしてバインディング。

**1. `core/sensors/amadeus-<id>.md` にマニフェストを置く。** ファイル名のステム(`amadeus-` 接頭辞を除く)は、フロントマターの `id:` と等しくなければなりません。フロントマターは短く、5つの必須フィールドといくつかのオプションです。

| フィールド | 必須 | それは何か |
|-------|----------|------------|
| `id` | yes | kebab-case。`amadeus-` を除いたファイル名のステムと等しい |
| `kind` | yes | 今日受理される唯一の値は `deterministic` |
| `command` | yes | ディスパッチャーが実行する正規の呼び出し接頭辞 |
| `default_severity` | yes | 今日受理される唯一の値は `advisory` |
| `description` | yes | 1行の人間向け説明 |
| `category` | no | 自由形式のラベル(同梱マニフェストは `document-shape`、`code-quality` を使う) |
| `matches` | no | センサーがどのファイル書き込みで発火するかを絞り込むグロブ |

`command:` は完全な argv ではなく**接頭辞**です。ディスパッチャーは発火時にランタイムコンテキストを追記します — 常に `--stage <slug>`、次にセンサーの入力形状にマッチするファイルフラグ: ドキュメントセンサーには `--output-path <path>`、コードセンサー(`linter`、`type-check`)には `--file-path <path>`。こうしてマニフェストは純粋な能力記述子のままとなり、発火ごとのフラグを決してエンコードしません。ディスパッチャーが組み立てる正確な呼び出しは [`command:` invocation contract](../reference/07-sensor-system.ja.md#command-invocation-contract) に文書化されています。完全なスキーマ — `input_schema`、`output_schema`、`timeout_seconds`、そして未知のキーに対する前方互換ポリシー — については [Sensor Manifest Schema](../reference/07-sensor-system.ja.md#sensor-manifest-schema) を参照してください。

**2. ステージの `sensors:` リストに id を追加してバインドする。** ディレクトリに置かれたマニフェストは、ステージがそれをインポートするまで何もしません。チェックを発火させたいステージを開き、そのフロントマターの `sensors:` リストに裸の id を追加すると、バインディングは次のコンパイルで効きます。複数のステージでセンサーを実行するには、それぞれに id を追加します — strict-additive で、考えるべき上書きレイヤーはありません。あるステージでセンサーの発火を止めるには、そのステージから id を削除します。マニフェストは決して変わりません。インポートリストだけが変わります。

`amadeus-` のファイル名接頭辞は、カスタムのものを含むすべてのセンサーで必須です — コンパイル解決器(`amadeus-graph.ts` の `loadSensors`)は `SENSOR_FILE_REGEX = /^amadeus-([a-z][a-z0-9-]*)\.md$/` でマニフェストを発見し、接頭辞のないファイルを静かにスキップするため、それは決して発見されず、ステージにもバインドされません。センサーは `core/sensors/amadeus-<id>.md` と命名し、`id: <id>` を設定してください。ファイル名対 id のルールは接頭辞の後のステムに適用されます。

## 判断が必要な点: `matches` と `default_severity`

マニフェストの大半は機械的です。2つのフィールドが本当のオーサリングの判断を担います。

**`matches` — どの形状のファイルをこれは解析すべきか?** このグロブは発火フィルターであり、実質的に必須です。フックは、書き込まれるパスがそれにマッチしたときのみセンサーを発火し、`matches` が**ない**エントリは決して発火しません。コード品質センサーはそれをコードグロブ(`amadeus-linter.md` は `**/*.{ts,js}`、`amadeus-type-check.md` は `**/*.{ts,tsx}`)に設定し、コード書き込みでのみ発火して散文には静かなままにします。ドキュメント形状センサーは成果物ツリーにスコープし、ステージが書くあらゆる markdown 成果物で発火します。チェックが意味を持つファイル形状を決め、それをカバーする最も狭いグロブを書いてください。空の `matches: ""` はパース時に拒否されます。そして、グロブがなければセンサーは決して実行されないため、「すべてで発火する」モードはありません — 形状を指定しなければなりません。フックは発火時に、書き込まれるパスをこのグロブと比較します。完全な振る舞いは [`matches` filter](../reference/07-sensor-system.ja.md#matches-filter) にあります。

**`default_severity` — 今のところ advisory のみ。** 今日受理される唯一の値は `advisory` なので、これは自由な選択ではなく固定された選択です。指定する価値があるのは、それがあなたが乗る契約を定義するからです。センサーは知らせるのであって、強制はしません。ゲートを止める `blocking` 値は将来のリリース向けに予約されています。それまでは、すべてのセンサーは人間が読むセカンドオピニオンであり、決して壁ではありません。予約値のポリシーは [`default_severity`](../reference/07-sensor-system.ja.md#default_severity) にあります。

## 学習ループがセンサーをあなたのためにインストールするとき

センサーを常に手作業で記述するわけではありません。§13 学習ゲートは、ワークフロー中にセンサー提案を確認したときに1つをインストールできます — 決定的なチェックがステージの出力で発火すべきだと判断し、ゲートでチェックを入れると、フレームワークはあなたが手作業でやるのと同じ2つの書き込みのインストールを行います。プロジェクト層のマニフェストをあなたのプロジェクトの `.claude/sensors/amadeus-<id>.md`(同梱のフレームワーク配布物ではなく)にスキャフォールドし、発端となったステージの `sensors:` インポートリストに新しい id をアトミックに追記します。そのゲート確認済みの経路は **`SENSOR_PROPOSED`** の監査行を発行するため、バインディングが静かにインストールされることは決してありません。ループとその `SENSOR_PROPOSED` 行は [ルールと学習ループ](05-rules-and-the-loop.ja.md) でカバーされています。ユーザー向けのウォークスルーは、ユーザーガイドの [Rules and the Learning Loop](../guide/09-rules-and-the-learning-loop.ja.md) にあります。

この章の手作業オーサリングの経路と、ループがインストールする経路は、同じ成果物 — マニフェストとインポートリストのエントリ — を生みます。違いは誰が起点になるかです。あなたがファイルを直接編集するか、あるいはゲートが、ワークフロー途中であなたが行った是正を捉えるか。
