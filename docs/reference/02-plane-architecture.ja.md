# プレーンアーキテクチャ

> 対象読者: Tier 2/3(チーム採用者、フレームワークコントリビューター)。

この章は AI-DLC の3プレーンアーキテクチャ — control、data、management の各関心事の分離 — と、それらを結合するコンパイル境界を説明します。[Sensor System](07-sensor-system.ja.md)(control プレーンのセンサーマニフェスト)、[Rule System](08-rule-system.ja.md)(control プレーンのルールファイル)、[State Machine](12-state-machine.ja.md)(data プレーンのライフサイクル)と相互リンクします。これは設計の基盤であり、センサーおよびルールのマニフェストは、その基盤が操作するソースファイルです。

---

## 1. 課題

v0.5.0 は、ルール、センサー、そしてワークフロー実行中に観察を捕捉し、次回自動的に発火する場所へ書き込む学習ループを導入します。基盤となるモデルには実際の複雑さがあります: 複数種類のステージごとのインポート、ファイル名で付与されるユニバーサルデフォルトルール、センサーマニフェスト、フックチェーン。顧客はフレームワークを使うためにそのすべてを学ぶ必要があってはならず、フレームワークはホットパスでルックアップコストを支払うべきではありません。

正しいフレーミングは「設計をより単純にする」ことでは *ありません*。「*このような種類のシステムのために既に存在する規律を借りる*」ことです。現代のネットワーキングの3プレーンアーキテクチャが最も近い類似物であり、その教訓はほぼ直接転用できます。

---

## 2. ネットワーキングの類推

現代のルーター(または SDN コントローラ)は、その仕事を3つのプレーンに分割します:

- **Management プレーン** — SNMP、syslog、ダッシュボード、NETCONF、gRPC、CLI。設定を入れ、可観測性を出す。人間のケイデンス(一度設定し、時折クエリする)。
- **Control プレーン** — BGP、OSPF、IS-IS、経路計算。トポロジが与えられたら、各パケットがどの経路を取るべきかを決定する。トポロジのケイデンス(秒から分)。
- **Data プレーン** — フローテーブル、ACL、ASIC フォワーディング。このパケットを今、ナノ秒で、テーブルルックアップにより転送する。ラインレート。

盗む価値のある5つの教訓:

### フィルタ評価はパケットレートでは決して行われない

OSPF はトポロジが変化したとき最短経路を一度計算します。その結果は data プレーンのフローテーブルにインストールされます。以降のすべてのパケットはテーブルルックアップ — ナノ秒、決定論的、フィルタロジックなし。賢い仕事は *パケット時* ではなく *トポロジ変化時* に行われます。

### プレーン間のインターフェースは明示的

OpenFlow、P4、NETCONF、gRPC。control プレーンは data プレーンのメモリに手を伸ばしません。「このフローをインストールせよ」と告げる構造化メッセージを送ります。data プレーンが確認します。これにより故障モードが診断可能になります — 拒否されたインストールは、静かなミスではなく明確なシグナルです。

### 故障はクリーンに分離する

control プレーンがクラッシュしても、data プレーンは最後にインストールされたルールを使って転送を続けます。data プレーンが故障すれば、control プレーンがハートビートで気づき、故障を迂回して再ルーティングします。どちらのプレーンも他方を巻き添えにしないことは、負荷を支える性質です。

### テレメトリがループを閉じる

data プレーンはフロー統計、ドロップ、レイテンシを報告します。control プレーンはテレメトリを消費して決定します: 輻輳を迂回して再ルーティング、サービスをスケール、ピアをダウンとしてマーク。*data プレーンが報告し返すことで、control プレーンはより賢くなります。*

### 3つのプレーン、3つのケイデンス

management プレーンは人間のケイデンスで動きます。control プレーンはトポロジのケイデンスで動きます。data プレーンはラインレートで動きます。各プレーンは、その仕事が許すなら遅くてもよいのです。

## 3. AI-DLC へのマッピング

マッピングは聞こえるより近いものです。(以下の `<record>/` = アクティブな intent の record dir、`amadeus/spaces/<space>/intents/<YYMMDD>-<label>/`。監査証跡は `<record>/audit/` 下のクローンごとのシャードのディレクトリです。)

| ネットワーキング | AI-DLC 類似物 |
|---|---|
| Control プレーン(BGP、OSPF、経路計算) | ステージ定義、ルール、センサー — 何が実行されるべきかのスキーマ |
| Data プレーン(パケット転送、フローテーブル) | ステージ実行、Bolt、エージェント呼び出し — 実際の実行 |
| Management プレーン(SNMP、ダッシュボード、CLI) | `/amadeus --doctor`、デザイナー(将来)、監査クエリ、`CLAUDE.md` |
| ルーティングプロトコル(BGP / OSPF) | コンパイル: `amadeus-graph.ts compile` がステージフロントマター + ルール + センサーを読み、各ステージのプルインポートをソースレジストリに対して解決し、グラフを発行する |
| ASIC にロードされた FIB(forwarding information base) | コンパイル時に解決されたステージごとの `rules_in_context` + `sensors_applicable` を持つ `stage-graph.json` |
| OpenFlow / NETCONF(インターフェース) | `stage-graph.json` — コンパイルとオーケストレーターの間の明示的インターフェース |
| テレメトリ(NetFlow、sFlow) | 監査ログ、センサー発火、memory.md エントリ → intent の `runtime-graph.json` |
| リアクティブなフローインストール(PACKET_IN) | 学習ループ: data プレーンが観察を報告 → ユーザーが確認 → rules/sensors へのファイル書き込み、加えて新しいセンサーバインディングが捕捉されたとき起点ステージのフロントマター編集 |
| プロアクティブな経路インストール(BGP アドバタイズメント) | フレームワーク PR: どのワークフローが実行される前にも新しいステージ/ルール/センサーを出荷する |
| トポロジ変化 → 経路の再計算 | ワークフロー開始 → コンパイルが現在のソースファイルを読む。以降の再計算は次のワークフローによってトリガーされる |
| プリインストールされた FIB がパケット飛行中に安定 | ワークフロー中の学習ループ書き込みは、飛行中のコンパイル済みビューに影響しない。次のワークフロー開始時に適用される(§5 を参照) |
| テーブルルックアップによるラインレート転送 | オーケストレーター + ディスパッチャがグラフノードから事前解決されたフィールドを読む。ランタイムでの解決ウォークなし |

`stage-graph.json` のプレーンラベルとランタイムの読み取りパターンは、FIB/経路テーブル/CLI の分離をミラーします。設定は management プレーンで入り、control プレーンがそれをコンパイルし、data プレーンがルックアップで実行します。

## 4. コンパイル境界

負荷を支える洞察: ネットワーキングは、その仕事が高価すぎるため、フィルタ評価や継承ウォークをパケットレートに置きません。control プレーンが一度計算し、FIB にインストールし、data プレーンがルックアップで転送します。AI-DLC は同じパターンに従います: ワークフロー開始時に一度コンパイルし、ワークフロー全体を通じて解決済みビューを読み、次のワークフロー開始時に再コンパイルします。

### 何がコンパイルされ、何がディスク上に留まるか

| 状態 | ライフサイクル | ディスク上のソース | コンパイル先 | 読み手 |
|---|---|---|---|---|
| ステージ DAG、スコープルーティング、成果物生成 | フレームワークバージョン管理(フレームワーク PR で変化) | ステージフロントマター(`.claude/amadeus-common/stages/*.md`) | `stage-graph.json` | オーケストレーター、doctor、デザイナー |
| **ルール**(散文、規範的) | 可変。フレームワーク PR または学習ループの書き込み | `amadeus/spaces/<space>/memory/<scope>.md`(ファイル名由来。org/team/project はすべてのステージに付与) | `stage-graph.json` のノードごとの `rules_in_context` | オーケストレーター(解決済みビュー)。Claude Code 自動ロードはインコンテキスト散文のためにソースを読む |
| **センサー**(マニフェスト、検証チェック) | 可変。フレームワーク PR または学習ループの書き込み(マニフェストは一度作成。ステージが id でインポート) | `.claude/sensors/amadeus-<id>.md` | `stage-graph.json` のノードごとの `sensors_applicable` | ディスパッチャがステージ入場時に解決済みリストを読む。PostToolUse がそこから発火 |
| ワークフロー実行テレメトリ | ワークフローごと、累積 | `audit/` シャード · `memory.md` · Bolt フォーク | `<record>/runtime-graph.json` | doctor、ゲートの儀式、将来のクロスワークフローオブザーバー |
| ステージごとの観察ログ | ステージ実行ごと | `<record>/<phase>/<stage>/memory.md` | (コンパイルなし — 直接読む) | このステージのゲートでのゲートの儀式 |

コンパイル成果物は2つ、どちらもワークフローごとです。`stage-graph.json` は解決済みの control プレーンビューを運びます: ステージ DAG、スコープルーティング、成果物生成、加えて継承が事前解決されたステージごとのルール・センサーリスト。`runtime-graph.json` は再コンパイルされます — 監査ログの完全なイベントソーシングウォーク — 遷移クラスの監査イベントごとに。ディスク上のソースファイルが作成サーフェスです。コンパイル済みグラフがランタイムの読むものです。

### 1回のコンパイル、ワークフロー開始時

コンパイルはステージフロントマターを読み、`amadeus/spaces/<space>/memory/` と `.claude/sensors/` を走査し、ファイル名でユニバーサルデフォルトルールを付与し(`org.md`、`team.md`、`project.md` はすべてのステージに適用)、その後各ステージのプルインポートをソースレジストリに対してルックアップします:

- ステージの `phase: <name>` フィールドが、対応する `phases/<name>.md` ルールを付与します(ステージごとに1ルール)。[Rule System](08-rule-system.ja.md) を参照。
- ステージの `sensors: [<id>, ...]` リストが各 id を `.claude/sensors/` に対して解決します。未知の id はコンパイルを大声で失敗させます — ステージが、自分がインポートするものの発火に静かに失敗することはできません。[Sensor System](07-sensor-system.ja.md) を参照。

コンパイルは、答えを各ステージノードに焼き込んだ `stage-graph.json` を発行します。ワークフロー全体を通じて、オーケストレーターとディスパッチャはそれらの事前解決フィールドを読みます。ワークフロー中の学習ループ書き込みはソースファイルを更新しますが、飛行中のコンパイル済みビューには影響しません — ユーザーは既にステージ内でオーケストレーターを修正しており、そのルールは次回のためのものです。次のワークフローのコンパイルがそれらを拾います。BGP がパケット飛行中に経路を再計算しないのと同じ形です。

### ロックされ、アトミック

コンパイルが初日から対処しなければならない2つの故障モード。この実装のBoltごとの worktree は、ほとんどの場合、並列エージェントの状態を分離しますが、`data/stage-graph.json` はリポジトリ共有であり、worktree スコープではありません — そしてユーザーは同じチェックアウトに対して2つのターミナルで `/amadeus` を起動できます — ので、コンパイルはいずれにせよ防御を必要とします。

- **並行コンパイル** は `data/stage-graph.json` への書き込みでレースします。コンパイルは v0.4.0 以降 `withAuditLock`(`lib.ts` を参照)の下で実行されるので、並行呼び出しは直列化されます — 2番目は1番目の完了を待ち、その後新鮮なソース状態に対して実行します。
- **書き込み途中のクラッシュ。** 書き込み途中で中断されたコンパイルは、消費者に不正な JSON を読ませてはなりません。コンパイルはテンポラリファイルに書き込み、出力を検証し、その後 POSIX `rename(2)` を介してアトミックにリネームします。読み手は前回のコンパイル出力か新しいものか、どちらかを見ます — 半分書かれたファイルを見ることは決してありません。

両方のパターンは既に状態・監査作業のためコードベースに存在します。コンパイルはそれらを継承します。

### 2つのグラフ、2つのライフサイクル

2つのコンパイル成果物は異なる消費者と異なる更新ケイデンスを持ちます:

- **`stage-graph.json`** — control プレーン。ワークフロー開始時にコンパイル。ワークフローの全生存期間を通じて安定。オーケストレーター(DAG トポロジ、スコープルーティング)、ディスパッチャ(ステージごとの `sensors_applicable`)が読む。Claude Code の自動ロードは散文コンテキストのためにソースルールファイルを並行して消費する。
- **`runtime-graph.json`** — data プレーン。`<record>/runtime-graph.json` のワークフローごとの成果物。遷移クラスの監査イベントごとに再コンパイル(監査ログの完全なイベントソーシングウォーク)。実行テレメトリを集約する: どのステージが実行されたか、どの Bolt がフォークしたか、どのセンサーが発火したか、どの memory.md ファイルが存在するか。ゲートの儀式(候補を表面化する)、doctor(実行健全性)、将来のクロスワークフローオブザーバーが読む。スキーマ、コンパイルライフサイクル、リカバリモデルについては [Runtime Graph](13-runtime-graph.ja.md) を参照。

これらを混ぜるとキャッシュ可能性の境界を侵害します。`stage-graph.json` はワークフローの生存期間キャッシュされたい。`runtime-graph.json` はイベントごとに変化する。同じ成果物にすると、data プレーンがセンサー発火を報告するたびに control プレーン全体を再発行することになります。

### なぜルールとセンサーは同じコンパイルを流れるのか

コンパイルはすべての control プレーン入力にわたって対称的です。ステージフロントマター、ルールファイル、センサーファイルはすべてワークフロー開始時に `amadeus-graph.ts` によって読まれ、すべてが解決済みのステージごとビューに寄与します。ネットワーキングもこれを行います — BGP と OSPF は同じ FIB に供給し、ACL とポリシーは同じフローテーブルに供給します。1つのコンパイル済みビュー、複数のソース。

具体的には、各ステージノードは2つのフィールドを得ます:

```json
{
  "slug": "requirements-analysis",
  "phase": "inception",
  "sensors": ["required-sections", "upstream-coverage"],
  "rules_in_context": [
    {"path": "amadeus/spaces/default/memory/org.md", "scope": "org"},
    {"path": "amadeus/spaces/default/memory/team.md", "scope": "team"},
    {"path": "amadeus/spaces/default/memory/project.md", "scope": "project"},
    {"path": "amadeus/spaces/default/memory/phases/inception.md", "scope": "phase"}
  ],
  "sensors_applicable": [
    {"id": "required-sections", "path": ".claude/sensors/amadeus-required-sections.md"},
    {"id": "upstream-coverage", "path": ".claude/sensors/amadeus-upstream-coverage.md"}
  ]
}
```

`matches`(解決済みセンサーエントリに存在する場合)は、センサー側のケイパビリティフィルタ — *「このセンサーはこの glob にマッチするファイルを分析する」* — で、コンパイルがマニフェストからステージの解決済みエントリへ逐語的にスナップショットします。PostToolUse フックは、マニフェストを再オープンすることなく発火時にそれを読みます。

2つの帰結:

- **ディスパッチャはランタイムで解決ウォークを行いません。** ステージ入場時にノードから `sensors_applicable` を読みます — 既にルックアップ済み、既に付与済み。PostToolUse はこの事前解決リストから発火します。
- **doctor とデザイナーは1つの解決済みビューをクエリします。** 「このステージにどのルールが適用されるか?」「このファイルを編集するとどのセンサーが発火するか?」どちらも `stage-graph.json` から直接答え、ファイルシステムウォークはありません。

### ネットワーキングの教訓、適用

control プレーン(コンパイル)は遅く賢い仕事を一度行います。その後 data プレーン(オーケストレーター + ディスパッチャの読み取り)が解決済みビュー上で高速に実行します。ワークフロー開始時のコンパイルがすべてのステージごとインポートを解決します — すべての `sensors:` id がルックアップされ、すべての `phase:` が付与され、ユニバーサルデフォルトルールがファイル名で拾われます。ワークフロー全体を通じて、ランタイムはグラフノードから事前解決フィールドを読みます。BGP はパケット飛行中に経路を再計算しません。AI-DLC はワークフロー途中で再コンパイルしません。ワークフロー中の学習ループ書き込みはソースファイルを更新し、次のワークフロー開始時にコンパイル済みビューへ入ります。

## 5. 創発的性質としてのリカバリ

今説明した data プレーン — `runtime-graph.json`、監査ログ、状態ドキュメント、`memory.md`、成果物ツリー — には、対価を払っていない性質があります。圧縮やクリーンな再起動の後に開始された新鮮なハーネスセッションは、これら5つのソースを一緒に読み、前のセッションが中断した場所から再開するのに十分なワークフロー状態を再構築できます。リカバリは v0.5.0 に追加された機能ではありません。設計のデータ規律の創発的性質です。

### 5つのソース、1つの絵

| ソース | 何を記録するか | 読む順序 |
|---|---|---|
| 成果物ツリー(`<record>/<phase>/<stage>/*.md`) | 決定そのもの、完成した形で | 最初 |
| ステージごとの `memory.md` | 意思決定中に気づかれたこと | 2番目 |
| 監査ログ(`<record>/audit/` シャード) | 各決定がいつ起きたか、誰が承認したか | 3番目 |
| 状態ドキュメント(`<record>/amadeus-state.md`、ステージごとの状態) | 今ワークフローのどこにいるか | 4番目 |
| `runtime-graph.json` | ステージ横断サマリー(所要時間、センサー発火、学習カウント) | 5番目 |

成果物が最初に来るのは、実際に合意されたものの永続的な記録だからです。他の4つのソースは道のりに関するものです — 何が検討されたか、いつ起きたか、何が保留か、ステージ横断でパターンがどう見えるか。他人の半分終わった作業を引き継ぐ人間も同じように読みます: まず出力、次にノート、次にタイムライン、次に現在のカーソル、最後にサマリービュー。

### リカバリがクリーンに再構築するもの

決定、出力、ステージ内コンテキスト、タイムライン、現在の位置、そしてどの Bolt がどんな結果でフォークしたか。5つのソースを読む新鮮なセッションは、ワークフローが何を生成したか、ユーザーがどんな修正をしたか、どのゲートが承認されたか、何がまだ保留か、そして前の実行がおおよそどう進んでいたかを知ります。

### リカバリが再構築できないもの

前のセッションの会話のリズム — ユーザーのステージ途中の Q&A パターン、`memory.md` にコミットされる前の LLM の作業仮説、飛行途中で中断された部分的なツール呼び出し。会話バッファはセッションとともに消えます。これは修正可能な設計上の性質ではなく、LLM セッションの根本的な性質です。リカバリの仕事は、前のセッションの正確な精神状態を再現することではなく、新しいセッションが再方向づけして続行するのに十分なものを与えることです。

### 一貫性の制約

異なるコードパスによって書かれた5つのソースは、リカバリが機能するために、すべて合理的に一貫性を保たなければなりません。監査ログが正規(append-only、「何が起きたか」の信頼できる情報源)であり、他の4つはそれと一貫します。なぜなら、それらの書き込みは v0.4.0 の同じ `withAuditLock` プリミティブでゲートされているからです。5つのいずれかを監査ログに対して読む新鮮なセッションは、ドリフトを検出し調整できます。

### これがフレームワークにとって意味すること

フレームワークは既にこの小さなバージョンを実践しています: 今日の `amadeus-state.md` は intent 誕生時に書かれセッション再開時に読まれる `Scope` フィールドを運ぶので、ワークフローのスコープはオーケストレーターが再導出することなくコンテキスト圧縮を生き延びます。一般化は、何か永続的なものを記録する data プレーンのすべての部分が、状態ファイルだけでなくリカバリサーフェスの一部になるということです。より深い原則は、*data プレーンがリカバリのために構造化されている* ということです: すべての永続的な記録がリカバリサーフェスの一部であるため、再開パスは、既に存在する基盤の上への小さな追加であり、後付けではありません。
