# ルールシステム

> 対象読者: Tier 2/3(チーム導入者、フレームワーク貢献者)。

この章は v0.5.0 のルールシステムに対するスキーマレベルのリファレンスです。ルールファイルがどこに置かれるか、スコープがどう導出されるか、継承チェーンがどう解決されるか、そしてどのフロントマターフィールドが有効かを扱います。これはリゾルバ(`amadeus-graph.ts compile`)と doctor のルールドリフトチェックが照合する仕様書です。ルールは制御ループのフィードフォワード側の半分であり、[センサーシステム](07-sensor-system.md)がペアとなる決定論的検証側の半分を扱います。ユーザー向けのウォークスルー — 学習ループの儀式、ANZ の実例、確定した学習がどのようにルールファイルに落ち着くか — については、User Guide の [ルールと学習ループ](../guide/09-rules-and-the-learning-loop.md) を参照してください。

## レイアウト

ルールはスペースのメモリ層 `amadeus/spaces/<space>/memory/` に置かれ(ワークスペースルートに手編集可能なセットが1つあり、各ハーネスがそのネイティブなインクルード機構 — Claude の `@`-import スタブ、Kiro のリソース glob、Codex の `AMADEUS_RULES_DIR` — を通じて読み込む)、スコープ名を冠した中立的なファイルとして構成されます:

```
amadeus/spaces/<space>/memory/
├── org.md
├── team.md
├── project.md
└── phases/
    ├── ideation.md
    ├── inception.md
    ├── construction.md
    └── operation.md
```

このレイアウトは、チームが著述するハーネス設定(以前は別の practices 名前空間にあった)と自己学習ガードレール(以前は2階層下にあった)を、`amadeus/spaces/<space>/memory/` という単一のスペースメモリディレクトリに統合したものです。

## ファイル名由来のスコープ

ルールファイルは `scope:` フロントマターフィールドを **持ちません**。スコープはファイル名から導出されます:

| ファイル名パターン | スコープ |
|---|---|
| `org.md` | `org` |
| `team.md` | `team` |
| `project.md` | `project` |
| `phases/<phase>.md` | `phase`(フェーズ値 = ファイル名) |

org、team、project、phase のルールはパススコープのフロントマターを持ちません — プル著述方式が関係をステージ側に置きます。org / team / project はユニバーサルなデフォルトとしてすべてのステージに適用されます(ファイル名由来)。対応するフェーズルールは、ステージのフロントマターの `phase: <name>` フィールドが `phases/<name>.md` に対するプルインポートとなることで付随します。

## 5層継承

ルールはワークフロー開始時に5層のチェーンを通じて解決されます:

```
org → team → project → phase → stage
```

org はフレームワークのデフォルトを担い、team 層と project 層がチーム承認済み・プロジェクト特化のコンテンツで拡張します。phase は直交しており — ステージが既にフロントマターで `phase: <name>` を宣言しているため、対応する `phases/<name>.md` ルールが付随します(`requires_stage` や `consumes` と同じ著述方向)。ステージルールは将来利用のために予約されています。著述された際には、各 `amadeus-stage-<slug>.md` がステージの `slug:` 宣言を通じて付随します。このチェーンが具体化する、より広範な2軸設定モデルについては [01-architecture.md § 設定層](01-architecture.md) を相互参照してください。

コンパイル出力(`stage-graph.json` のステージごとの `rules_in_context` フィールド)は、解決済みのチェーンを各ステージノードに焼き込みます。ランタイムがチェーンを辿ることは決してなく — コンパイルが解決を所有します。

## 厳密加算的ランタイムモデル

適用可能なすべてのルールが `rules_in_context` に現れます。org、team、project のルールは連結され、ランタイムで何も脱落しません。フェーズルールは、そのルールのファイル名がステージの `phase:` 宣言と一致したときに付随します — glob フィルタも、具体パスの合成もありません。エージェントはセッション開始時にチェーン全体を読みます。

矛盾(より狭いスコープがより広いポリシーに反するもの)は、メモリゲート — §13 の学習の儀式(Learnings Ritual)の受け入れチェック — において、学習がリゾルバに到達する前に却下されます。このチェックはセクションレベルです。提案された日付付き学習エントリが `memory/project.md`(または `memory/team.md`)に書き込まれようとするとき、オーケストレーターはそれを `memory/org.md` の対応する見出しと LLM チェックで比較します。矛盾が見つかった場合、ユーザーは **修正・スキップ・エスカレーション** します(オーバーライドの経路はありません)。practices-discovery の承認ゲートがもう1つの受け入れゲートですが、そのプロモーションは決定論的なセクション置換(`amadeus-state.ts practices-promote`)であり、ユーザーの承認によって正当化されます — 自動的な org 矛盾チェックは実行されません。org と team/project コンテンツ間の書き込み後のドリフトは、doctor のルールドリフト行(後述)によって別途表面化されます。

この設計は、以前の `enforcement: enforced` キーワードと `overrides:` ブロックモデルを置き換えます。両キーワードともスキーマから削除されています。フロントマターのパースは、後述の未知キー許容ポリシーによってそれらを却下します — 例外を投げるのではなく黙って通過させますが、リゾルバは無視します。

doctor のルールドリフトチェックは、書き込み後のドリフトをオンデマンドで表面化します。org ルールが team または project コンテンツが既にディスク上に落ちた後に変更されると、doctor は team/project のプラクティスファイル(`memory/team.md`、`memory/project.md`)が *内容の埋まった* org 見出しと共有している `##` 見出しを決定論的に見つけ出し、各重複を助言候補として表面化します — ファイル、セクション、そして引用された org の文 — `Rule drift: N team/project rule(s) overlap org policy (review for contradiction)` としてレンダリングされます。doctor 自体は LLM を実行しません。検出はバイト再現可能な見出し/文字列処理です。矛盾の判定 — 受け入れゲートが実行するのと同じセクションレベルの LLM チェック — は、それを消費するオーケストレーターが観測時に、ノンブロッキングで行います。その後、人間が表面化されたドリフトを修正・エスカレーション・受け入れします。

## `pairing:` フィールド

ルールは `.claude/sensors/amadeus-*.md` の決定論的センサーとペアリング **できます**:

```yaml
---
pairing: amadeus-required-sections
---
# または
pairing: feedforward-only
---
```

有効な値:

- `feedforward-only` — そのルールにセンサーの相棒がないことの明示的な宣言(フレームワークが決定論的に検証できない)。
- `<sensor-id>` — 既存の `.claude/sensors/amadeus-<id>.md` の `id:` フィールドと一致しなければならない。

doctor のペアカバレッジ行は、ペア済み対 feedforward-only のルールをカウントし、ペアのないルールをカバレッジギャップとして表面化します([ルールドリフト検出](#rule-drift-detection)を参照)。

## ルールドリフト検出

`/amadeus --doctor` は、ルール/センサーの状態を観測する2つの助言行を出荷します。両方とも読み取り専用で常にパスします — どちらもヘルスチェックの終了コードを変更しません。(v0.6.10 時点で `--doctor` はコールドセーフです。ペアカバレッジチェックが発する `GUARDRAIL_LOADED` 監査行は、アクティブな intent の `audit/` シャードが既に存在するときにのみ書き込まれます。まだ intent のない新しいシェルでは、doctor は行を表示しますが何も発せず、ファイルも作成しません。)

- **ルールドリフト** — 各 team/project プラクティスファイル(`memory/team.md`、`memory/project.md`)について、doctor は `memory/org.md` 内で *内容の埋まった* 見出しの下にも現れる `##` 見出しを見つけ、各重複を候補ペアとして表面化します: ファイル、セクション、そして最初の org の文をそのまま引用したもの。org の本文が空である見出し(例: フレームワークデフォルトの `## Forbidden`、`## Mandated`、`## Corrections` は HTML コメントのみを保持)はカウントされません — 重複は両側にコンテンツを持たなければなりません。カウント N は *構造上の* 候補ペアの数であり、LLM で確定された矛盾の数ではありません。doctor は決定論的に検出し、矛盾の判定はオーケストレーター LLM が観測時に行います([厳密加算的セクション](#strict-additive-runtime-model)を参照)。
- **ペアセンサーカバレッジ** — `pairing: <sensor-id>` を持つ各ルールについて、doctor は `amadeus-` プレフィックスを剥がし、名前付きセンサーが少なくとも1つのステージの解決済みセンサーセット(`sensors_applicable`)に存在することを確認します。この行は `Paired sensor coverage: P/(M-X) guardrails paired (X feedforward-only)` と読めます。ここで M は `pairing:` 値を持つルール、X は feedforward-only のルール(センサーを必要としない)、P は名前付きセンサーが解決するルールです。ペアのないルール(センサー id が指定されているがどのステージにもバインドされていない)はインラインで列挙されます。これは **ファイル存在チェックであり、意味的なチェックではありません** — バインディングが解決されることを確認するもので、センサーがルールに適合することを確認するものではありません。この行は、初期化済みプロジェクトで実行ごとに1回 `GUARDRAIL_LOADED` 監査イベントを発します(`audit.md` のない真新しいプロジェクトでは emit が抑制されます — 上記のコールドセーフ注記を参照)。

## 前方互換性ポリシー

ルールのフロントマターは加算的拡張によって前方互換です:

- 新しいフィールドは加算的に導入されます — 既存のルール行を書き換える必要はありません。
- コンシューマは未知のフロントマターキーを許容(無視して通過)しなければ **なりません**。これがレガシーな `enforcement:` / `overrides:` キー(ユーザー拡張オーバーレイに存在する場合)がエラーなく読み込まれる仕組みです。
- ルール本文の規約(`## Forbidden`、`## Mandated` などのセクション)は、既存のガードレール規約に従います。
- 学習の儀式(メモリゲート)は、確定した各学習をプラクティスとしてスペースメモリファイル — `memory/project.md`(デフォルト)と `memory/team.md`(ワンクリックでプロモート、org スコープへの書き込みはなし)— に、その diary 見出しのトピックの下に日付付きエントリとして直接書き込みます。確定した学習は *即ち* プラクティスです。これらは practices-discovery が承認するのと同じファイルであり、別の `*-learnings.md` サーフェスではありません。リゾルバはそれらをクリーンな整数チェーン `SCOPE_PRIORITY`(org:0、team:1、project:2、phase:3)でソートします — 1.5 / 2.5 のような分数階層は存在しません。`amadeus-learnings.ts` のメモリゲートは、上記の `## 厳密加算的ランタイムモデル` で説明した受け入れ矛盾チェック(単一の提案された日付付きエントリ 対 `memory/org.md` の対応するセクション。修正 / スキップ / エスカレーション、オーバーライド経路なし)の実際の実装者です。

v0.5.0 以前のルールファイルは、リネーム + フラット化が完了した後も変更なしで読み込まれます。移行はパスのみです。
