# ルールと学習ループ

ルールは、エージェントがあなたのプロジェクトでどのように働くかを形づくる、永続的な散文の指示です。学習ループは、それらのルールがどのように書かれるかの仕組みです: あなたはワークフロー中にエージェントを修正し、ステージの承認ゲートでその修正を確認します。すると、フレームワークがそれを保存し、同じ修正が二度と必要にならないようにします。

本章はユーザー向けのツアーです。ルールがどこに存在するか、5 つの層がどのように積み重なるか、学習ループが修正をどのように捕捉するか、そしてセンサーがルールと並んで決定論的なセカンドオピニオンをどのように提供するかをカバーします。スキーマレベルの仕組みについては、各ステップで開発者リファレンスへ相互リンクしています。

---

## ルールの概観

ルールは、space メモリ層の `amadeus/spaces/<space>/memory/` に Markdown ファイルとして存在します — ワークスペースルートにある単一の手編集可能なセットで、各ハーネスがそのネイティブインクルード(Claude の `@` インポートスタブ、Kiro のリソース glob、Codex の `AMADEUS_RULES_DIR`)を介して読み取ります。各ファイルは、そのスコープにちなんで命名されます:

```
amadeus/spaces/<space>/memory/
├── org.md                 # フレームワーク + 組織全体のデフォルト
├── team.md                # あなたのチームが確約したプラクティス
├── project.md             # このプロジェクトの特化
└── phases/
    ├── ideation.md
    ├── inception.md
    ├── construction.md
    └── operation.md
```

ファイル内に `scope:` フィールドはありません — スコープはファイル名から来ます。`org.md` はフレームワークのデフォルト(トランクベース開発、テスト姿勢、walking-skeleton ポリシー)を担います。`team.md` は、あなたのチームが確約したものを保持します。`project.md` は、この 1 つのプロジェクトに固有のものを保持します。4 つの `phases/<phase>.md` ファイルは、あるフェーズのすべてのステージに適用されるルールを担います — たとえば、inception フェーズのルールは、すべてのアーキテクチャ決定が少なくとも 2 つの代替案を文書化することを要求します。

各ファイルは、トピック別の見出し(`## Way of Working`、`## Testing Posture`、`## Deployment`、`## Code Style` など)の下のプレーンな散文です。あなたはそれらを読み、手で編集でき、フレームワークは学習ループを通じてそれらに書き込みます。

スキーマ、ファイル名からスコープへの対応表、そしてリゾルバの仕組みは、開発者リファレンスの [ルールシステム](../reference/08-rule-system.ja.md) に文書化されています。

## 5 層チェーン

ルールは、すべてのワークフローの開始時に 5 層チェーンを通じて解決されます:

```
org → team → project → phase → stage
```

モデルは**厳格加算的**です。適用可能なすべてのルールがエージェントのコンテキストに現れます — 何も静かに落とされず、ランタイムで上書きされることもありません。org のデフォルト、team のプラクティス、project の特化が連結されます。一致するフェーズルールが付加されます。ステージがすでにそのフェーズを宣言しているからです。(第 5 層、ステージごとのルールは、将来のリリース用に予約されています。)

これは、以前のバージョンからの意図的な変更です。もはや `overrides:` ブロックも `enforcement:` キーワードもありません。適用可能なすべての層が一度に存在し、矛盾はルールが書かれるとき、ランタイムに到達する前に捕捉されます([受理時の矛盾チェック](#admission-time-conflict-checks) を参照)。

チェーンは、フレームワークがステージ定義、ルール、センサーを単一のグラフにコンパイルするとき、ワークフロー開始時に**一度**解決されます。ワークフロー全体を通じて、エージェントは解決されたビューを読みます。実行中にチェーンを再度辿ることは決してありません。そのコンパイル境界は、planes モデルが記述するものと同じです — 本章末尾の [Planes: どのように組み合わさるか](#planes-how-it-fits-together) を参照してください。

## 学習ループ

学習ループは、一度きりの修正を耐久性のあるルールに変える仕組みです。ほとんどのステージ実行は何も追加しません — そしてそれは健全です。ループは、ステージ中に浮上した何かを、あなたが保持する価値があると判断したときにのみ発火します。

ループには、ユーザーに見える 4 つの瞬間があります: エージェントはステージ実行中に日記をつけ、ゲートが候補を浮上させ、あなたが何を保持するかを確認し、フレームワークが保持された項目を次回のために書き込みます。

### memory.md 日記

ステージ実行中、フレームワークは `<record>/<phase>/<stage>/memory.md` に実行中の観察ログを保持します — intent の record ディレクトリ `amadeus/spaces/<space>/intents/<YYMMDD>-<label>/` の配下です。これはステージ開始時に自動的に作成され、あなたのために維持されます — 決して手編集しません。エントリは 4 つの標準見出しの下に入ります:

- **Interpretations** — ステージの散文が曖昧だったところでエージェントが下した選択
- **Deviations** — エージェントが意図的にステージの散文から逸脱した箇所と、その理由
- **Tradeoffs** — 検討された代替案と、選択の理由
- **Open questions** — 次の実行前に確認すべきこと、またはまだ不確かなコンテキスト

各エントリはタイムスタンプを持ちます。日記は、このループで言語モデルが持つ唯一の仕事です: ステージ中に `memory.md` に観察を書き込むこと。ステージ終了後のすべて — エントリのカウント、それらの浮上、適切なファイルへのルーティング、書き込み — は、決定論的なツールまたはゲートでのあなたの明示的な選択によって行われます。

### ゲートの儀式

各承認ゲートの前に、フレームワークは学習ゲートを実行します(プロトコルはこれを §13 の儀式と呼びます)。2 つのソースから候補を収集し、それらを 1 つの確認リストとして提示します:

1. **あなたのエージェントの日記、逐語的に浮上。** 決定論的なツールが `memory.md` を読み、4 つの見出しの下の各非空行を、そのソース見出しを添えて候補として発行します。言い換えも、「興味深い」フィルタリングもありません — 行は書かれたとおりに表示されます。
2. **常に「次回のために追加することはありますか?」と尋ねる自由記述チャネル。** あなたは観察を入力し、それが 4 つの見出しのどれに属するかを選びます。その見出しの選択が、あなたに求められる唯一の分類です。

保持したい候補にチェックを入れます。ステージで `memory.md` が空だった場合、日記が保持されたかどうかを証言するよう誰も求めません — フレームワークはそれを静かに記録し、先へ進みます。

### 保持された学習の行き先

あなたがファイルパスを選ぶことは決してありません。見出しが行き先を決定します:

- Interpretations、Deviations、Tradeoffs は、`amadeus/spaces/<space>/memory/project.md` のトピック別見出しの下にプラクティスとして入ります(確認された学習はプラクティス*である*)。
- Open questions は昇格しません — それらはリサーチ項目であり、インストールするルールではありません。

デフォルトのスコープは **project** です。ワンクリックの「promote to team」の仕掛けが、教訓がこの 1 つのプロジェクトを超えて適用される場合に、保持された学習を `memory/project.md` から `memory/team.md` へ広げます。org へ広げるパスはありません: org ルールはフレームワーク同梱、または別のプロセスを通じて組織が作成するので、学習ループが org スコープで書き込むことは決してありません。最も狭いスコープにデフォルトすることで、1 つのプロジェクトの驚きが偶発的に組織全体のルールになることを防ぎます。

確認された学習はプラクティス*である*: それは、practices-discovery が確約するのと同じ space メモリファイル(`amadeus/spaces/<space>/memory/project.md`、`memory/team.md`)に入ります — 別のローリングな `*-learnings.md` サーフェスはありません。それらのファイルへの 2 つのパスは、ライフサイクルによって異なります: practices-discovery はセクション全体を決定論的に確約するのに対し、学習ループはゲートを通じて、日付とトピック見出しを持つエントリを一度に 1 つずつ追記します。

保持された学習がルールではなく**センサーバインディング**である場合(ステージの出力に対して新しい決定論的チェックを発火させたい場合)、フレームワークは 2 つの書き込みインストールをアトミックに行います: センサーマニフェストをスキャフォールドし、新しいセンサーの id を発生元ステージのインポートリストに追記します。日記、ゲートの確認、結果として生じるファイル書き込みは、それぞれ監査行(`RULE_LEARNED` または `SENSOR_PROPOSED`)を残すので、ルールが静かにインストールされることは決してありません。

### 受理時の矛盾チェック

保持された学習がディスクに入る前に、フレームワークは `memory/org.md` に対してセクションレベルのチェックを実行します。提案されたエントリが同じ見出しの下の org ルールと矛盾する場合、ゲートは停止し、矛盾する org の文をインラインで引用します。あなたはその後、エントリを修正するか、スキップするか、org ルールのオーナーにエスカレーションするかを選びます。矛盾するルールが矛盾を含んだまま入ることは決してないので、ランタイムのリゾルバはシンプルなままです — すでに矛盾チェックを通過したルールしか見ません。

同じセクションレベルのチェックが、practices-discovery の確約ゲートを守ります。そして、team または project ルールがすでにディスク上にある*後で* org ポリシーが変わった場合、`/amadeus --doctor` が結果として生じるドリフトをオンデマンドで浮上させます: ファイル、セクション、矛盾する org の文を名指しするので、チームは対処できます。doctor チェックは助言的であり、決してブロックしません。2 つの doctor 助言行は、[CLI コマンド](12-cli-commands.ja.md) と [トラブルシューティング](15-troubleshooting.ja.md) に記述されています。

### 実行中ではなく次のワークフローで適用される

あるゲートで捕捉された学習は、現在のワークフローの残りに対してルールを**変更しません**。あなたはこの実行のために会話ですでにエージェントを修正しました。ルールは次回のためのものです。新しい行はディスク上にありますが、進行中のワークフローは開始時のコンパイル済みビューを保ちます。

次にワークフローを開始するとき、コンパイルが新しいファイルを読み、ルールはステージ 1 から先に適用されます。これは、ルーターが BGP から得るのと同じ安定性の特性です: ルートはパケット飛行中に再計算されず、AI-DLC はワークフロー途中で再コンパイルしません。見返りは予測可能性です — ワークフローの早い段階で承認したゲートは、安定したルールセットを証言しており、フレームワークは進行中の実行の足元を変えません。

## 実践例: ANZ 銀行プロジェクト

具体的なウォークスルーが、ループを端から端まで示します。

Sam は ANZ 銀行プロジェクトで `/amadeus feature` を実行します。ワークフローは `requirements-analysis` に到達します。Sam は「the transaction shouldn't duplicate on retry」というステークホルダーノートを書きます — 銀行のトランザクション、処理される支払いを意味しています。product エージェントは「transaction」をデータベーストランザクションと読み、そのノートを ACID セマンティクスの要件として解釈します。Sam はそれを修正します。エージェントは成果物を更新して続行します。

フレームワークの中でこれを予測したものは何もありませんでした。ANZ 固有の用語についてのルールはなく、一般語対ドメイン語の衝突を捕らえるセンサーもありません。エージェントは単に曖昧性にぶつかっただけです。

**1. 日記がそれを記録する。** フレームワークは `<record>/inception/requirements-analysis/memory.md` の `## Interpretations` の下にエントリを追記します:

```
- 2026-05-21T09:14:32Z — Stakeholder note used "transaction"; interpreted as
  database-transaction. Sam corrected to mean banking-transaction (the payment
  being processed, not the DB write). Worth flagging for ANZ project context.
```

まだルールはインストールされていません — これは単にエージェントの日記です。

**2. ゲートがそれを浮上させる。** ステージが終わります。承認ゲートの前に、学習ゲートが `memory.md` を読み、解釈の行を候補として表示します。また「次回のために追加することはありますか?」と尋ねます。Sam は transaction 用語の候補にチェックを入れ(Interpretation → `memory/project.md` に入る)、自由記述のノートを追加します:「エージェントが AWS アカウント用語にデフォルトし続けた — 銀行顧客エンティティには常に『ANZ customer』と言うべき」。Sam はその追加に Deviation 見出しを選び、同じ `memory/project.md` ファイルにルーティングします。

**3. 矛盾チェックが実行される。** フレームワークは両方のエントリを org プラクティス(`memory/org.md`)と比較します。「ANZ transaction」も「ANZ customer」の用語も org ルールでカバーされていないので、両方通過します。決定論的なツールが両方の行を provenance とともに `memory/project.md` に書き込み、監査ログがそれぞれに `RULE_LEARNED` イベントを記録します。

**4. 現在のワークフローは変わらず続行する。** ステージは承認され、ワークフローは `user-stories` に進みます。新しい行はディスク上にありますが、このワークフローのコンパイル済みビューには入りません — Sam はこの実行のためにステージ内ですでにエージェントを修正しました。

**5. 次のワークフローがそれらを拾う。** その日の後、Sam は `/amadeus bugfix` を実行します。ワークフロー開始時のコンパイルが space メモリ層を辿り、`memory/project.md` を拾い、すべてのステージのコンテキストに含めます。bugfix ワークフローのステージ 1 から、エージェントは「transaction」が支払いを意味し、顧客エンティティが「ANZ customer」であることを知っています。

コストは一度支払われ — 1 回のゲート確認、1 回のファイル書き込み — ディレクトリ walk 内の 1 つ多いファイルの代償で、将来のすべてのワークフローで元が取れます。

## センサー: 決定論的なセカンドオピニオン

ルールはエージェントが読む散文です。センサーは、エージェントがステージの出力を書くときに自動的に実行される決定論的チェックです。ルールが「ユーザーストーリーは Given/When/Then 形式に従う」と言うところ、センサーは、必要な見出しが実際にファイルに存在することをバイト単位で検証できます。ルールはエージェントをステージへと前方に送り込みます。センサーは、エージェントが実際に生成したものをフィードバックします。

### センサーの発火の仕組み

エージェントがステージ中に出力ファイルを書いたり編集したりすると、PostToolUse フックが、そのステージにどのセンサーが適用されるかをチェックし、一致する各センサーを実行します。マッチングはファイルの形状によります — コード品質センサーは `**/*.{ts,js}` を分析すると宣言するので、TypeScript と JavaScript の書き込みでのみ発火します。任意のステージ出力で発火するドキュメント形状センサーは、フィルタを省略します。ワークフロー中にセンサーを手で呼び出すことはありません。それらはすべての Write と Edit に相乗りします。

センサーの結果は、このリリースでは**助言的**です。失敗するセンサーは、監査行と、何が欠けているかを正確に指し示す詳細ファイルを生成しますが、ステージの承認ゲートをブロックしたり、ワークフローを停止したりはしません。あなたはシグナルを見て、それをどうするかを決めます。

### 監査ログで見えるもの

センサーの活動は、intent の `audit/` シャードに `Sensor Fired`、`Sensor Passed`、`Sensor Failed` の行として現れます。失敗した行は、特定のギャップ — 欠けている見出し、参照されていない上流成果物、lint エラー — を列挙する詳細ファイル(たとえば `<record>/.amadeus-sensors/<stage-slug>/required-sections-<timestamp>.md`)にリンクします。監査ログは [状態と監査](10-state-and-audit.ja.md) でカバーされています。

### 4 つのフレームワークセンサー

4 つのセンサーがフレームワークに同梱されます:

| センサー | 発火対象 | チェック内容 |
|--------|----------|--------|
| `required-sections` | 任意の record ディレクトリの markdown 出力 | 出力が必要な H2 見出しを含むこと(汎用的なコンテンツ形状チェック) |
| `upstream-coverage` | 任意の record ディレクトリの markdown 出力 | 出力の散文が、ステージが消費すると宣言した各上流成果物を参照すること |
| `linter` | `.ts` / `.js` コード出力 | 設定したリンター(デフォルトは ESLint)をラップ |
| `type-check` | `.ts` / `.tsx` コード出力 | 設定した型チェッカー(デフォルトは `tsc`)をラップ |

各ステージは、その出力に対してどのセンサーが発火するかを宣言します。独自のセンサーを追加できます — `.claude/sensors/` の下にマニフェストを作成し、その id を実行すべきステージに追加します。学習ループも、ゲートで確認したときにセンサーをインストールできます。マニフェスト形式、ステージごとのマトリクス、作成のウォークスルーは [センサーシステム](../reference/07-sensor-system.ja.md) に存在します。自分のプロジェクトに追加するには、[カスタマイズ](13-customization.ja.md) を参照してください。

## Planes: どのように組み合わさるか

planes について考えなくても、上記のすべてを使えます。しかし、基盤となる設計はネットワーキングから規律を借りており、それに名前を付けることで、「次のワークフローで適用される」挙動が腑に落ちます。

現代のルーターは、その作業を 3 つの plane に分割します。AI-DLC はその分割を反映します:

- **Control plane** — 何を実行すべきかの*スキーマ*。ステージ定義、ルール、センサー。ネットワーキングの用語では、これはルート計算です: 設定が与えられたら、何がどこに適用されるかを決める。Control plane は一度だけ実行されるので、遅く賢いことが許されます。
- **Data plane** — *実際の実行*。ステージ実行、エージェント呼び出し、intent の record ディレクトリ内のファイル。ネットワーキングの用語では、これはパケット転送です: 速く、繰り返し、ルックアップによる。Data plane は解決された答えを読みます。それらを再導出しません。
- **Management plane** — *観察と設定*のサーフェス。`/amadeus --doctor`、監査ログ、`CLAUDE.md`。あなたはここで設定し、ここで問い合わせます。人間のペースで。

Control plane は、ワークフロー開始時にあなたのルールとセンサーをグラフに**一度**コンパイルします。Data plane は、実行の残りの間、そのグラフから事前解決された答えを読みます。だからこそ、ワークフロー途中で捕捉された学習は次のコンパイルを待ちます: フレームワークは「トポロジー変更時」(ワークフロー開始)に答えを計算するのであり、「パケット時」(各ステージ)ではありません。結果は、再現可能な実行と、再起動後のクリーンな回復です。

ユーザーとして、あなたはたいてい一度に 1 つの水平スライスに触れます — ワークフローの実行、学習の捕捉、チームプラクティスのカスタマイズ、起きたことの監査。各スライスは、あなたに推論させることなく、下の planes に触れます。完全なモデル、コンパイル境界、回復特性は [Plane アーキテクチャ](../reference/02-plane-architecture.ja.md) に、それが生成するテレメトリ成果物は [ランタイムグラフ](../reference/13-runtime-graph.ja.md) に文書化されています。
