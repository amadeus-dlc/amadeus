# UC001 並行開始を判断する

## ユースケース

Maintainer または Agent が、新しい Issue の作業を進行中の Intent と並行させてよいかを、policy の判断基準を根拠に決める。

## アクター

- ACT001 Maintainer
- ACT002 Agent

## 外部システム

- なし

## 事前条件

- 並行運用ポリシーが steering の索引から参照できる。
- 進行中の Intent の変更対象が成果物（Bolt の実装対象、PR）から読める。

## 基本フロー

1. Maintainer または Agent は、候補 Issue の変更対象（skill、ファイル群、promote 単位）を列挙する。
2. 進行中の Intent の変更対象と突き合わせ、接触面の有無を確認する。
3. policy の判断基準（接触面がない場合は並行可、同一 skill への変更集中や promote 単位の重なりがある場合は避ける）に照らして並行可否を決める。
4. 並行する場合は、worktree 単位で作業を分け、同一 worktree 内の作業は直列に実行する。

## 代替フロー

| 条件 | 扱い |
|---|---|
| 接触面の有無を成果物から判断できない。 | 並行を開始せず、変更対象の記録（実装対象、Bolt）を先に確認する。 |
| 接触面が索引行の追加だけである。 | 統合手順（UC002）で解消できる小さい接触として、並行可と判断できる。 |
