# Unit Design Brief

## 概要

この文書は Unit Design Brief である。
Inception では、共有インデックス生成契約の課題解決方針を定め、Bolt 分割と Construction へ渡す設計入力だけを扱う。
再生成スクリプトの引数体系、見出しの抽出規約、検査の実装方式は Construction で確定する。

## 設計戦略

- インデックスの手書きを禁止するのではなく、モジュールファイルへの記述と 1 行の再生成コマンドを既定の動きにすることで置き換える。
- 情報の定義元をモジュールファイルと `state.json` に一元化し、インデックスは導出だけを行う。導出に人間の判断を挟まないことで決定論性を確保する。
- 再生成スクリプトを validator と同じ skill に同梱し、生成規則と検査規則の乖離を構造的に防ぐ。検査は生成規則の再利用として実装する。
- 生成マーカーで「このファイルは生成物」をファイル自身に語らせ、validator がマーカーを検査して手書き編集の混入を検出する。
- migration は生成結果を正とする。既存インデックスの情報をモジュールファイルへ移してから再生成し、差分は情報の欠落がないことで確認する。

## 責務境界

- 所有するもの: 再生成スクリプトと検証、モジュールファイルの見出し契約、生成マーカー、validator の不整合検査、writer skill の手順記述、steering テンプレートの index 雛形、workspace と examples の migration。
- 所有しないもの: `glossary.md` など他の共有成果物の生成物化、Intent と Discovery のモジュールファイル本体の内容、validator の他の検査カテゴリ。
- 依存してよいもの: `StateScaffold.ts` の同梱スクリプトパターン、validator の既存検査基盤（`checkIntents` など）、promote 手順、Bun 実行環境、既存の index 実データ。
- 後続で再確認が必要になる条件: index の列構成が変わった場合、Intent と Discovery 以外の成果物種別が index を持つようになった場合。

## 構成候補

- 見出し契約: Intent モジュールファイルの概要と依存（理由付き）の記述規約を扱う。
- 収集と導出: 配下モジュールと `state.json` からの行情報の収集と導出を扱う。
- 出力: 決定論的な並び順、生成マーカー、Markdown 表の出力を扱う。
- 不整合検査: インデックスと導出結果の一致検査、マーカー検査を扱う。
- 手順とテンプレート: writer skill の手順記述と steering テンプレートの雛形を扱う。
- migration: 既存データの見出し追加と index の再生成置き換えを扱う。

## データと契約候補

- 入力候補: 対象 workspace の path、`intents/` と `discoveries/` 配下のモジュールファイル、各 `state.json`。
- 出力候補: 先頭マーカー付きの `intents.md` と `discoveries.md`、実行結果の要約。
- 状態候補: 生成（初回）、再生成（更新）、不整合検出（validator fail）。
- 事前条件候補: モジュールファイルが見出し契約を満たす。`state.json` が JSON として解釈できる。
- 事後条件候補: 生成直後の validator がインデックスに起因する fail を出さない。
- 不変条件候補: 同じ入力からの再実行は出力を変えない。モジュールファイルの内容は変更されない。

## 検証観点

- 決定論性（同じ入力から同じ出力）と冪等性を固定入力の検証で確認する。
- 並行統合シナリオ（2 モジュール追加 → 統合 → 再生成で両行を含む）を検証に含める。
- 不整合の注入（行の削除、内容の改変、マーカーの削除）が fail になることを確認する。
- 実装前に検証が失敗することを確認する（RED の記録）。
- promote 同期と `npm run test:it:promote-skill` の pass を確認する。

## Bolt 分割方針

- B001 で再生成スクリプトと検証を実装する（見出し契約の定義と RED から GREEN を含む）。
- B002 で validator の不整合検査と生成マーカー検査を追加する。
- B003 で writer skill の手順と steering テンプレートを更新し、promote で同期する。
- B004 で workspace と examples の migration を実施し、標準検証の pass を確認する。
- B002 と B003 は B001 の完了後に互いに独立して実行できる。B004 は B001 から B003 の完了を前提にする。

## Construction への引き継ぎ

- Functional Design で確定する事項: モジュールファイルへ追加する見出しの名前と位置、`discoveries.md` の列の抽出元規約、行の並び順規則、スクリプトの名前と引数体系、検査の実装方式（完全一致か行単位か）。
- 文書変更で確定する事項: writer skill の手順の文言と、手順を更新する skill の全リスト。
- 検証時に確定する事項: 検証の置き場所と repo の test chain への組み込み。
- migration で確定する事項: examples を同じ PR で扱うか後続 PR に分けるか、既存 21 Intent の概要文の出典（現行 index の文言を正とするか）。
