## 質問記録

### Q001

- 確定判断: GD001
- 確認したいこと: インデックスにしか存在しない情報（Intent の概要、依存、依存理由）の定義元をどこへ移すか。
- 確認が必要な理由: この境界が生成入口の契約と validator 検査の両方を決めるため、Inception の最初に確定する必要がある。
- 推奨回答: 各 Intent のモジュールファイルへ移す（概要と依存の見出しを追加する）。
- 推奨理由: 定義元が Intent モジュールに一元化され、並行 branch は自分のディレクトリだけを触る。Markdown のまま人間が読み書きでき、メタデータ JSON の分離のような情報の分散が起きない。部分生成では追記衝突が残り、Issue #334 の受け入れ条件を満たせない。
- ユーザー回答: 案 1（モジュールファイルへ移す）を採用する。実例による説明の後に確定した。

### Q002

- 確定判断: GD002
- 確認したいこと: インデックス再生成の入口（スクリプト）をどこに置くか。
- 確認が必要な理由: 配布先ユーザー環境で実行できることが受け入れ条件であり、配置先が promote と検査の構成を決めるため。
- 推奨回答: validator 同梱 scripts に置く。
- 推奨理由: `StateScaffold.ts` と同じ確立済みパターンであり、生成と不整合検査（validator 本体）が同じ skill にまとまるため、生成規則と検査規則の一致を保ちやすい。各 phase skill への分置は生成規則の実装が重複し、新しい専用 skill は公開入口 10 個のシンプルさを崩す。
- ユーザー回答: validator 同梱 scripts に置く。

### Q003

- 確定判断: GD003
- 確認したいこと: 生成物になったインデックスと手書き編集のすみ分けをどうするか。
- 確認が必要な理由: 生成物と気づかずに直接編集する事故を防ぐ手段が、生成規則と検査の範囲に影響するため。
- 推奨回答: 全体生成とし、ファイル先頭に生成マーカーを置く。
- 推奨理由: validator がマーカーの存在を検査でき、ファイル単体を読んだ人間もその場で気づける。マーカーなしでは事故が起きやすく、生成と手書きセクションの混在は生成規則と検査を複雑にして衝突面も部分的に残る。
- ユーザー回答: 全体生成 + 先頭マーカーにする。
