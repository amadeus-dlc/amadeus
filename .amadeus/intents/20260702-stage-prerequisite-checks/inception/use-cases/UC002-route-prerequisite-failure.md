# UC002 前提不成立を分類する

## ユースケース

Agent が phase skill 起動時の前提不成立を、現在 phase の補修、上流 feedback、後続 Issue 候補へ分類する。

## アクター

- ACT002 Agent

## 外部システム

- なし

## 事前条件

- UC001 で前提確認が実行されている。
- 前提不成立の対象が、成果物構造、前段成果物、現在 Intent 外の課題のどれかとして観測できる。

## 基本フロー

1. Agent は、前提不成立の原因が成果物構造だけか確認する。
2. 成果物構造だけで解ける場合、Agent は `repair_only` に分類する。
3. Agent は、前段 phase または前段 stage の不足が現在の成功条件を妨げているか確認する。
4. 現在の成功条件を妨げている場合、Agent は `upstream_feedback_required` に分類する。
5. Agent は、現在 Intent の成功条件外で小さく追跡できる課題か確認する。
6. 該当する場合、Agent は `follow_up_issue_candidate` に分類する。

## 代替フロー

| 条件 | 扱い |
|---|---|
| 分類に必要な判断材料が不足している。 | `amadeus-grilling` の一問確認候補として扱う。 |
| 同じ前提不成立が繰り返し発生する。 | 人間判断を仰ぐ。 |
