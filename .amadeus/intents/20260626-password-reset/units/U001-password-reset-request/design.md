# U001：パスワード再設定要求の Unit Design Brief

## 概要

この文書は Unit Design Brief である。
Inception では、U001 の課題解決方針を定め、Bolt 分割と Construction へ渡す設計入力だけを扱う。
詳細な Domain Design、Logical Design、実装設計、テスト設計は Construction で確定する。

U001 は、利用者からのパスワード再設定要求を受け付け、再設定トークンを発行し、再設定手段の送信を依頼する。
要求 R001 とユースケース UC001 を満たす設計戦略を扱う。

## 設計戦略

パスワード再設定要求では、利用者の入力を受け付けつつ、アカウントの存在有無を推測できない応答を維持する。
再設定トークン発行と再設定手段送信依頼は同じ価値境界の中で扱い、認証情報更新は U002 に委ねる。

## 責務境界

所有するもの:

- パスワード再設定要求の入力受付。
- アカウント存在有無を推測できない応答。
- 再設定トークン発行条件。
- 再設定手段送信依頼の成立条件。

所有しないもの:

- 認証情報更新。
- 再設定トークンの消費。
- メール配信基盤の配送保証。
- アカウント存在有無を推測できる詳細理由の通知。

依存してよいもの:

- BC001 のアカウントモデル。
- 再設定手段送信先としての EXT001。
- R001、S001、UC001。

再確認が必要になる条件:

- 再設定手段がメール以外に増える。
- 再設定要求の入力識別子が複数になる。
- アカウント存在有無を推測できない応答の制約が変わる。

## 構成候補

- 要求入力を受け付ける責務。
- アカウント識別子を扱う責務。
- 再設定トークンを発行する責務。
- 再設定手段送信を依頼する責務。
- 利用者へ同一性のある応答を返す責務。

実装コンポーネント名と実装ファイルは未確定である。

## データと契約候補

参照する要求:

- R001

参照するユースケース:

- UC001

参照する契約:

- PRE001
- PRE002
- POST001
- POST002
- INV006

入力候補:

- 利用者が提示するアカウント識別子。

出力候補:

- アカウント存在有無を推測できない再設定要求受付応答。

状態候補:

- 条件を満たす場合、再設定トークンを期限付きで発行する。
- 条件を満たす場合、再設定手段送信を依頼する。

事前条件候補:

- 利用者が再設定対象を識別する情報を提示している。

事後条件候補:

- 再設定要求を受け付けたことを利用者へ返す。
- 条件を満たす場合、再設定手段送信が依頼されている。

不変条件候補:

- 応答からアカウントの存在有無を推測できない。

## 検証観点

- アカウントに紐づく連絡先が存在する場合、再設定要求を受け付けることを確認する。
- 再設定トークンが期限付きで発行されることを確認する。
- 存在しないアカウントに対しても、存在有無を推測できる応答差が出ないことを確認する。
- 再設定手段送信依頼に失敗した場合でも、存在有無を推測できる情報を返さないことを確認する。

Construction では、要求受付の単体検証、再設定トークン発行の結合検証、存在有無を推測できない応答の振る舞い検証を候補にする。

## Bolt 分割方針

- B001 は、R001 と UC001 を満たすため、受付境界、再設定トークン発行、送信依頼、存在有無を推測できない応答を扱う。

## Construction への引き継ぎ

- Domain Design では、再設定要求と再設定トークン発行をどの業務概念として扱うかを確定する。
- Logical Design では、再設定手段送信依頼の同期、非同期境界を確認する。
- 実装時には、既存認証基盤、アカウント保存方式、メール送信基盤との接続条件を確認する。
- 検証時には、応答本文、ステータス、遅延、送信有無、ログ露出から存在有無を推測できないことを確認する。
