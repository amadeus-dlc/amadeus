# D001: 記録照合セマンティクスの採用

## 背景

R002 は「記録済みの値を再計算して照合し、md5 不一致、commit 不一致、参照先欠落を検出して失敗として報告できる」ことを求めているが、照合対象を「現在のファイル」にするか「記録時点の内容」にするかは、Inception の Unit Design Brief で Construction の Functional Design に引き継がれた未確定事項だった。

## 判断

`provenance:check` は「現在のファイル」とは比較しない。記録された `commit` と `path` の組から `git show <commit>:<path>` で記録時点の内容を取り出し、md5 を再計算して記録値と照合する。

## 理由

- 歴史的記録がツール更新や後続変更で fail しない（時間に対して安定である）。
- 手書きの捏造値（もっともらしく間違った md5 や commit）は、当時の実測と一致しないため検出できる。「現在のファイル」との比較では、記録後にファイルが正しく変更された場合と誤って記録された場合を区別できない。
- 検出する drift は3種類とする。(1) md5 不一致（`git show` の内容から再計算した md5 と記録値の不一致）、(2) commit 不一致（記録された commit が git 履歴に存在しない）、(3) 参照先欠落（記録された repo 相対 path が記録 commit 時点で存在しない）。
- workspace の絶対 path は環境依存のため照合対象にせず、記録だけを行う。
- スキーマ不適合（9 項目の欠落、JSON 解釈不能）も失敗として扱う。

## 影響

- R002、UC002 はこの照合セマンティクスを前提にする。
- `functional-design/business-rules.md` の BR003、BR004、BR009 の根拠になる。
- `provenance:check` の eval は、md5 不一致、commit 不一致、参照先欠落、スキーマ不適合の4状態を固定入力で検証する。
