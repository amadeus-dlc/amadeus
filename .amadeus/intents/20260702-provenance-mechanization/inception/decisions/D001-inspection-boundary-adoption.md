# D001: 検査責務境界の採用

## 背景

親 Issue #315 は、証拠記録と評価に関わる複数の子 Issue（#296、#297、#240 など）に検査責務が分散することを想定しており、境界がどこかの子 Issue で記録されることを受け入れ条件にしている。grilling session（2026-07-02）で、記録先（`provenance/`）の存在確認を amadeus-validator に含めるかが論点になった（GD003）。

## 判断

validator = 成果物構造の検証、provenance:check = 実測値の照合、evaluator = 意味と接続性の評価（#240 の対象）という検査責務の境界を採用する。記録先の存在確認は validator に含めず、存在確認も内容照合も `provenance:check` が担う。

## 理由

既存コード分析で `AmadeusValidator.ts`（3778 行）に provenance に関する検査が存在しないことを確認した。validator は既存契約上、成果物 Markdown の見出し、テーブル、参照整合性という成果物構造を検査する責務を持ち、実測値の再計算とは異なる関心事である。実測値の照合を validator に混在させると、validator の実行時間と責務が肥大化し、成果物構造の検査という既存契約から逸脱する。evaluator は意味と接続性の評価を扱う #240 の対象であり、md5 や commit の機械的な一致確認とは異なる関心事である。

## 影響

R005 と UC003 はこの境界を前提にする。B003 でこの決定を decisions として確定し、親 Issue #315 の受け入れ条件を満たす。
